# ADR-1: Use Socket Mode for Slack Connection

## Details

Date: 2026-06-15

Status: APPROVED

Actors: Software Engineer

## Context

The app needs to communicate with Slack to receive events (slash commands, @mentions, DMs) and
send responses. Slack offers two connection methods:

1. Socket Mode - The app opens an outbound WebSocket connection to Slack. Events are delivered
   over this persistent connection. No public HTTP endpoint is needed.

2. HTTP Mode (Events API) - Slack sends HTTP POST requests to a public URL hosted by the app.
   Requires request verification (X-Slack-Signature), a publicly accessible endpoint, and ngrok
   for local development.

This app is a showcase/demo deployed to a single Heroku workspace - not a distributed App Directory app.

## Decision

Use Socket Mode for the Slack connection.

## Consequences

Positive:

- No public URL required - simpler Heroku setup (no custom domain, no SSL concerns beyond what Heroku provides)
- No ngrok needed for local development - just `npm run dev` and it connects
- No request verification middleware needed - Socket Mode connections are pre-authenticated
- Faster to set up and iterate on

Negative:

- Limited to single-workspace installs - cannot distribute via Slack App Directory (out of scope per PRD)
- Requires a Socket Mode app token (xapp-) in addition to the bot token
- Heroku dyno must stay running to maintain the WebSocket - if the dyno restarts, the connection drops and auto-reconnects

If we revisit:

- Switching to HTTP Mode would require: adding request verification middleware, exposing a
  public URL, updating Slack app settings to use Events API + Request URL. The Bolt framework
  supports both - the switch is mostly config.
