# ADR-2: Caching, Polling, Source of Truth, and Manual Persistence

## Details

Date: 2026-06-16

Status: APPROVED

Actors: Software Engineer

## Context

The app needs to serve live match data (scores, events, statistics) during active matches while
also maintaining historical match results for AI grounding and commands. Heroku's filesystem
is ephemeral - files written at runtime are lost on every deploy, dyno restart, and daily
cycling. This creates a tension between:

1. Real-time data needs (live scores must be fresh, updated every 15 seconds)
2. Persistence needs (finished match results should survive restarts)
3. Cost constraints (minimize API calls to stay within rate limits)
4. Operational simplicity (no database dependency for a showcase app)

The key questions:

- What is the source of truth for live data vs. historical data?
- When and how does data get persisted to disk?
- How does the app recover after a dyno restart?
- How do we minimize redundant API calls?

## Decision

Adopt a four-tier data architecture with manual persistence, ordered by query priority:

### Tier 1: In-Memory Cache (Source of Truth During Matches)

- Live match data polled from the football API every 15 seconds
- Stored in a `Map<matchId, { score, events, statistics, status, lastUpdated }>`
- Source of truth for all live queries (grounding, commands, broadcasts)
- One API call (`/fixtures?live=all`) feeds all consumers (no duplicate calls)
- Lost on dyno restart - immediately rebuilt on next poll cycle

### Tier 2: Per-Fixture Detail Files (Historical Results - High Value)

- `src/data/fixtures/{id}-{homeTeamId}-{awayTeamId}.json` - one file per finished fixture
  (e.g. `1-MEX-RSA.json`)
- Contains: final score, score object (halftime/fulltime/extratime/penalty), all events
  (goals, cards, subs with minutes), statistics
- No lineups stored (removed to keep file size manageable)
- Created exclusively by the manual sync script after fixtures finish
- Committed to git - persists permanently as historical record
- The app reads these on startup for grounding historical queries
- This is the most valuable persistent data - it's what makes the AI useful for past fixtures

### Tier 3: Static Data Files (Base Configuration)

- `src/data/fixtures.json` - tournament schedule with scores (72 entries: id, dateAndTimeInUTC,
  stage, group, externalId, status, teams with homeTeamId/homeTeamExternalId/awayTeamId/awayTeamExternalId,
  finalScore, score object). This is the "fixture list."
- `src/data/teams.json` - 48 teams with group, rank, points, goalsDiff
- `src/data/players/{teamId}.json` - per-team squads (~26 players each)
- Committed to git - serves as the base dataset deployed with every dyno
- Rarely changes (only when sync script refreshes squads or schedule corrections)

### Tier 4: Manual Sync Script (Bridge Between API and Disk)

- `npm run sync` - run manually by the developer
- Fetches current state from the football API (squads, schedule, finished match results)
- Writes to Tier 2 files (fixture details) and Tier 3 files (schedule, teams, players)
- Developer commits the updated files and deploys
- Frequency: run once for initial setup, then after each match day to persist results

### Query Priority (Data Resolution)

When any component needs fixture data (implemented in `src/data/index.js`):

1. Check in-memory cache (Tier 1) - if fixture is live, this is authoritative
2. Check `fixtures/{id}-{homeTeamId}-{awayTeamId}.json` (Tier 2) - for finished fixtures with full detail
3. Check `fixtures.json` schedule (Tier 3) - fixture exists but may not have events/statistics

### Polling Lifecycle

- Every 15 minutes: check `/fixtures?date=today` to detect if any matches are scheduled
- When a match goes live: switch to 15-second aggressive polling of `/fixtures?live=all`
- When no matches are live: stop aggressive polling, revert to 15-minute checks
- On dyno restart: cache is empty; first 15-minute check triggers live polling if matches
  are in progress; within 15 seconds of restart, cache is warm again

### Why No Auto-Persistence

The app does NOT automatically write match results to disk when a match finishes because:

1. Heroku's ephemeral filesystem means any file written at runtime is lost on next deploy
2. Auto-writing creates a false sense of persistence (data appears saved but isn't)
3. The sync script approach is explicit and controlled - developer knows exactly what state
   is on disk and in git
4. Committed JSON files in git are the true persistent store (deployed to every new dyno)

## Alternatives

- Heroku Redis (persistent cache): eliminates restart gap but adds cost (~$15-25/mo) and
  operational complexity (monitoring, eviction policies, connection management). Rejected
  for a showcase app - overhead outweighs the 15-second restart gap.
- Heroku Postgres (full database): enables auto-persistence and complex queries but adds
  significant complexity (schema management, migrations, connection pooling). Rejected -
  JSON files are sufficient for 72 matches and ~1,200 players.
- Auto-write on match finish (write to ephemeral FS): appears to persist but is lost on next
  deploy. Rejected because it creates a false sense of durability and confuses the mental model.
- Cron job to run sync: would eliminate manual step but adds infrastructure (Heroku
  Scheduler add-on, error handling for unattended runs). Deferred - can add later if manual
  sync becomes burdensome during group stage.

## Consequences

### Positive

- Zero database dependency - no Redis, no Postgres, no add-ons
- Clear mental model: live=memory, results=fixtures/{id}-{home}-{away}.json, schedule=fixtures.json, bridge=sync
- No orphaned or stale data on disk (files only change via deliberate sync + commit)
- Fast recovery after restart (~15 seconds to first live poll)
- Single API call serves all consumers (poller, broadcaster, grounding, commands)
- Git history shows exactly when match data was synced (commit log is the audit trail)

### Negative

- After a match finishes and before sync is run, historical queries for that match
  return "no data" (unless the in-memory cache still has it from the live poll)
- Developer must remember to run `npm run sync` after match days - not automated
- If the developer forgets to sync, the deployed app has stale historical data
- Dyno restart during a live match means ~15 seconds of missing live data (acceptable;
  users may see brief "data unavailable" before cache warms up)
