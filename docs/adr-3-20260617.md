# ADR-3: Web Search Flow - Post-LLM Fallback

## Details

Date: 2026-06-17

Status: APPROVED

Actors: Software Engineer

## Context

The AI assistant (MIA) can only answer questions grounded in local data -
fixtures, teams, players, and live match events. When users ask about news,
injuries, tactics, transfer rumors, or historical context not in local data, the
bot responds "I don't know." This limits the app's usefulness as a showcase for
Heroku Managed Inference Agents.

Adding web search enables the bot to answer a broader range of questions. The
key design question: at what point in the pipeline should the web search be
triggered, and who decides to call it?

Three approaches were evaluated:

### Option Comparison

| Criteria            | Pre-LLM Intent Gate                | Post-LLM Fallback                           | LLM Tool Use                          |
|---------------------|------------------------------------|---------------------------------------------|---------------------------------------|
| Who decides         | App code (keyword heuristic)       | App code (detects "I don't know")           | LLM (tool call)                       |
| When                | Before LLM call                    | After first LLM call fails                  | Mid-conversation (tool loop)          |
| LLM calls (happy)   | 1                                  | 1                                           | 1                                     |
| LLM calls (search)  | 1                                  | 2                                           | 1-2 (tool loop)                       |
| Search accuracy     | Heuristic may false-pos/neg        | Only on genuine gaps (zero false-positives) | LLM may over-call                     |
| Latency (no search) | ~1s                                | ~1s                                         | ~1s                                   |
| Latency (search)    | ~2s (1s search + 1s LLM)           | ~3.5s (1s LLM + 1s search + 1.5s LLM)       | ~2.5s (1s LLM + 1s search + 0.5s LLM) |
| Cost per search     | 1 search + 1 LLM                   | 1 search + 2 LLM                            | 1 search + 1-2 LLM                    |
| Security surface    | High - deterministic, app controls | High - deterministic, app controls          | Lower - LLM controls trigger          |
| Maintenance         | Keyword classifier to maintain     | Detect "don't know" in output               | Depends on MIA tool support           |
| Complexity          | Low                                | Medium                                      | High                                  |

## Decision

Adopt Post-LLM Fallback (Option 2).

Flow:

```text
user message
-> guardrails (PII mask, sanitize)
-> ground (local data)
-> call MIA (attempt 1, local context only)
-> check response: indicates "I don't know" / low confidence?
  ├─ NO -> respond (done, no web search needed)
  └─ YES -> webSearch(sanitized query)
           -> sanitize web results (treat as untrusted data)
           -> call MIA (attempt 2, local + web context)
           -> filter output
           -> respond with source citations
```

### Why Post-LLM Fallback

1. Zero false-positive searches - web search only fires when MIA genuinely
   cannot answer. No heuristic classifier to maintain or get wrong.
2. Cost predictable - most questions are answerable from local data (scores,
   fixtures, players). The majority of requests never hit web search.
3. Security maintained - app code controls the trigger (not the LLM). Web
   content is sanitized before being injected as grounding context.
4. No MIA tool-use dependency - works with any OpenAI-compatible chat API.
   Does not require MIA to support function calling.

### Why Not the Others

- Pre-LLM Intent Gate: Keyword classifiers are brittle. "Tell me about
  Messi's injury" should search, but "who scored?" shouldn't - and edge cases
  pile up. False positives waste search calls; false negatives leave users
  without answers with no retry path.
- LLM Tool Use: Most "agentic" but couples security to LLM behavior. The
  LLM decides when to search, making guardrails harder. Also requires MIA to
  support tool/function calling (unverified capability).

### Trade-off Accepted

The fallback path is slower (~3.5s vs ~2s) because it makes two LLM calls. This
is acceptable because:

- It only happens on genuine knowledge gaps (minority of queries)
- Users already expect slightly longer waits for "I had to look that up" answers
- The fast path (local data answers) remains ~1s - no regression for common queries

## Consequences

### Positive

- Search costs stay low (only fires on genuine gaps, not false-positive triggers)
- No classifier to build, tune, or maintain over time
- Simple "I don't know" detection is robust (pattern match on output)
- Full guardrail pipeline applies to web content (sanitize, filter, audit)
- Source citations make web-sourced answers transparent to users
- Rate limiting can be added as a thin layer later without changing the flow

### Negative

- 2 LLM calls on fallback path (higher latency + cost when search fires)
- Need reliable "I don't know" detection (false negatives = missed search
  opportunities; over-detection = unnecessary searches)
- Slight UX inconsistency: some answers are fast (~1s), some slow (~3.5s)
- If MIA improves and stops saying "I don't know" (hallucinating instead),
  fallback never fires - need monitoring to catch this
