# ADR-4: Daily Scheduler - In-Process Clock via setInterval

## Details

Date: 2026-06-19

Status: APPROVED

Actors: Software Engineer

## Context

The highlights feature requires a daily job at a configurable UTC hour to
fetch the playlist, match videos to fixtures, enrich data, and post a digest.
The key design question: how should this job be scheduled on Heroku?

The worker dyno already runs 24/7 for live match broadcasting. The existing
`broadcast/scheduler.js` uses setInterval + tick (custom clock process
pattern). Heroku's own docs recommend custom clock processes over their
Scheduler add-on for reliable execution.

## Decision

Adopt In-Process setInterval in the existing worker dyno.

Architecture:

```text
worker dyno (node src/index.js)
  ├── app.start() (Slack Socket Mode)
  ├── livePoller.start() (live match data)
  ├── broadcastScheduler.start(client) (match cards, alerts, recaps)
  └── highlightsScheduler.start(client) (daily digest) <-- new
```

The worker process IS the custom clock process that Heroku recommends. All
schedulers share the Slack client, live cache, and data layer in-process.

Rationale:

1. Already the established pattern - `broadcast/scheduler.js` uses identical
   setInterval + tick architecture. No new pattern to maintain.
2. Shared Slack client - digest posting reuses `app.client` directly. No
   need to initialize a fresh Bolt app or manage OAuth tokens in a one-off
   dyno.
3. Reliable - Heroku Scheduler explicitly documents itself as "best effort"
   and may skip runs. Our job runs exactly once daily with built-in
   missed-run recovery on dyno restart.
4. Minimal granularity - 1-minute tick checks allow precise run-hour
   targeting. Heroku Scheduler only offers 10-min minimum.
5. Zero infrastructure cost - worker dyno is already running 24/7 for live
   match broadcasting. Adding another setInterval has negligible overhead.
6. Self-contained missed-run recovery - on restart, compare lastRunDate to
   today and fire immediately if the window was missed. Heroku Scheduler
   has no equivalent.

Trade-off accepted: the worker dyno must stay running for the scheduler to
fire. This is already a requirement for live match broadcasting, so the
highlights scheduler adds no new operational constraint. If the dyno restarts
(Heroku daily restart, deploy, crash), missed-run recovery handles it.

## Alternatives

- Separate Clock Dyno: The Heroku-recommended pattern for apps that need
  scheduling separate from their web process. But our worker IS the clock
  process - it has no web traffic to contend with. Splitting would duplicate
  Slack initialization, require IPC or shared state for the digest to access
  live match data, and add an extra dyno ($$) - all for no benefit.
- Heroku Scheduler Add-on: Documented as "best effort" - may miss runs with
  no recovery mechanism. Requires spinning up a one-off dyno each time,
  which must initialize Slack client from scratch. Does not support
  sub-10-minute granularity. Adds external dependency for a simple daily job.

| Criteria               | In-Process setInterval (clock)         | Separate Clock Dyno                  | Heroku Scheduler Add-on           |
|------------------------|----------------------------------------|--------------------------------------|-----------------------------------|
| Heroku pattern         | Custom clock process (recommended)     | Custom clock process (recommended)   | Managed cron                      |
| Reliability            | Guaranteed (process is always up)      | Guaranteed                           | "Best effort" (can miss runs)     |
| Granularity            | Any (we use 1-minute checks)           | Any                                  | 10min / hourly / daily only       |
| Slack client access    | Shared in-process (zero cost)          | Must connect fresh or use API        | Must initialize fresh each run    |
| Missed-run recovery    | Built-in (check lastRunDate on start)  | Built-in                             | Not supported                     |
| Infrastructure         | None (worker dyno already running)     | Extra dyno ($$)                      | Add-on + one-off dyno per run     |
| Boot time              | 0s (already running)                   | ~5-10s on restart                    | ~5-10s (one-off dyno spin-up)     |
| Existing pattern       | broadcast/scheduler.js uses this       | Would split codebase                 | Not used in this app              |
| Complexity             | Low (10 lines of scheduling code)      | Medium (IPC or separate Slack init)  | Low (add-on config)               |
| Cost                   | $ (worker already runs 24/7)           | $$ (dedicated dyno)                  | $ (add-on free, dyno boot costs)  |

## Consequences

### Positive

- Consistent architecture with broadcast scheduler (one pattern to understand)
- Zero additional infrastructure or add-ons
- Reliable execution with missed-run self-healing
- Sub-minute precision available if needed in future
- Full access to in-process Slack client, data layer, and live cache
- Configurable via single env var (HIGHLIGHTS_RUN_HOUR_IN_UTC) - simpler than
  cron syntax for a once-daily job

### Negative

- Dyno must stay running (already required for broadcast - not a new constraint)
- All schedulers share one process - a crash in highlights could theoretically
  affect broadcast (mitigated by try/catch isolation in each scheduler's tick)
- No built-in dashboard for run history (mitigated by console.log + Heroku
  log drain)
- Less flexible than cron syntax - only supports hour-level scheduling, not
  minute-specific or multi-run-per-day (sufficient for this use case)
- If app grows significantly, may eventually want to extract schedulers to
  a dedicated clock dyno (future refactor, not current concern)
