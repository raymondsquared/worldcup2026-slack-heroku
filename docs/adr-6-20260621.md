# ADR-6: MIA Client-Side Tool Calling - Model-Driven Grounding Retrieval

## Details

Date: 2026-06-21

Status: APPROVED

Actors: Software Engineer

## Context

MIA grounds its answers through "Secure Data Retrieval / dynamic grounding" (one
of the trust-layer controls implemented in `src/mia/`, see ADR-5 and AGENTS.md):
before calling the model, `ask()` retrieves only approved World Cup 2026 data and
injects it as context. That retrieval was done by a hand-coded keyword router
with exactly three regex branches:

```js
if (/fixture|match|schedule|play|game|next|when/.test(text)) { getUpcomingFixtures(3) }
if (/group|standing|table/.test(text))                       { getGroups() }
if (team && /player|roster|squad|who/.test(text))            { getPlayers(team.id) }
```

Real match-day use exposed three gaps where the router fails to retrieve data
the app already has locally:

1. Team-aware fixtures: "when is the next England match" injects the global
   next-3 fixtures, not England's, so England's actual next match is absent.
2. Past results and events: "what was the score in the Brazil game" or "who
   scored" matches no branch, even though 36 finished fixtures and their full
   event lists live in the data layer.
3. Player lookup: a player named without their team ("what number does Mbappe
   wear") is invisible, because the router only recognizes team entities.

In each gap the data is local, but the keyword router does not retrieve it; the empty or
wrong context drives a low-confidence answer and the web-search fallback fires
for a question the app could have answered offline. Every new question type
needs another keyword list plus another accessor call, and the router still
misses paraphrases and never narrows to the entity the user named.

The question: how do we retrieve the right local data for a much wider range of
questions without hand-coding a regex branch per question type, while keeping the
existing trust-layer guardrails and the web-search fallback intact?

## Decision

Replace the keyword router with model-driven tool calling, implemented
client-side in the existing inference client. When the tool loop grounds nothing,
a direct single model call (no tool context) is the deterministic fallback.

### Model selects read-only tools over the data layer

`ask()` offers the model a fixed whitelist of seven read-only tools (all
delivered in task-37), each a thin wrapper over an existing data accessor except
the derived `get_player_goals`:

| Tool                  | Backing accessor                     | Answers                          |
| --------------------- | ------------------------------------ | -------------------------------- |
| get_upcoming_fixtures | getAllFixtures / getUpcomingFixtures | next-match / upcoming (per team) |
| get_team_results      | getAllFixtures (finished)            | past scores / results            |
| get_fixture_events    | getFixtureEvents                     | who scored / cards               |
| get_standings         | getGroups                            | group table / points             |
| get_team_squad        | getPlayers                           | squad list                       |
| get_player            | getAllPlayers (task-35 index)        | player number / position         |
| get_player_goals      | getAllPlayers + getFixtureEvents     | goal tally (derived)             |

The model chooses which tool(s) to call; the client executes each via a
whitelist `dispatchTool`, feeds `role: tool` results back, and loops until the
model returns a final answer. Adding a new capability becomes "declare one tool
schema", not "add a regex branch plus prompt wiring". `get_player_goals` is the
one tool that is derived logic rather than a thin wrapper - it tallies `Goal`
events across fixtures by `playerExternalId`, counting penalties and excluding
own goals; see plan-task-37 for the counting rules.

### Client-side loop, not the Heroku Agents server-side endpoint

The loop runs in `src/mia/client.js` (`chatWithTools`) and `src/mia/index.js`,
using the OpenAI-compatible `/v1/chat/completions` `tools` contract. The
existing `chat()` string return is preserved for recap and other callers;
`chatWithTools()` returns the full assistant message so the caller can read
`tool_calls`.

### Direct single call as the deterministic fallback

`ask()` falls back to a direct single model call (no tool context) when the
configured model does not support function calling, when the model answers with
no tool call, when the loop is exhausted, or when the loop exceeds its wall-clock
budget. Grounding happens on the primary path - the tools retrieve approved data
before the model answers - so the Secure Data Retrieval property holds via the
tool loop.

### Gating, bounds, and telemetry

- The tool loop is the unconditional primary retrieval path (no runtime flag).
  A one-shot capability probe confirmed the configured `INFERENCE_MODEL_ID`
  returns `tool_calls` (2026-06-21: PASS, `claude-4-5-haiku`). The probe script
  has been removed; re-verify manually if the model id changes.
- The loop is bounded by a maximum iteration count (`MAX_TOOL_ITERS = 4`) and a
  total wall-clock budget (`LOOP_BUDGET_IN_MS = 10000`), checked at the top of
  each iteration; once exceeded it falls back to the direct single call rather than letting a
  degenerate loop run up to the 4x15s per-request ceiling (~60s). Because the
  check is per-iteration, the actual worst case is the budget plus one in-flight
  request (~budget + 15s = ~25s), not a hard 10s; that is acceptable on the async
  Events API reply path, which has no Slack 3-second deadline.
- `logInteraction` records a `retrievalPath` (`tool_call`, `context`, `direct`,
  or `web_search`; plus `blocked` and `recap` on the non-answer paths) so the
  web-search fallback rate is observable and SC-8 is measurable.

```text
ask()  (trust-layer pipeline, guardrails unchanged)
  mask PII -> sanitize -> toxic filter (input)
    -> retrieval:
         caller supplied authoritative context? single call -> retrievalPath = context
         else tool loop: model -> tool_calls -> dispatchTool (whitelist) -> feed back
             grounded answer            -> retrievalPath = tool_call
             no tool ran / exhausted / over budget / error
               -> direct single call (no tool context) -> retrievalPath = direct
    -> parseResponse (unchanged XML contract)
    -> low-confidence AND football-related? web search + retry  -> retrievalPath = web_search
  toxic filter (output) -> demask -> audit log (with retrievalPath)
```

## Alternatives

The options are summarized in turn below; the first is the decision recorded
above, the rest are the alternatives considered.

### Model-driven client-side tool calling (+ direct-call fallback) - Chosen

- Mechanism: model picks read-only tools from a whitelist; client runs each via
  `dispatchTool`, feeds `role: tool` results back, loops to a final answer.
- Strengths: closes the 3 retrieval gaps; narrows to the named entity; new
  capability = one tool schema; guardrails unchanged; `retrievalPath` telemetry.
- Trade-off: extra model round-trips (latency, tokens); depends on model
  tool-calling capability.

### Keep extending the keyword regex router - Rejected

- Mechanism: add another keyword list + accessor call per question type.
- Strengths: deterministic, model-free, single cheap call, no tool-capability
  dependency.
- Trade-off: misses paraphrases; cannot narrow to the named entity; unbounded
  hand-coding; the status quo whose gaps motivated this change.
- Why not: the keyword router was replaced, not extended.

### Dump all data into every prompt - Rejected

- Mechanism: send every fixture, event, and squad as context on every call.
- Strengths: no retrieval logic to maintain; model sees everything.
- Trade-off: costly; dilutes model focus; scales badly as data grows.

### Expose the data layer as an MCP server - Rejected

- Mechanism: run the data accessors as an MCP server the app connects to.
- Strengths: standardized tool interface; reusable by other MCP hosts.
- Trade-off: the chat-completions loop is unchanged (still list tools -> convert
  to OpenAI schemas -> send -> dispatch -> feed back); adds a process boundary,
  transport, and lifecycle in place of a synchronous local `dispatchTool` call.
  More moving parts, no new capability.

### Heroku Agents server-side orchestration - Deferred

- Mechanism: move the tool loop server-side to the Agents endpoint (this is also
  where a server-side MCP-orchestration model would land).
- Strengths: less in-process loop code; managed orchestration.
- Trade-off: heavier setup; less in-process control over the guardrails (PII,
  toxicity, audit) that wrap the loop.
- Status: deferred; revisit if the client-side loop proves limiting (e.g.
  parallel tool calls or streaming).

### Tool loop only, no deterministic fallback - Rejected

- Mechanism: tool loop with no fallback path at all.
- Strengths: single retrieval path, least surface to maintain.
- Trade-off: a model lacking tool support, emitting no `tool_calls`, or running
  slow becomes a hard failure with no fallback.
- Why not: the direct single call is kept as the fallback.

A note on MCP, since it is a natural question here: the inference endpoint is a
raw OpenAI-compatible `POST /v1/chat/completions` that takes tool schemas inline
and returns `tool_calls`; it does not speak MCP. MCP therefore lands in one of
the two options above, neither simpler for this app: as a data-layer MCP server it
keeps the exact same client loop and only swaps a local function call for a
network round-trip, and as server-side orchestration it is the deferred Heroku
Agents option, parked because the in-process trust-layer guardrails cannot wrap
a server-side loop the same way.

## Consequences

### Positive

- Closes the three documented retrieval gaps: team-aware fixtures, past
  results/events, and player-name lookup are all answered from local data.
- Adding a capability is one tool schema, not a regex branch plus prompt wiring.
- The model narrows to the entity the user named (it passes a team or player
  argument), which the keyword router could not do.
- Telemetry (`retrievalPath`) makes the web-search fallback rate observable, so
  the SC-8 improvement is measurable on real traffic, not just in unit tests.
- Trust-layer guardrails are unchanged: PII masking/demasking, toxic input/output
  filtering, audit logging, the system-prompt scope instruction, and the
  web-search low-confidence fallback all still wrap the new path. Tools are
  read-only over existing local accessors, so no new external data source is
  added.
- Rollout was low-risk: a capability-probe gate plus the direct-call fallback,
  which still catches any model that loses tool support.

### Negative

- Extra round-trips: each tool call adds a model request (model -> tool_call ->
  execute -> model), so the tool path uses more latency and tokens than the
  single-call direct path. Bounded by the iteration cap and the wall-clock
  budget. MIA's `ask()` runs on the asynchronous Events API reply path (mention /
  DM via `say()`), not the slash-command path, so there is no Slack 3-second
  request deadline; the cost is the user waiting longer for the bot reply.
- The model may select a poorly-fitting tool or one that returns an empty result;
  in that case it receives a valid-but-thin result, produces a low-confidence
  answer, and the existing web-search fallback handles it - the same graceful
  degradation as an empty tool result today.
- Tool calling depends on model capability, which can change if
  `INFERENCE_MODEL_ID` is swapped; mitigated by the probe and the direct-call
  fallback, which absorbs a model that emits no `tool_calls` at runtime (re-run
  the probe after a model change).
- Two retrieval paths now coexist (the tool loop and the direct single call),
  which is more surface than one router; accepted because the direct fallback is
  a trivial no-tools call and both paths share the same downstream
  parse/guardrail/fallback code.

## task renumber

This tool-calling work is task-37. It was renumbered twice before commit, both
documentation bookkeeping with no change to any decision in this ADR:

- `get_player_goals` originally shipped as its own task-40 (with the
  live-card-transition fix as task-41); the derived tool was folded into the
  tool-calling task it belongs to, and the live-card-transition fix became its
  own task.
- The iteration-5 live-path bug fixes (live-card source, poller kickoff, daily
  restart, event dedup on minute drift, and event player-name resolution) were
  then consolidated into a single task-35. That absorbed the former dedup and
  name-resolution tasks, shifting the tool-calling task from task-39 to task-37
  and the live-card-transition fix to task-38.
- The live-card-transition fix was finally folded into task-35 as well (Part 6),
  since it is another bug fix on the same differ -> handleDiffs -> updateCard
  live-broadcast path discovered in the same match-day run. task-35 now carries
  all seven live-path parts (a later match-end card-render fix joined the same
  task as Part 7); the tool-calling task stays task-37.

Every `get_player_goals` reference above now points at task-37, and its counting
rules live in plan-task-37. The `getAllPlayers()` squad index the player tools
rely on is now delivered in task-35 Part 5 (formerly its own name-resolution
task).
