# ADR 7: STDIO Transport for Football API MCP Server

## Details

Date: 2026-06-25

Status: APPROVED

Actors: Product Manager, Software Engineer

## Context

Heroku's MCP documentation and reference implementation (`mcp-code-exec-node`)
support multiple transport modes for MCP servers:

1. STDIO - Process-based communication via stdin/stdout
2. SSE (Server-Sent Events) - HTTP streaming protocol
3. HTTP - Request/response with streamable responses

Each transport has different operational characteristics, security properties,
and cost implications when deployed to Heroku.

For the Football API MCP server, we must choose a transport mode that balances
security, cost, and integration simplicity with Heroku Managed Inference.

Critical Constraint: Heroku Private Spaces do not support MCP integration with
Heroku Managed Inference. MCP servers must be deployed to Common Runtime (public) to
be registered and used with the `/v1/agents/heroku` endpoint. This limitation applies
to all MCP transports (STDIO, SSE, HTTP), not just STDIO.

Reference: [Heroku Inference - Working with MCP](https://devcenter.heroku.com/articles/heroku-inference-working-with-mcp#deploy-a-custom-mcp-server-to-heroku)

> "Currently, MCP servers running in Private Spaces cannot be registered or used by `/v1/agents/heroku`."

This necessitated migrating the entire application from Private Space to Common Runtime.
See: `.agents/artifacts/migration-to-common-runtime.md` for migration details.

## Decision

Use STDIO transport with `/v1/agents/heroku` endpoint for the Football API MCP server.

### Why `/v1/agents/heroku` (not `/v1/chat/completions`)

Both endpoints can use MCP tools, but they require different server architectures:

- `/v1/agents/heroku` — automatic orchestration. Heroku spawns STDIO processes
  in one-off dynos, executes tools, feeds results back to the model, returns final
  answer. No control loop needed on our side.
- `/v1/chat/completions` — manual orchestration. Requires "multi-tenant long-running
  SSE or streamable HTTP MCP servers, which can be less secure and costly" and building
  your own control loop.

We chose `/v1/agents/heroku` because it supports STDIO transport via on-demand one-off
dynos. The `/v1/chat/completions` approach requires long-running HTTP/SSE servers per
the docs, which would mean rewriting our STDIO server as an always-on endpoint.

Reference: [Heroku Inference - Working with MCP](https://devcenter.heroku.com/articles/heroku-inference-working-with-mcp)

The MCP server will:

- Implement the MCP protocol over stdin/stdout using
  `@modelcontextprotocol/sdk` StdioServerTransport
- Run in one-off dynos spawned per-request via `/v1/agents/heroku`
- Not expose any HTTP endpoints or persistent web processes
- Accept ~300-500ms cold start latency per tool call (acceptable for demo)

## Demo Deployment Notes (2026-06-25)

### Deployment Requirements

- Must use Common Runtime (public) -
  [Heroku docs](https://devcenter.heroku.com/articles/heroku-inference-working-with-mcp#deploy-a-custom-mcp-server-to-heroku):
  "Currently, MCP servers running in Private Spaces cannot be registered or used by `/v1/agents/heroku`."
- Process type declared in Procfile: `mcp-football: node src/mcp/football-server.js`
- Scale to 0 (on-demand): `heroku ps:scale mcp-football=0`
- Heroku spawns process automatically when tools are called

### Demo Characteristics

- ~300-500ms cold start per tool call (acceptable for demo)
- Cost-efficient (~$7/mo for 1 worker dyno, MCP runs on-demand)
- Security isolation (one-off dynos per request)
- No persistent HTTP endpoints exposed

## Alternatives

### Alternative 1: SSE Transport (Always-On Web Dyno)

Description: HTTP endpoint with Server-Sent Events for persistent MCP server

Why not chosen for this demo:

- Requires always-on web dyno (higher cost ~$7-25/month vs. metered one-off dynos)
- Demo/POC doesn't require production-level latency
- ~300-500ms cold start acceptable for tournament demonstration
- Cost optimization appropriate for time-bound demo (40-day tournament)

Note: This is the recommended approach for production chat applications where
latency matters. The ~$15-50 tournament cost is acceptable for demos, but production
apps should prioritize user experience over cost.

### Alternative 2: HTTP Transport

Description: Standard HTTP request/response with streamable responses

Why rejected:

- Same drawbacks as SSE (always-on web dyno, exposed endpoint)
- No streaming benefit for simple read-only tool calls
- More attack surface (HTTP vulnerabilities vs. process isolation)

### Alternative 3: Dual Transport (STDIO + SSE)

Description: Support both STDIO and SSE via `REMOTE_SERVER_TRANSPORT_MODULE` env var

Why rejected:

- Increased complexity for no current use case
- SSE endpoint unused but requires maintenance
- Can add later if external HTTP access is needed

## Consequences

### Positive

- Better security - Code execution isolated in one-off dynos per request
- Lower cost - No always-on web dyno; pay only for MCP tool call duration
- Simpler deployment - No HTTP routing, authentication, or API key distribution
- Heroku recommended - Aligns with official Heroku MCP best practices
- Automatic scaling - One-off dynos scale with request volume

### Negative

- Latency - One-off dyno cold start adds ~300-500ms per tool call (acceptable for demo)
- No external access - Cannot call MCP server from outside Heroku Inference
- Debugging harder - Cannot curl HTTP endpoint; must use MCP Inspector or inference API
- Stateless only - Cannot maintain server-side state between tool calls (not needed)
- Not production-ready for chat - SSE/HTTP with always-on dyno recommended for production

### Migration Path to Production

To migrate from STDIO (demo) to SSE (production):

1. Implement SSE server using `@modelcontextprotocol/sdk` SSE transport
2. Update Procfile: `mcp-football:` → `web: node src/mcp/football-server-sse.js`
3. Add authentication middleware (API key validation)
4. Scale web dyno: `heroku ps:scale web=1`
5. Update `INFERENCE_URL` to point to web dyno endpoint
6. Test latency improvement (~300-500ms → ~50-100ms)
