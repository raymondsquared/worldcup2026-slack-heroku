# ADR 8: Direct Football API Calls vs Data Layer Reuse

## Details

Date: 2026-06-25

Status: APPROVED

Actors: Product Manager, Software Engineer

## Context

The project has two data access patterns for World Cup information:

1. Data Layer (`src/data/index.js`) - Tiered resolution: live cache (15s fresh) →
   detail files (finished fixtures) → static JSON (schedule)
2. Direct API (`src/live-data/api.js`) - HTTP calls directly to Football API

ADR-6 client-side tools wrap the data layer for grounded AI responses. The
Football API MCP server must choose which pattern to follow.

Data Layer Benefits:

- Cached responses (faster, fewer API calls)
- Tiered fallback (works offline with static data)
- Integrated with poller lifecycle

Direct API Benefits:

- Always fresh data (no cache staleness)
- Simpler implementation (no cache dependencies)
- Independent of bot's internal state

## Decision

MCP server tools will call the Football API directly for chat-based AI queries
(@mentions, DMs), providing always-fresh data at the cost of slightly higher latency.

Use Case Architecture:

- Chat queries (AI @mentions, DMs): Use MCP tools → Direct Football API
  - Prioritizes freshness over speed
  - Acceptable ~300-500ms latency for live match data
  - Always returns current state (no stale data)
- Slash commands (`/worldcup2026 schedule`, etc.): Use ADR-6 tools → Data layer
  - Fast cached responses (instant, may be 15s old)
  - Good enough for schedule viewing and historical data
- Background jobs (poller, sync scripts): Continue using `src/live-data/api.js`
  - Populate live cache and static files as before
  - No changes to existing infrastructure

Each MCP tool will:

- Make HTTP requests to `https://v3.football.api-sports.io`
- Use `x-apisports-key` header for authentication
- Apply 10-second timeout per request
- Filter responses to World Cup 2026 fixtures

## Alternatives

### Alternative 1: Reuse Data Layer

Description: MCP tools call `src/data/index.js` accessors (`getUpcomingFixtures`,
`getFixture`, etc.)

Why rejected:

- Couples MCP server to bot's internal cache state
- Live cache only populated if poller is running
- Adds dependency on in-memory state not accessible from one-off dynos
- ADR-6 tools already provide this interface

### Alternative 2: Hybrid Approach

Description: Use data layer for static data (schedule), direct API for live data

Why rejected:

- Increased complexity (two code paths)
- Inconsistent freshness guarantees across tools
- Still requires coupling to data layer

### Alternative 3: MCP Server Manages Own Cache

Description: MCP server maintains separate cache in one-off dyno

Why rejected:

- One-off dynos are stateless (cache lost between calls)
- Would need external cache (Redis) - out of scope
- Duplicate caching logic vs. existing poller

## Consequences

### Positive

- Always fresh for chat - Chat users get live data, never stale
- Simple implementation - No cache staleness detection logic needed
- Clear use case split - Chat=fresh via MCP, commands=fast via cache
- ADR-6 unchanged - No refactoring of existing 4-day-old tools
- Independent - MCP server works regardless of bot state
- One tool call - No retry/fallback complexity

### Negative

- Higher latency for chat - ~300-500ms cold start + API call vs instant cache read
- More API calls - Each chat query hits Football API (rate limit risk)
- No offline mode - Chat fails if Football API unavailable (slash commands still work)
- Duplicate functionality - MCP tools overlap with ADR-6 tools (intentional for freshness)
