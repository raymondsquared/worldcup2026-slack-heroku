# PRD Plan: Live Data, Thread-Aware Mentions, and Real Squads

Date: 2026-06-16

## Objective

Create a PRD for the next iteration of the World Cup 2026 Slack app that transitions it from a
static demo to a live, data-driven experience - adding real-time scores, full player squads,
and context-aware AI conversations in match broadcast threads.

## Problem Statement

The current app uses static/mock data and has no awareness of live match state. When a user
@mentions the bot in a broadcast thread asking "what's the score?", the bot cannot answer because:

1. It has no live score feed - all match data is hardcoded JSON with no scores
2. It has no thread context - it doesn't know which match the broadcast thread is about
3. Player data is mock - asking "who scored?" returns fabricated player names

This limits the app's showcase value during the actual World Cup 2026 tournament (June 11 –
July 19, 2026). With the tournament now live, the app should demonstrate Heroku MIA grounded
in real, live data.

## Target Users

- Slack workspace members following World Cup 2026 during the tournament
- Developers evaluating Heroku MIA - seeing AI grounded in live, real-world data is far more
  compelling than static mocks
- Heroku advocates demoing the platform with a live, working reference app

## Proposed Solution

Three interconnected enhancements:

### 1. Live Score Integration

Add a score polling module that fetches live match data from the football API (the most popular
football data API). The module polls every ~15 seconds during active
matches, detects new events (goals, cards, subs), and caches results in-memory. This feeds into:

- Updated broadcast cards (edit posted messages with current score)
- Minute-by-minute event broadcasting to match threads (goal alerts, cards, subs)
- AI grounding (the bot can answer "what's the score?" with real data)
- Slash commands (schedule shows live scores for in-progress matches)

### 2. Thread-Aware @Mentions

When the bot is @mentioned in a broadcast thread, it automatically knows which match that
thread corresponds to. The grounding pipeline injects that match's live data (score, events,
statistics) as primary context before calling MIA. This enables natural conversations like:

- "What's the score?" -> "USA 2-1 Mexico, 67th minute"
- "Who scored?" -> "Pulisic (23'), Weah (51') for USA; Lozano (38') for Mexico"
- "How's the game going?" -> AI-generated commentary grounded in live match events

Outside broadcast threads, the bot continues to work as before (general World Cup Q&A).

### 3. Real Player Squads

Replace mock player data with real 2026 World Cup squad data for all 48 teams. Source from
the football API's `/players/squads?team={id}` endpoint which provides full squad arrays with
names, positions, and jersey numbers. Add a sync script (`npm run sync`) that fetches
and writes per-team squad files (`players/{teamId}.json`) with fields: id, name, teamId,
position (raw from API), number, externalId.

## Data Source: 3rd-Party Football API

- Most popular football data API with largest developer community
- Covers FIFA World Cup 2026 (tournament currently live)
- Provides: live fixtures, scores, events (goals/cards/subs with minute markers),
  team squads, standings, statistics, events (goals/cards/subs)
- Authentication: `x-apisports-key` header (stored in Heroku config var `FOOTBALL_API_KEY`)
- Base URL: `https://v3.[football-api-base-url]/` (direct access, no RapidAPI middleman)
- Pricing: paid tier required for live data (~$50-100/mo estimated for WC coverage)

Key endpoints:

- `GET /fixtures?live=all` - all live matches with scores and events
- `GET /fixtures?date=YYYY-MM-DD` - matches by date (summary only, no events/statistics)
- `GET /fixtures?id={id}` - single fixture with full detail (events, statistics)
- `GET /standings?league={id}&season=2026` - group standings (all 48 teams in one call)
- `GET /players/squads?team={id}` - full squad with positions, numbers

Important: `/fixtures?date=` does NOT return events or statistics. Only `/fixtures?id=`
returns full detail. The sync script uses date-based fetching for discovery, then fetches
each fixture individually by ID for events/statistics.

Live match response includes `events[]` array with goal scorers, assists, cards,
substitutions - each with `time.elapsed` (minute) for timeline reconstruction.

Polling every 15 seconds provides near-real-time event detection.

API call estimate: ~2,400 calls/day (1 call every 15s × ~10 hours of active matches/day).
Well within paid tier limits.

## Draft Success Criteria

- SC-1: Live score data appears in broadcast cards within 30 seconds of a goal
- SC-2: Thread-aware mentions return match-specific answers (not generic) when asked in a
  broadcast thread
- SC-3: Real player names appear in all AI responses (no fabricated/mock names)
- SC-4: App degrades gracefully when the football API is unreachable (no crashes, cached data served)
- SC-5: Polling does not exceed API rate limits during multi-match periods
- SC-6: System recovers thread context after dyno restart (via Slack message metadata)

## Key Features (Scope)

- Live score polling from the football API every ~15 seconds during active matches
- Minute-by-minute event broadcasting - goals, cards, subs posted to match threads as they happen
- Broadcast card updates - edit posted match cards with current score
- Thread-context detection - bot knows which match a broadcast thread is about
- Thread-grounded AI - MIA responses include live match data in broadcast threads
- Real player/team data - full 48-team squads with names, positions, numbers, clubs
- Data sync script - `npm run sync` to refresh squads from API
- Cached fallback - app works with cached JSON if API is unreachable

## Technical Approach

- Existing `src/live-data/` module: API client (`api.js`) and mapper (`mapper.js`); future poller, event differ, cache
- Use native `fetch` (Node 22 built-in) - no new dependencies, consistent with existing MIA client pattern
- Enhance existing `src/broadcast/scheduler.js`: store `{matchId -> messageTs}` map, call
  `chat.update()` to edit posted cards with scores, post event messages to threads.
  Current scheduler already posts pre-match cards with 15-min window check.
- Enhance existing `src/handlers/chat.js`: detect broadcast thread via stored map, build match
  context string from score cache, pass it to `ask(text, matchContext)` using the existing
  `context` parameter (no changes to `ground.js` signature needed). Current handler already
  handles @mentions (in-thread) and DMs.
- Poller lifecycle (to be built in `src/live-data/poller.js`): check `/fixtures?date=today`
  every 15 minutes to detect upcoming matches; when any match is live, switch to 15s polling
  of `/fixtures?live=all` (via existing `fetchLiveFixtures()` in `src/live-data/api.js`);
  stop aggressive polling when no live matches remain.
- Slack rate limiting: rely on Bolt's built-in `@slack/web-api` retry-after handling for
  429 responses (automatic). On top of that, apply application-level ordering:
  1. User-initiated responses: await immediately (highest UX priority)
  2. Broadcast card updates: fire sequentially with small delay between
  3. Event thread posts: batch with staleness check (drop if >30s old)
  During peak (4 simultaneous matches), Bolt's auto-retry handles any 429s transparently.
  No custom queue needed - just ordering discipline in the event processing loop.
- Existing `scripts/sync-data.js`: manual script with subcommands (teams, fixtures, players,
  fixture-events, all) to fetch real data from the football API and write to `src/data/` JSON
  files. For fixture-events: two-step approach fetches by date to discover finished fixtures
  (API is source of truth for status), then fetches each individually via `/fixtures?id=` for
  full events/statistics. Not called during live operation.
- Data layout:
  - `src/data/fixtures.json` - tournament schedule with scores (72 entries: id, dateAndTimeInUTC,
    stage, group, externalId, status, teams with homeTeamId/homeTeamExternalId/awayTeamId/awayTeamExternalId,
    finalScore, score object with halftime/fulltime/extratime/penalty)
  - `src/data/fixtures/{id}-{homeTeamId}-{awayTeamId}.json` - per-fixture detail (score,
    events, statistics); no lineups; created via sync script for finished fixtures
  - `src/data/teams.json` - 48 real teams (id, name, group, rank, points, goalsDiff, externalId)
  - `src/data/players/{teamId}.json` - per-team squads (~26 players each)
- Caching and data priority:
  - Live data from the football API cached in-memory = source of truth during active fixtures
  - Only poll when fixtures are in progress (no API calls when nothing is live)
  - Reuse cached data for grounding, commands, and broadcast updates (one API call serves all)
  - When fixture finishes: app does nothing (no auto-write to disk)
  - `npm run sync:fixture-events` (manual) writes `fixtures/{id}-{home}-{away}.json` for
    finished fixtures (two-step: fetch by date to discover finished, then fetch each by ID for full detail)
  - Query priority: live cache -> fixtures/{id}-{home}-{away}.json on disk -> fixtures.json schedule
- Static data (teams, players) fetched once by sync script, committed to repo
- Event diffing: compare cached events array between polls, broadcast only new events
- ID strategy: keep sequential IDs (1, 2, 3...) as canonical. Each fixture entry in
  `fixtures.json` has an `externalId` field:
  `{ "id": 1, "externalId": 1489369, ... }`
  `src/live-data/mapper.js` builds an `externalFixtureIdToInternalId` Map on startup.
  Teams use 3-letter FIFA codes internally (e.g. "MEX", "RSA") with numeric `externalId`
  for API lookups. `externalTeamIdToTeamId` Map populated via `registerTeamMapping()`.
  Fixture files named `{id}-{homeTeamId}-{awayTeamId}.json` (e.g. `1-MEX-RSA.json`).
- Thread context via Slack message metadata: when broadcasting a match card, attach the
  matchId as Slack message metadata (Slack supports custom metadata on messages). When the
  bot is @mentioned in a thread, read the parent message's metadata to identify the match.
  This survives dyno restarts, deploys, and scaling - no file persistence needed. Also
  maintain an in-memory `{messageTs -> matchId}` cache for fast lookups (rebuilt from Slack
  on restart if needed, but metadata is the source of truth).
- Resilience and degradation:
  - On API failure: exponential backoff (15s -> 30s -> 60s -> 120s max)
  - After 5 consecutive failures: mark data as stale, log warning
  - Stale data still served to users (with staleness note in AI context)
  - Automatic recovery on next successful poll (reset backoff)
  - API client must never log authentication headers/keys in error output

## Technical Constraints

- Node.js 22+ (native fetch built-in - no HTTP dependency needed)
- the football API paid tier required for live data (target ~$50-100/mo tier; exact plan TBD
  based on their current WC 2026 offering). Must handle HTTP 429 responses (respect
  `Retry-After` header, fall back to cached data).
- Polling interval: 15s during live matches for near-real-time event detection
- In-memory cache (resets on dyno restart - re-polls on start)
- Must maintain all existing guardrails (PII masking, toxicity, audit)
- Broadcast card updates require `chat:write` scope (already granted)
- Thread detection uses Slack message metadata (matchId stored on broadcast messages)
- Requires `metadata.event_type` registration in Slack app manifest
- API key stored in Heroku config var `FOOTBALL_API_KEY`

## Design Decisions

- football API as data provider: most popular football API, largest developer community,
  confirmed World Cup 2026 coverage, well-documented endpoints with events/statistics/squads.
- Native fetch (Node 22): zero new dependencies, matches existing `src/mia/client.js` pattern
- In-memory cache over Redis: keeps the app simple and dependency-free for a showcase
- Edit messages over re-post: cleaner UX, less channel noise, Slack best practice
- 15s poll interval: fast enough for near-real-time event detection in Slack
- Event diffing: compare `events[]` array length/content between polls to detect new events
