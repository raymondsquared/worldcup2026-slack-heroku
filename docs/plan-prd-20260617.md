# PRD Plan: Web Search, Personas, Card Redesign, and Post-Match Summary

Date: 2026-06-17

Iteration 3 of the World Cup 2026 Slack app - adding intelligent web search,
per-thread commentary personas, redesigned live/final broadcast cards, incomplete
event filtering, and auto-generated AI match recaps.

## Objectives

### Bug Fixes

1. Incomplete Event Filtering: The football API sometimes sends duplicate
   events - first with player `null`/empty/`"Unknown"`, then seconds later with
   the real player name. The differ treats these as two separate events (different
   composite key), causing double-posts. Fix: before broadcasting, check
   `playerName` - if null/undefined/empty/"unknown" (case-insensitive), log to console and skip.
   Exception: Var events always broadcast regardless of player (VAR decisions
   are about the ruling via `event.detail`, not a specific player). The complete
   event with real player data arrives on the next poll and gets broadcast
   normally.

2. Live Card Elapsed-Time Refresh: The broadcast live card currently only
   updates when the score changes (`scoreChanged`). During goalless periods the
   elapsed minute freezes, making it look broken. Fix: refresh the card on every
   poll cycle (every ~15s) so the minute ticks up even without score changes.

### New Features

1. Broadcast Card Redesign: Replace the current single-line card with a
   structured 3-block layout using Slack Block Kit:

   - Header block: Status indicator (🟢/🔴) + full status text (no
     abbreviations - "First Half", "Second Half", "Halftime", "Extra Time",
     "Match Finished") + elapsed minute when applicable
   - Section block: Flag emoji + team name + score + team name + flag emoji
     (score as the hero visual element)
   - Context block: Goal scorers line - flag emoji before each scorer name,
     home scorers left of `·`, away scorers right, with minutes

   Examples:

   ```text
   🟢 First Half · 17'
   🇦🇷 Argentina  1 - 0  Algeria 🇩🇿
   ⚽ 🇦🇷 Messi 17'
   ```

   ```text
   🟢 Second Half · 67'
   🇦🇷 Argentina  2 - 1  Algeria 🇩🇿
   ⚽ 🇦🇷 Messi 17', 🇦🇷 Álvarez 45+2' · 🇩🇿 Mahrez 38'
   ```

   ```text
   🔴 Match Finished
   🇦🇷 Argentina  3 - 1  Algeria 🇩🇿
   ⚽ 🇦🇷 Messi 17', 🇦🇷 Álvarez 45+2', 🇦🇷 Álvarez 78' · 🇩🇿 Mahrez 38'
   ```

2. Match-End Final Card: When a match finishes, the football API drops it
   from `/fixtures?live=all`, the poller switches to passive mode, and the card
   remains frozen at the last live state. Add explicit end-of-match detection
   that updates the card to 🔴 + full "Match Finished" status with final score
   and complete scorer list.

3. Per-Thread Match Personas: Add commentary personas (sporty, funny,
   serious) that change the tone of AI responses within a match thread. The
   persona is set per-thread when the broadcast card is posted (random or
   configurable) or changed via a reaction/command in the thread. Different
   matches can have different vibes. This is a system prompt modification only -
   no new data sources needed, same grounded data with different personality.

4. Web Search (Post-LLM Fallback): Integrate web search as a fallback when the
   LLM can't answer from local data. Flow: call MIA with local grounding first.
   If response indicates "I don't know" / low confidence, then call the web
   search API, sanitize results, and call MIA again with web context included.
   This means web search only fires when there's a genuine knowledge gap - no
   heuristic classifier needed. Full guardrail pipeline applies to web-sourced
   content. Trade-off: 2 LLM calls on fallback path (slower), but zero
   false-positive searches.

5. Post-Match AI Summary: After the final card is posted, automatically
   generate an AI-written match recap and post it to the match thread. Uses MIA
   grounded on the match's full event data (goals, cards, subs, statistics) to
   produce a narrative summary - e.g., "Mexico dominated the opening half,
   Quinones breaking the deadlock at 23'..." The persona (if set) carries through
   to the recap tone. No new APIs needed - pure MIA + existing events data.

## Scope

### In Scope

- Web search API client with input sanitization (strip PII, prompt injection markers)
- Post-LLM fallback detection ("I don't know" / low confidence response check)
- Web result sanitization layer (treat fetched content as untrusted data)
- Source citation in AI responses ("According to...")
- Rate limiting / cost guard for web search (max N calls per minute)
- Three personas: sporty, funny, serious (system prompt variants)
- Persona stored in Slack message metadata on the broadcast card
- Persona detection from thread parent metadata (same pattern as matchId)
- Persona selection: random on card post, changeable via thread command
- Event filter: skip broadcast when playerName is null/undefined/empty/"unknown" (case-insensitive)
  (log only). Exception: Var events always pass through (player irrelevant)
- Card redesign: 3-block layout (header + score section + scorers context)
- Full status text in header (no abbreviations: "First Half", "Halftime", etc.)
- Flag emojis on score line and before each scorer name in context
- Scorers grouped by team: home goals · away goals (with minutes)
- Card refresh on every active poll cycle (not just on score change)
- Match-end detection via status transition (live -> finished)
- Final card: 🔴 "Match Finished" + score + complete scorer list
- Auto-generated match recap posted to thread after final card
- Recap grounded on full match events (goals, cards, subs, statistics)
- Recap uses the thread's persona for tone consistency
- Tests for all new code paths

### Out of Scope

- Web search as always-on augmentation (only fallback for unanswerable queries)
- Persistent search result caching across dyno restarts
- User preference for enabling/disabling web search
- Streaming web search results (single request-response)
- Per-user persona preferences (persona is per-thread only)
- Custom/user-defined personas beyond the three built-in ones
- Persona affecting broadcast event alerts (only affects AI chat responses)
- Match recap editing or regeneration by users
- Recap for matches that were not actively tracked (only live-tracked matches)
- WebSocket-based live updates (staying with 15s polling)

- Functional Requirements
- Technical Requirements
- Constraints and Assumptions
- Tasks section (continuing from iteration 2)

## Key Decisions (will document as ADR)

- ADR-3: Web search flow decision. Compares 3 approaches (pre-LLM intent gate,
  post-LLM fallback, LLM tool use). Decision: post-LLM fallback - call MIA
  first, only search on genuine "I don't know" responses. Includes full
  comparison table of trade-offs (accuracy, latency, cost, security, complexity).
  Web content treated as untrusted data, never instructions.

## Risks

- Web search results may contain prompt injection attempts -> mitigated by
  existing sanitization + treating results as data with clear delimiters
- Increased API costs from web search calls -> mitigated by post-LLM fallback
  (only searches when MIA can't answer from local data) and rate limit cap
- Persona could produce inappropriate tone for sensitive events (injuries) ->
  mitigated by keeping personas light and professional within each style
- Card refresh every 15s increases Slack API calls -> mitigated by only refreshing
  cards for matches that are actually live (typically 1-4 matches)
- Match-end detection depends on seeing the transition in a poll -> mitigated by
  also checking the date-based endpoint during passive checks

## Execution Order

Bugs first, then new features. Sequential execution (no parallel).

## Alignment

- Extends existing trust architecture (zero data retention for web content,
  sanitization, toxicity filtering on web results)
- OWASP LLM Top 10: addresses Supply Chain (approved web search provider),
  Data Poisoning (untrusted web content handling), Prompt Injection (web content
  as data not instructions)
- Persona leverages existing Slack metadata pattern (same as matchId) - no new
  infrastructure, just extends the metadata payload
- Card redesign uses standard Block Kit blocks (header, section, context) - no
  custom attachments or unfurls
- Continues the pattern of progressive enhancement without rewriting
