# PRD Plan: Daily Highlight Playlist Review

Date: 2026-06-19

Iteration 4 of the World Cup 2026 Slack app - adding a daily automated review
of the official FIFA highlight playlist from the previous day's matches, with
fixture data enrichment.

## Objectives

### New Feature

1. Daily Playlist Review: Each day at 12:00 UTC, the app fetches the official
   FIFA World Cup 2026 highlight playlist, identifies highlight
   videos published for the previous day's matches, matches them to local
   fixture data, updates `fixtures.json` entries with highlight URLs, and posts
   an AI-generated daily highlight digest to the broadcast channel.

   The flow:

   ```text
   Scheduler (daily at 12:00 UTC) -> Fetch playlist items via API
     -> Paginate all items (50/page)
     -> Filter videos with publishedAt in previous UTC day
     -> Match videos to fixtures (algorithm below)
     -> Update fixtures.json with highlightsURL field (upsert)
     -> Generate AI digest (MIA grounded on match results + video list)
     -> Post digest to broadcast channel
   ```

   This gives users a single daily touchpoint summarizing yesterday's action
   with direct links to official FIFA highlights - no need to hunt through
   video platforms manually.

## Video-to-Fixture Matching Algorithm

FIFA's official title format is consistent:
`"TeamA v TeamB | Match Highlights | FIFA World Cup 2026"`

Matching logic (pseudo-code):

```text
For each video published yesterday:
  1. Extract candidate team names from title:
     - Split on " v ", " vs ", " - " (common separators)
     - Take the first segment before " | " as the matchup portion
     - Normalize: strip diacritics, lowercase
  2. For each fixture played yesterday (UTC date match on dateAndTimeInUTC):
     - Check if BOTH home team name AND away team name appear in the title
     - Match against: teams.json full name, team ID (FIFA code), and
       countries.json aliases (all case-insensitive, diacritic-stripped)
  3. Scoring / tie-breaking:
     - Exact team name match scores higher than alias match
     - If a video matches multiple fixtures (unlikely): pick the fixture
       whose date is closest to the video publishedAt
     - If a fixture matches multiple videos: take the one whose title
       contains "Highlights" (skip "Extended", "Full Match", "Press Conf")
  4. Skip videos that:
     - Don't contain "Highlight" in title (filters out interviews, recaps)
     - Match zero or only one team (log as unmatched for observability)
```

## Fixture Data Update Strategy

- `highlightsURL` is an optional string field on fixture entries
- Upsert behavior: if the field already exists, overwrite (FIFA may
  re-upload a corrected highlight); if absent, add it
- Atomic write: read `fixtures.json` -> merge `highlightsURL` into matched
  entries -> write to a temp file -> rename over original (prevents partial
  writes on crash). Same directory ensures same filesystem for atomic rename.
- Serialized access: the highlights job and sync scripts both run on the
  single-process worker dyno. To prevent async interleaving, the write
  operation uses a module-level mutex (simple promise queue) so only one
  read-modify-write cycle runs at a time. Sync scripts run via CLI (separate
  process), so no in-process conflict; file-level atomicity (temp + rename)
  handles that case.
- Rollback: if the playlist fetch or matching fails, no write occurs (fail
  before write). If the write succeeds but digest fails, the data is still
  correct (digest is cosmetic)

## Date Boundary Logic

- "Previous day" means the previous UTC calendar day (00:00:00Z to
  23:59:59Z)
- Video filtering: `publishedAt` (UTC timestamp from playlist API) falls
  within the previous UTC day
- Fixture filtering: match on kickoff time (`dateAndTimeInUTC`) falling
  within the previous UTC day (not on event times - a match starting at
  23:00 UTC counts as that day even if it ends after midnight)
- Running at 12:00 UTC ensures all Americas time zones (UTC-5 to UTC-8) have
  completed their evening matches and FIFA has had 6+ hours to publish
  highlights
- Missed run on restart: persist `lastRunDate` to a file
  (`src/data/.last-highlights-run`). On startup, read it; if the app
  restarted after 12:00 UTC and hasn't run today, execute immediately.
  Prevents both missed runs and duplicate runs.

## Deduplication and Retry Logic

- The job is idempotent: re-running for the same day overwrites the same
  `highlightsURL` values (upsert) and re-posts the digest. To prevent
  duplicate digest posts, the `lastRunDate` file tracks which UTC date was
  last processed successfully.
- If the digest Slack post fails but `highlightsURL` was written, the job
  marks itself as incomplete (does not update `lastRunDate`). Next scheduler
  tick will retry the full job for the same date, re-matching and re-posting.
  Since fixture writes are upserts, this is safe.
- Processed video IDs are not tracked separately - the date filter +
  title matching is deterministic, so re-runs produce identical results.

## Success Criteria

- SC-1: Daily digest posts to broadcast channel at 12:00 UTC (within 2 min)
  on days when previous-day highlights exist
- SC-2: At least 90% of highlight videos are correctly matched to fixtures.
  Automated validation: log `matched_count / total_published_count` each run.
  Alert (console.warn) if ratio drops below 0.9
- SC-3: Matched fixtures have `highlightsURL` persisted in `fixtures.json`
  within the same job run
- SC-4: No digest posted on rest days (zero highlights found = zero posts)
- SC-5: Playlist API usage stays minimal (single daily fetch with pagination)
- SC-6: Digest is engaging, concise (2-4 paragraphs), includes all matched
  highlights with links, grounded on actual scores/events

## Scope

### In Scope

- Playlist API client (fetch playlist items) with API key auth
- Pagination support (handle playlists exceeding 50 items)
- Daily scheduler (configurable time via env var, default 12:00 UTC)
- Startup missed-run detection (run immediately if missed today's window)
- Video-to-fixture matching with team name normalization and alias support
- Fixture data enrichment: add optional `highlightsURL` field to
  `fixtures.json`
- AI-generated daily digest: MIA produces a summary with highlight links
- Digest posted to broadcast channel as a standalone message (not threaded)
- Sync script extension: `npm run sync:highlights` for operator backfill
  (fetches full playlist, matches all videos regardless of date)
- Configurable playlist ID via env var (`HIGHLIGHTS_PLAYLIST_ID`)
- Configurable API key via env var (`HIGHLIGHTS_API_KEY`)
- Error handling: graceful degradation if playlist API fails (log, skip,
  retry next day)
- Observability: log matched/unmatched video counts, API usage
- Tests: unit tests for matcher (team name variations, edge cases),
  integration tests with mock API responses, digest generation tests
- Guardrails: video titles sanitized before MIA context, audit trail for
  digest

### Out of Scope

- OAuth or user-specific playlists (API key auth only, public playlist)
- Video content analysis or transcription (title-based matching only)
- Embedding video players in Slack (link unfurling handles this natively)
- User preferences for digest timing or opt-out
- Real-time highlight notifications per match (daily batch only)
- HTML scraping or RSS feeds (proper API only)
- Custom highlight playlists beyond the configured one
- Highlight links surfaced in AI chat responses (future iteration)
- Multi-language digest support

## Key Decisions

- No ADR needed for this iteration. The approach is straightforward:
  public playlist API (official, quota-based, no scraping), daily batch
  (not real-time), title-based matching (reliable for official FIFA naming
  conventions). Playlist ID and API key are configurable via env vars.

## Risks

- Playlist API unavailable or quota exceeded: mitigated by graceful
  degradation (log error, skip, retry next day)
- FIFA changes video title format: mitigated by fuzzy matching (full names,
  FIFA codes, aliases), logging unmatched videos for manual review
- Playlist may not have highlights published by 12:00 UTC: mitigated by the
  12-hour buffer since last match ends (~05:00 UTC latest). If highlights are
  late, they'll be picked up on the next day's run.
- API key requires provider account setup: documented in setup, same pattern
  as existing `FOOTBALL_API_KEY`
- App restart causes missed run: mitigated by startup missed-run detection
- FIFA playlist ID changes: mitigated by configurable env var
  (`HIGHLIGHTS_PLAYLIST_ID`)
- Video matches wrong fixture (false positive): mitigated by requiring BOTH
  teams in title + "Highlight" keyword filter + date constraint

## Error Handling

- API 429 (rate limit): log warning, skip this run, retry next day
- API 5xx (service unavailable): log error, skip, retry next day
- Playlist empty or not found: log warning, skip digest, no crash
- Zero highlights matched: log info, skip digest post (no empty messages)
- AI digest generation fails: log error, still persist highlightsURL values
  (data enrichment is independent of digest post)
- Slack post fails: log error, highlightsURL values still persisted (data
  not lost)

## Execution Order

Single feature, broken into logical implementation tasks with clear
dependencies: Playlist client -> Matcher -> Fixture enrichment -> AI digest ->
Scheduler -> Manual sync command.

## Alignment

- Extends existing trust architecture (sanitize external content, audit trail)
- Uses same broadcast channel infrastructure (no new Slack setup)
- Follows existing data patterns (enriching fixtures.json like sync scripts do)
- MIA generates the digest text (demonstrates content generation use case)
- OWASP LLM Top 10: Supply Chain (approved API, official playlist), Data
  Poisoning (video titles sanitized before LLM context)
