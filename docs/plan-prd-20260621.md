# PRD Plan: Live-Data Reliability and Data-Model Consistency

Date: 2026-06-21

Iteration 5 of the World Cup 2026 Slack app - primarily a reliability iteration
that hardens the live-broadcast path discovered during real match-day operation,
plus a data-model hygiene change aligning internal team ids to FIFA codes, a
data-consistency fix resolving event player names to the canonical squad name,
and one MIA grounding improvement that replaces the hand-coded keyword
router with model-driven tool calling over the local data accessors.

## Objectives

### Reliability Fixes

1. Live-card correctness: the live match card showed "No goals yet" for an
   entire match even while goal alerts posted correctly in the thread. Every
   live event resolved to a null team, so the scorers line could never group
   goals to a side. Resolve the team on every event at runtime and source the
   card and recap from the Tier-1 cache merge.

2. Precise kickoff activation: the poller stayed in 15-minute passive mode
   through kickoff and only flipped to 15-second active polling on the next
   passive tick, leaving a 10-14 minute blind window at the start of matches.
   Schedule a one-shot timer for the exact next kickoff so the first card
   update lands within 15s of kickoff with zero wasted API calls.

3. Scheduled daily restart: Heroku cycles worker dynos roughly every 24 hours
   at an unpredictable time. A restart inside a pre-match window or just after
   full time re-posts a card or recap because the dedup state is in memory.
   Schedule an in-process restart at a known-safe UTC hour so the 24h cycle
   lands in a window with no matches.

4. Live card transition at kickoff: the match card stayed on its pre-match
   "Fixture Starting Soon!" layout after kickoff and only switched to the live
   score card on the first score/elapsed change - which, for the first goal,
   made it look like the card only updates "when the first event comes through".
   The differ's first-observation restart guard seeds dedup state and emits no
   diff, and `updateCard` only fires on a change. Emit a one-time initial diff on
   first observation (no historical events re-broadcast) so the card transitions
   within one poll of going live.

### Polish and Consistency

1. Event team attribution: threaded event alerts (cards, subs, missed
   penalties) named only the player and minute, so a viewer could not tell
   which team the event belonged to. Prefix the team flag onto goal, card,
   sub, and missed-penalty alerts (VAR is not team-attributed, left unchanged).

2. Duplicate event alerts: the same substitution or card was posted more than
   once because the differ keyed dedup on the minute and the API revises an
   event's minute between polls. Key dedup on each event's stable identity
   instead (subs on the off/on pair, cards on detail+player; goals keep the
   minute for braces).

3. Team-id alignment: four teams used an internal id that differed from their
   official FIFA code (BOS/CAP/CON/CUR). Rename them so id === fifaCode for all
   48 teams, removing a latent source of confusion and matching bugs.

4. Player-name consistency: the football API labeled the same player differently
   between events (a card read "Carlens Arcus", his sub read "C. Arcus"), and
   the mapper stored each raw API name. Resolve both name fields to the one
   canonical squad name by `externalId` at the mapper seam, and backfill the
   already-synced fixture files.

### Grounding Improvement

1. MIA tool-calling retrieval: the retrieval path is a hand-coded keyword router whose
   three regex branches missed common questions - a named team's next match (it
   injected the global next-3 instead), past results and scorers (no branch at
   all), and a player named without their team (invisible) - so MIA searched the
   web for questions answerable from local data. Replace the router with
   model-driven tool calling: expose a whitelist of read-only tools over the
   existing data accessors and let the model choose which to call, with a direct
   single model call as the fallback. Adding a capability becomes "declare one tool"
   rather than "add another regex branch".

## Why now

These surfaced only once the app went into real match-day use. They are
credibility bugs: a broadcast card that says "No goals yet" at 5-1, or a
duplicate recap, undermines the showcase more than any missing feature. The
fixes concentrate on the most-watched surface (live match threads) during the
exact moments it gets the most attention.

## Root-Cause Summary

The live-card defect was a data-source bug, not a formatter bug. The mapper's
`externalTeamIdToTeamId` map was only seeded by the offline `scripts/sync-data.js`
script, never in the running worker, so every `event.teamId` resolved to null
at runtime. `formatScorers` then matched no goal to either side and fell back
to "No goals yet". The fix seeds the map from `teams.json` at module load,
mirroring how the mapper already seeds `externalFixtureIdToInternalId`.

The event team-attribution feature depends on this fix: before it, every
`event.teamId` was null, so a flag helper would have produced nothing.

## Success Criteria

- SC-1: When a goal is scored, the live card scorers line shows the correct
  scorer grouped under the scoring team within one active poll cycle (~15s)
- SC-2: The poller is in active mode within 15 seconds of kickoff, with no
  10-14 minute blind window at the start of a match
- SC-3: No duplicate pre-match cards or post-match recaps are produced as a
  result of Heroku's 24h dyno cycling
- SC-4: Every threaded event alert (goal, card, sub, missed penalty) identifies
  the team involved via its flag, while VAR alerts remain unchanged
- SC-5: All 48 teams have an internal id equal to their official FIFA code, and
  flag rendering is unchanged by the alignment
- SC-6: An event whose minute is revised by the API between polls is posted to
  the thread exactly once, while a genuine brace still produces two goal alerts
- SC-7: A player is shown under one consistent name across every event in a
  match (the canonical squad name), regardless of how the API labels them
- SC-8: Questions answerable from local data (a named team's next match, a past
  result or its scorers, a player's squad record) are answered from local data
  via tool calls without a web search, while genuinely-absent data still falls
  back to web search
- SC-9: When a match goes live, its broadcast card transitions from the
  pre-match layout to the live score card within one active poll cycle (~15s) of
  the first poll that observes it live, without waiting for a goal or other
  event and without re-posting historical events to the thread

## Scope

### In Scope

- Runtime team-id resolution so live events carry a resolved team
- Live card and recap sourcing live events from the Tier-1 cache merge
- A precise one-shot kickoff activation timer in the poller
- A scheduled in-process daily restart at a safe UTC hour, configurable by env
- Team flag prefix on goal/card/sub/missed-penalty alerts
- A minute-independent event dedup key so API minute drift does not re-post
  the same substitution or card
- Renaming four internal team ids to their official FIFA codes, including the
  associated data content and per-team / per-fixture filenames
- Resolving event player and assist names to the canonical squad name by
  `externalId`, plus a one-time backfill of the already-synced fixture files
- Replacing the keyword router with MIA client-side tool calling: a
  whitelist of read-only tools over the existing data accessors, with a direct
  single model call as the fallback when the tool loop grounds nothing
  (see ADR-6)
- An initial card-refresh diff on first observation of a fixture so the live
  card transitions from the pre-match layout at kickoff without waiting for the
  first event
- Tests covering the above and updated assertions for the renamed ids

### Out of Scope

- Any change to README or architecture diagrams beyond what already documents
  the live path
- Re-rendering already-finished match cards retroactively
- Heroku paid scheduler add-ons or platform-level restart configuration
- Changes to highlight matching or persona behavior
- The Heroku Agents server-side orchestration endpoint (the tool-calling work
  uses the client-side `/v1/chat/completions` loop)
- New per-player stat fields not present in local data (height, age); these
  continue to fall through to web search

## Key Decisions

- The reliability and data-consistency work needs no new ADR. Those changes
  extend existing decisions: ADR-2 (live-data caching and polling tiers) covers
  the mapper seeding, cache-merge sourcing, kickoff timer, the event differ's
  dedup key, and the player-name resolution at the mapper seam; the in-process
  scheduled restart reuses the established in-process scheduler pattern. The
  team-id alignment is a behavior-preserving primary-key rename, not an
  architectural choice. The kickoff card-transition fix (task-35 Part 6)
  likewise extends ADR-2: it adds a one-time initial diff to the existing
  differ -> handleDiffs -> updateCard flow and changes no architecture.
- The MIA tool-calling change proposes ADR-6 (MIA client-side tool calling);
  ADR-5 is already used for trust-layer topic grounding, so this takes the next
  free number. It introduces a new retrieval pattern - the model selects
  read-only tools over the data layer and the client runs an execute-and-feed-
  back loop - that supersedes the keyword router as the primary path,
  so it warrants its own decision record. Client-side tool calling is chosen
  over the Heroku Agents server-side endpoint to keep guardrails in-process and
  minimize change to the existing client.

## Risks

- Restart lands in an unexpected window if match scheduling changes: mitigated
  by `DAILY_RESTART_HOUR_IN_UTC` being configurable; default 11:00 UTC sits in
  the verified 05:00-11:00 no-match window
- A live event without a resolvable external team id renders no flag: handled
  by the empty-prefix guard (no white-flag fallback leak)
- Team-id rename touches filenames and content broadly: mitigated by
  word-boundary grep verification, collision checks against existing ids, and
  validating every renamed JSON parses with all squads/fixtures still resolving
- In-memory timers reset on restart: acceptable because both the kickoff timer
  and restart timer are re-derived on startup
- Goal-minute drift can still re-post a goal because goals keep the minute in
  their dedup key (needed so a brace stays two alerts): accepted as a rare case
  and documented in the code; revisit with a tolerance window only if observed
- The configured inference model may not support function calling: mitigated by
  a one-shot capability probe and the deterministic fallback (a direct single
  model call with no tool context), so an unsupported model changes nothing
- Tool calling adds inference round-trips (more latency and tokens than the
  single-call direct path): bounded by a maximum tool-iteration count and a
  total wall-clock budget. MIA's `ask()` runs on the asynchronous Events API
  reply path (mention / DM via `say()`), not the slash-command path, so there is
  no 3-second request deadline; the cost is the user waiting longer for the bot
  reply
- A tool dispatch surface could widen the attack surface: mitigated by a fixed
  read-only whitelist (`dispatchTool` rejects unknown names and never evaluates
  arbitrary input), defensive argument parsing, and the unchanged PII/toxic
  guardrails wrapping the loop
- The initial card-refresh diff could re-broadcast historical events or post a
  spurious card: mitigated by keeping `newEvents` empty on first observation (no
  event re-broadcast) and by `updateCard` already no-opping when no message
  timestamp is tracked (a worker starting after kickoff falls through to the
  existing first-event card fallback rather than double-posting)

## Alignment

- Extends ADR-2 live-data tiers (cache merge already in `getFixtureById`)
- Reuses the existing in-process scheduler pattern (no new dependency)
- Keeps flag rendering keyed off `flagISO`, so the id rename is invisible to
  users
- Preserves the trust architecture: VAR alerts stay un-attributed, the
  tool-calling path adds no new external data sources (tools are read-only over
  existing local accessors) and keeps the PII/toxic guardrails and web-search
  fallback intact
- Proposes ADR-6 for the tool-calling retrieval pattern; the tools wrap the
  same data accessors the commands already use, so the data layer is unchanged
