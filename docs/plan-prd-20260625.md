# PRD Plan: Football API MCP Server

Date: 2026-06-25  
Agent: Product Manager

## Problem Statement

The Heroku Managed Inference Agents (MIA) service powering the Slack bot's AI
features currently has no mechanism to access live World Cup 2026 data from the
Football API. When users ask real-time questions like "What matches are live?"
or "What's the current score?", the AI cannot fetch current data and must rely on:

1. Static JSON files - Outdated after deployment, requires manual sync scripts
2. Live cache - Only available through the bot's internal poller, not accessible
   to MIA
3. No data - Cannot answer time-sensitive questions accurately

This limits the AI's usefulness during match days and requires developers to
manually sync data before testing AI features.

## Proposed Solution

Create an MCP (Model Context Protocol) server that wraps the Football API,
following the Heroku MCP pattern (`heroku/mcp-code-exec-node`). The MCP server will:

1. Deploy to the existing Heroku app as an additional Procfile process
   (`mcp-football`)
2. Register with Heroku Managed Inference - Tools automatically available to MIA
3. Expose 5 read-only tools that wrap existing `src/live-data/api.js` functions
4. Run in STDIO mode for security and isolation (Heroku best practice)

The AI can then invoke tools like `football_get_live_fixtures` through standard
MCP protocol, enabling real-time answers without custom integration code.

## Target Audience

Primary:

- Heroku MIA (AI agent) - Consumes tools to answer Slack user questions
- Slack users - Benefit from accurate, real-time AI responses about matches

Secondary:

- Developers - Test AI features with live data without running sync scripts
- Future AI agents - Other systems using the same inference endpoint

## Scope

### In Scope (Phase 1)

1. MCP Server Implementation
   - Create `src/mcp/football-server.js` following MCP SDK patterns
   - Implement 5 tools (see Tools section below)
   - STDIO transport mode (recommended by Heroku)
   - Error handling with API key sanitization

2. Heroku Integration
   - Add `mcp-football: node src/mcp/football-server.js` to Procfile
   - Configure buildpacks (if needed - we already have Node.js)
   - Attach Heroku Inference add-on to app (if not already)
   - Register MCP server with inference endpoint

3. Security & Guardrails
   - Read-only operations (GET requests only)
   - API key redaction from error messages
   - 10-second timeout on all requests
   - No PII handling (World Cup data is public)
   - Follows existing AGENTS.md guardrails

4. Testing
   - Manual test with MCP Inspector (local)
   - Verify registration with `/v1/mcp/servers` endpoint
   - Test tool calls via curl to inference API
   - Integration test: Ask AI a live match question in Slack

5. Documentation
   - Update README with MCP server info
   - Add example tool calls to docs
   - Document deployment steps

### Out of Scope (Defer to Future)

- Write operations - No mutations to Football API
- Caching layer - Use Football API directly (rate limits assumed sufficient)
- Batch operations - One fixture/team per tool call
- Additional MCP servers - Highlights, Web Search (separate projects)
- SSE/HTTP transport - STDIO only for phase 1
- Local development MCP setup - Heroku deployment only

## MCP Tools Specification

Following the pattern from `heroku/mcp-code-exec-node`, each tool will be
registered with name, description, and JSON schema for parameters.

### Tool 1: `football_get_live_fixtures`

Description: Get all currently live World Cup 2026 fixtures with real-time scores
and match status
Parameters: None
API Call: `GET /fixtures?live=all` (filtered to World Cup)
Returns: Array of fixtures with id, teams, score, status, league, round
Use Case: "What matches are happening right now?"

### Tool 2: `football_get_fixtures_by_date`

Description: Get World Cup 2026 fixtures scheduled for a specific date
Parameters:

- `date` (string, required): Date in YYYY-MM-DD format (e.g., "2026-06-15")

API Call: `GET /fixtures?date={date}` (filtered to World Cup)
Returns: Array of fixtures for that date with venue, teams, status
Use Case: "Show me matches on June 20th"

### Tool 3: `football_get_fixture_details`

Description: Get detailed information for a specific fixture including events,
lineups, and statistics
Parameters:

- `fixture_id` (number, required): The external fixture ID from Football API

API Call: `GET /fixtures?id={fixture_id}`
Returns: Full fixture object with goals, cards, substitutions, lineups, team statistics
Use Case: "Tell me about the USA vs Mexico match" (AI looks up fixture ID first,
then fetches details)

### Tool 4: `football_get_team_squad`

Description: Get the squad/roster for a World Cup team
Parameters:

- `team_id` (number, required): The external team ID from Football API

API Call: `GET /players/squads?team={team_id}`
Returns: Team info + array of players with names, numbers, positions
Use Case: "Who plays for Argentina?"

### Tool 5: `football_get_standings`

Description: Get World Cup 2026 group standings
Parameters:

- `league_id` (number, optional, default: 1): League ID (World Cup)
- `season` (number, optional, default: 2026): Season year

API Call: `GET /standings?league={league_id}&season={season}`
Returns: League standings with groups and team rankings, points, goal difference
Use Case: "What are the Group A standings?"

## Key Architectural Decisions (ADRs Required)

These decisions will be documented as ADRs during PRD generation:

1. ADR: STDIO vs SSE/HTTP Transport
   - Decision: STDIO mode only (align with Heroku best practice)
   - Rationale: Better isolation, security, lower cost (no always-on web dyno)
   - Trade-off: Requires one-off dyno execution (handled by Inference orchestration)

2. ADR: Tool Naming Convention
   - Decision: Prefix all tools with `football_` (e.g., `football_get_live_fixtures`)
   - Rationale: Namespace collision prevention with future MCP servers (highlights, search, etc.)
   - Alternative considered: No prefix (rejected - could conflict with built-in tools)

3. ADR: Direct API Calls vs Reuse Existing Code
   - Decision: Reuse `src/live-data/api.js` functions where possible
   - Rationale: DRY principle, consistent error handling, already tested
   - Trade-off: Slight coupling to existing code structure

4. ADR: Error Handling Strategy
   - Decision: Return structured error responses to AI, log sanitized errors
   - Rationale: AI can decide whether to retry, inform user, or use fallback
   - Example: `{error: "Request timed out", retriable: true}`

5. ADR: MCP SDK Version
   - Decision: Use `@modelcontextprotocol/sdk` (official SDK)
   - Rationale: Standard implementation, maintained by Anthropic, matches Heroku examples
   - Alternative considered: Custom implementation (rejected - unnecessary complexity)
