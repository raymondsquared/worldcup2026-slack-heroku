## Description

Replace mock data with real World Cup 2026 data and create sync scripts that pull from
the football API. The first iteration used intentionally simple, hardcoded mock data to
get the app working end-to-end quickly. Now that we're integrating with a real football
API, the data layer needed to be refactored to handle the more sophisticated data shapes
the API returns - richer fixture objects with nested score breakdowns, external IDs for
cross-referencing, event timelines, and per-team statistics.

The fixtures.json schema now includes `externalId`, `status`, `finalScore`, `score`
(halftime/fulltime/extratime/penalty), and `teams` with both internal IDs and externalIds.

This task delivered:

1. `scripts/sync-data.js` with subcommands: teams, fixtures, players, fixture-events, all
2. Updated `src/data/index.js` with tiered query resolution per ADR-2: live cache ->
   per-fixture detail file -> static schedule
3. `src/data/fixtures/` directory for per-fixture detail files (`{id}-{homeTeamId}-{awayTeamId}.json`)
4. Multiple npm scripts: `sync:teams`, `sync:fixtures`, `sync:players`, `sync:fixture-events`, `sync`

## Actual Implementation

### scripts/sync-data.js

Subcommands:

- `sync:teams` - fetches `/standings?league=1&season=2026`, rebuilds `teams.json` (48 teams)
  with id, name, group, rank, points, goalsDiff, externalId
- `sync:fixtures` - fetches all fixtures by date range (2026-06-11 to 2026-07-19), rebuilds
  `fixtures.json` from scratch (72 fixtures). Resolves teams by numeric externalId first,
  name-based fallback. Derives group from home team's group in teams.json.
- `sync:players` - fetches `/players/squads?team={id}` for all teams, writes per-team
  `players/{teamId}.json` files with id, name, teamId, position (raw from API), number, externalId
- `sync:fixture-events` - two-step approach:
  1. Fetch by date to discover which fixtures are finished (API is source of truth for status)
  2. Fetch each finished fixture individually via `/fixtures?id=` for full events/statistics
  - Writes `fixtures/{id}-{homeTeamId}-{awayTeamId}.json`
  - Single fixture mode: `npm run sync:fixture-events -- 1`
- `sync` - runs teams -> fixtures -> players -> fixture-events in order

Key helpers:

- `fetchAllFixtures()` - iterates tournament date range, filters for World Cup fixtures
- `buildTeamNameMap()` - FIFA codes + existing teams.json for name-based resolution
- `normalizeName()` - strips diacritics for ASCII-only names
- `resolveTeamId()` - name-based fallback lookup
- Rate limiting between API calls (200ms-1000ms pauses)

### src/data/index.js

Tiered resolution (per ADR-2):

- `getFixtureById(id)` - live cache -> detail file -> static schedule
- `getFixtureEvents(id)` - live cache -> detail file -> empty array
- `getLiveScore(id)` - live cache -> detail file -> static schedule -> null
- `setLiveCache(cache)` - registers the live cache module (hook for task-11)
- `loadFixtureDetail(id)` - constructs filename from fixtures.json teams field

Renamed (part of the `match` -> `fixture` rename below): `getAllMatches()` ->
`getAllFixtures()`, `getUpcomingMatches(n, referenceDate)` -> `getUpcomingFixtures(n, referenceDate)`

Unchanged functions: `getGroups()`, `getPlayers(teamId)`, `getTeamName(id)`

### src/live-data/mapper.js

- `externalFixtureIdToInternalId` Map - built on startup from fixtures.json
- `externalTeamIdToTeamId` Map - populated via `registerTeamMapping()`
- `findTeamIdByExternalId()` - resolves API team ID to internal 3-letter code
- `mapFixtureWithEvents(fixture)` - reshapes API fixture to internal format with events + statistics (no lineups)
- Events shape: { type, minute, extraMinute, teamId, teamExternalId, playerExternalId,
  playerName, assistPlayerExternalId, assistPlayerName, detail, comments }
- Statistics shape: { teamId, statistics: [...raw API stats...] }

### Downstream rename: `match` -> `fixture`

Moving to the real API data shape required renaming the data accessors (in `src/data/index.js`,
see above) and updating every consumer. `getAllMatches()` became `getAllFixtures()` and
`getUpcomingMatches()` became `getUpcomingFixtures()`, and consumers switched from the flat
`match.homeTeamId`/`match.awayTeamId` fields to the nested `fixture.teams.homeTeamId`/
`fixture.teams.awayTeamId` shape. User-facing copy changed from "match" to "fixture".

Data file rename:

- `src/data/matches.json` removed; replaced by `src/data/fixtures.json` (new schema)
