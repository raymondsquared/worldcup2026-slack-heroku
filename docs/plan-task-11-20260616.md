# Task 11: Live Data Poller and Cache

## Details

Date: 2026-06-16

Dependencies: task-10

Decisions: none

## Description

Build a live data polling system that fetches in-progress match data from the football API every
15 seconds and caches it in memory. The poller follows a two-phase lifecycle: passive monitoring
(checks `/fixtures?date=today` every 15 minutes) and active polling (checks `/fixtures?live=all`
every 15 seconds when live matches detected). Implements exponential backoff on failures, graceful
degradation when API unreachable, and serves as the foundation for broadcast enhancements in
task-12.

This is a CRITICAL task - without live data infrastructure, the app cannot transition from
static demo to real-time World Cup showcase.

## Acceptance Criteria

### FR-1: Live Match Polling

- (State-driven) While any World Cup match is in progress, the system shall poll the football
  API's `/fixtures?live=all` endpoint every 15 seconds
- (Ubiquitous) The system shall cache the live fixture response in memory, indexed by internal
  fixture ID

### FR-10: Graceful Degradation

- (Unwanted) If the football API is unreachable, the system shall continue functioning using cached
  JSON data from `fixtures.json` and fixture detail files
- (State-driven) While serving stale data due to API failure, the system shall mark cached entries
  with a `stale: true` flag

### TR-4: Exponential Backoff

- (Unwanted) If a polling request fails, the system shall implement exponential backoff: 15s -> 30s
  -> 60s -> 120s (max)
- (Unwanted) If 5 consecutive polling failures occur, the system shall mark all cached data as
  stale

### TR-6: Poller Lifecycle

- (Ubiquitous) The system shall check `/fixtures?date=today` every 15 minutes to detect live
  matches
- (Event-driven) When live matches are detected, the system shall switch to aggressive polling mode
  (15s intervals)
- (Event-driven) When no live matches remain, the system shall return to passive monitoring mode
  (15 minute intervals)

## Implementation Plan

### Step 1: Create Live Cache Module (`src/live-data/cache.js`)

Create an in-memory cache that:

- Stores live fixtures by internal ID (not external ID)
- Tracks polling state (idle/active/backoff)
- Implements exponential backoff timing
- Marks data as stale after 5 consecutive failures
- Exposes `getFixture(id)` for tier-1 resolution in `src/data/index.js`

Key functions:

- `updateFixtures(apiFixtures)` - merge new API data into cache
- `getFixture(id)` - retrieve cached fixture by internal ID
- `markStale()` - flag all cached data as stale after failures
- `getPollingInterval()` - return next poll interval (15s, 30s, 60s, 120s)

### Step 2: Create Live Data Poller (`src/live-data/poller.js`)

Build the poller state machine with two modes:

Passive mode (15 minute interval):

- Calls `fetchFixturesByDate(today)` every 15 minutes
- Filters for World Cup fixtures with a live (in-play) `status.long` value via `isMatchLive()`:
  `First Half`, `Kick Off`, `Halftime`, `Second Half`, `2nd Half Started`, `Extra Time`,
  `Break Time`, `Penalty In Progress`, `Match Suspended`, `Match Interrupted`
- Switches to active mode when live matches found

Active mode (15 second interval):

- Calls `fetchLiveFixtures()` every 15 seconds
- Passes results to cache via `updateFixtures()`
- On API failure: implements exponential backoff via `getPollingInterval()`
- After 5 consecutive failures: calls `markStale()` on cache
- Switches back to passive mode when no live matches

Key functions:

- `start()` - initialize poller, begin passive monitoring
- `stop()` - cleanup timers
- `checkForLiveMatches()` - passive mode handler
- `pollLiveFixtures()` - active mode handler
- `handlePollError(err)` - exponential backoff logic

### Step 3: Integrate Poller Lifecycle with Application

Update `src/index.js`:

- Import `{ start: startPoller }` from `src/live-data/poller.js`
- Import cache and register with data layer via `setLiveCache(cache)`
- Call `startPoller()` after `app.start()` but before scheduler
- The poller runs independently - no Slack client needed

### Step 4: Map API Fixture Response to Cache Format

Update `src/live-data/mapper.js`:

- Add `mapLiveFixture(apiFixture)` function
- Extract: status, score (halftime/fulltime/extratime/penalty), basic stats
- Return flat object compatible with cache storage
- Do NOT include events/lineups/detailed stats (those are task-12)

### Step 5: Add Fixture Date Utility

Create `src/utils/dates.js`:

- `getTodayUTC()` - return YYYY-MM-DD in UTC (for passive polling)
- `isMatchLive(status)` - check if a `status.long` value indicates an in-progress match
  (matches the long live set listed in Step 2)
- Used by poller to determine mode transitions

### Step 6: Environment Validation

Update `src/index.js` validation:

- Do NOT require `FOOTBALL_API_KEY` at startup (app should run without it)
- Poller logs warning if key missing: "Live polling disabled - FOOTBALL_API_KEY not set"
- App continues with static data only

### Step 7: Tests

Create `src/test/live-data-cache.test.js`:

- Cache stores and retrieves fixtures by ID
- Exponential backoff intervals: 15s -> 30s -> 60s -> 120s
- Data marked stale after 5 failures
- Cache clears stale flag on successful poll

Create `src/test/live-data-poller.test.js`:

- Passive mode checks every 15 minutes
- Switches to active mode when live matches detected
- Active mode polls every 15 seconds
- Returns to passive mode when no live matches
- Handles API failures with backoff

Update `src/test/data.test.js`:

- `getFixtureById()` returns live cache data when available
- Falls back to fixture detail file when cache empty
- Falls back to `fixtures.json` when no detail file

### Step 8: Logging

Add console logging for observability:

- "Live poller started (passive mode)" - on `start()`
- "Switching to active polling - N live matches detected" - on mode transition
- "No live matches - returning to passive mode" - on mode exit
- "Poll failed (attempt N/5): [error]" - on failure
- "Marked cache as stale after 5 consecutive failures" - on stale flag

- poller.test.js: Lifecycle state machine (mode transitions, timers, error handling)
- mapper.test.js: Live fixture mapping (status, scores, basic structure)

- Mock `fetchLiveFixtures()` and `fetchFixturesByDate()` from `api.js`
- Verify poller transitions between modes correctly
- Verify cache updates on successful polls
- Verify exponential backoff on failures
- Verify app starts successfully when `FOOTBALL_API_KEY` is unset

- Start app with valid `FOOTBALL_API_KEY`
- Observe console logs showing passive -> active -> passive transitions
- Simulate API failure (invalid key) and observe backoff + stale marking
