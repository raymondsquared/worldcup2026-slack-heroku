# Task 12: Broadcast Enhancements

## Details

Date: 2026-06-16

Dependencies: task-11

Decisions: none

## Description

Enhance the broadcast system to deliver live match updates during active games. Currently,
`src/broadcast/scheduler.js` posts a static "Starting Soon!" card 15 minutes before kickoff.
Task 12 upgrades it to:

1. Attach Slack message metadata (matchId) to broadcast cards so threads can be identified
2. Edit broadcast cards with live scores as goals happen (`chat.update`)
3. Detect new events (goals, cards, subs) by diffing the cache between polls
4. Post event alerts to match broadcast threads in real time
5. Handle Slack rate limits gracefully (drop stale events >30s, rely on Bolt retry-after)

This transforms the broadcast from a one-shot pre-match notification into a live match
experience - the card becomes a live scoreboard and its thread becomes a match commentary feed.

## Acceptance Criteria

- (Event-driven) When the poller updates the cache with new fixture data, the system shall
  compare the new events array against the previously seen events to detect new goals, cards,
  and substitutions.
- (Event-driven) When a new goal is detected, the system shall post a formatted goal alert to
  the match's broadcast thread including scorer name, minute, assist, and updated score.
- (Event-driven) When a new card or substitution is detected, the system shall post a formatted
  event message to the match's broadcast thread.
- (Event-driven) When the score changes for a live match, the system shall edit the original
  broadcast card message with the current score using `chat.update`.
- (Ubiquitous) The system shall attach message metadata (`event_type: "worldcup_match"`,
  `event_payload: { matchId }`) to all broadcast card messages.
- (Ubiquitous) The system shall maintain a `matchId -> messageTs` map so it can post to threads
  and edit cards for active matches.
- (Unwanted) If an event alert is older than 30 seconds by the time it would be posted, the
  system shall drop it silently (log at debug level).
- (State-driven) While the poller is in active mode, the system shall call the differ after each
  successful poll and dispatch any new events to the broadcaster.

## Implementation Plan

### Step 1: Create event differ (`src/live-data/differ.js`)

Compares a fixture's current events against previously seen events using a composite key
Set for deduplication. Each event is keyed by `${type}-${minute}-${playerExternalId}` to
handle API reordering (e.g., VAR-overturned goals inserted retroactively at earlier minutes).

```javascript
// differ.js
// previousKeys: Map<internalId, Set<string>>
// previousScores: Map<internalId, { home, away }>
// diffAll(cacheEntries) -> [{ fixtureId, newEvents, currentScore, previousScore }]
```

The differ holds an internal Map of `fixtureId -> Set<eventKey>` and `fixtureId -> lastScore`.
On each poll cycle, the poller passes the full fixture data through `diffAll(cacheEntries)`
which returns an array of `{ fixtureId, newEvents, currentScore, previousScore }` for fixtures
with changes.

Restart guard: On first poll after startup (when differ has no prior state for a fixture),
seed the key Set and score from the current data WITHOUT emitting events. Only subsequent polls
trigger alerts. This prevents flooding threads with duplicate events after dyno restart.

### Step 2: Enhance broadcast scheduler

Refactor `src/broadcast/scheduler.js` into a richer module:

- Metadata on post: When `chat.postMessage` sends the pre-match card, include
  `metadata: { event_type: "worldcup_match", event_payload: { matchId: fixture.id } }`.
- Track message timestamps: Store `matchId -> messageTs` in a Map (populated from the
  `postMessage` response's `ts` field).
- `updateCard(matchId, score)`: Call `chat.update` with the original `ts` and channel to
  edit the card with the live score. Card shows: "🟢 LIVE · 67' · MEX 2-1 RSA".
- `postEventAlert(matchId, event)`: Post to the thread (`thread_ts` = stored `messageTs`)
  with a formatted event message.
- Staleness check: Before posting an event alert, check if `Date.now() - detectedAt > 30000`.
  If stale, skip it. The `detectedAt` timestamp is set once when the differ runs, shared by all
  events in that poll cycle (so inter-post delays don't push events past the threshold).
- Error handling: Wrap `chat.update` and thread `postMessage` with try/catch. On
  `message_not_found` or `channel_not_found`, remove the stale `ts` from the map and log a
  warning. On other errors, log and continue (don't crash the poll loop).
- Logging: Log each broadcast action at info level - card posted, card updated, event alert
  sent, stale event dropped.

### Step 3: Wire poller -> differ -> broadcaster

After `cache.updateFixtures(cacheEntries)` in `pollLiveFixtures()`:

1. Call `differ.diffAll(cacheEntries)` to get changes
2. For each fixture with `scoreChanged`: call `broadcaster.updateCard(matchId, newScore)`
3. For each `newEvent`: call `broadcaster.postEventAlert(matchId, event)` with
   `detectedAt: Date.now()`

Decoupling pattern: The poller does NOT import the Slack client or broadcaster directly.
Instead, expose a callback: `poller.onChanges(callback)`. In `index.js`, wire the callback to
the broadcaster:

```javascript
poller.onChanges((diffs) => broadcaster.handleDiffs(app.client, diffs));
```

This keeps the poller as a pure data concern (fetch + cache + diff) and the broadcaster as the
Slack-aware layer. The existing `broadcaster.start(client)` already has the client reference.

Priority ordering (TR-7): User responses are handled by Bolt event listeners (synchronous
callbacks) and always run immediately. The poller's post-poll dispatch is async and fires after
the cache update - card updates first, then event alerts with the 500ms inter-post delay. This
natural ordering means user responses are never blocked by broadcast work. Bolt's built-in
retry-after handles any 429s transparently on all three paths.

### Step 4: Format event messages

Create `src/broadcast/format.js` with Block Kit builders:

- `formatGoalAlert(event, score)` -> ⚽ Goal! Player (Assist) · 23' · MEX 1-0 RSA
- `formatCardAlert(event)` -> 🟨 Yellow Card · Player · 45'
- `formatSubAlert(event)` -> 🔄 Sub · Player On ↔ Player Off · 60'
- `formatLiveCard(fixture, score)` -> Updated Block Kit card with live score

### Step 5: Handle edge cases

- Multiple goals in one poll: Process sequentially with short delay (500ms between posts)
- Bot hasn't posted card yet (match started between passive checks): Post an initial card
  with current score, store the `ts`, then continue updating from there
- Dyno restart mid-match: `matchId -> messageTs` map is lost; on restart, post a new card
  for any active matches and start fresh (metadata on the old card still works for thread
  context in Task 13)

## Test Strategy

### Unit tests (`src/test/live-data-differ.test.js`)

- diffAll with no previous state -> seeds keys, returns empty (restart guard)
- diffAll with same events -> returns empty (no changes)
- diffAll with new events -> returns only unseen events (by composite key)
- diffAll handles API reordering (event inserted at earlier minute)
- diffAll detects score change (goals differ from previous)
- diffAll handles multiple fixtures independently
- diffAll ignores fixtures with no internal ID mapping

### Unit tests (`src/test/broadcast-format.test.js`)

- formatGoalAlert returns Block Kit with scorer, minute, score
- formatCardAlert returns yellow/red card message
- formatSubAlert returns substitution message
- formatLiveCard returns updated card blocks with live score

### Integration tests (`src/test/broadcast-live.test.js`)

- Poller poll -> differ detects goal -> broadcaster posts to thread
- Score change -> card updated via chat.update
- Multiple events in one poll -> all posted sequentially
- Stale event (>30s) -> dropped
- Match with no prior card -> new card posted, ts stored
- chat.update fails (message deleted) -> ts removed from map, no crash
- First poll after restart -> seeds differ, no event alerts emitted

### Manual verification

- Start app with live match -> confirm card updates in Slack
- Verify message metadata visible in Slack message JSON
- Confirm thread posting works (event appears under the card)
- Test with API down -> no crash, stale events dropped gracefully
