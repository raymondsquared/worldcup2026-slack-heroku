# Task 13: Thread-Grounded AI

## Details

Date: 2026-06-16

Dependencies: task-12

Decisions: none

## Description

When the bot is @mentioned in a broadcast thread, it should know which match the thread
belongs to and ground its AI response on that match's live data (score, events, teams).
Outside of broadcast threads, existing general grounding continues to work unchanged.

The mechanism: broadcast cards (Task 12) now attach message metadata
(`event_type: "worldcup_match"`, `event_payload: { matchId }`). When the bot is mentioned
in a thread, we read the parent message's metadata to identify the match, build a rich
context string from the live cache + fixture data, and pass it to `ask(text, context)`.

The `ask()` function already accepts an optional `context` parameter that bypasses the
generic `ground()` function. All guardrails (PII masking, sanitization, toxicity, audit)
remain active since they run before and after the MIA call regardless of context source.

## Acceptance Criteria

- (Event-driven) When the bot is @mentioned in a broadcast thread, the system shall read
  the parent message's metadata to identify the match ID.
- (Event-driven) When a match ID is identified from thread context, the system shall build
  a match-specific context string including: team names, current score, elapsed time, match
  status, and recent events (goals, cards, subs).
- (Event-driven) When match context is available, the system shall pass it to `ask(text,
  context)` so the AI response is grounded on live match data.
- (State-driven) While the bot is mentioned outside a broadcast thread (channel mention or
  DM), the system shall use the existing general grounding logic unchanged.
- (Ubiquitous) The system shall maintain all existing guardrails (PII masking, prompt
  defense, toxicity detection, audit trail) for thread-aware AI responses.
- (Unwanted) If metadata lookup fails (parent message deleted, API error), the system shall
  fall back to general grounding rather than erroring.
- (Unwanted) If the match is known but no live data is available (not started, cache empty),
  the system shall return a partial context with team names and status rather than falling
  back to general grounding.
- (Ubiquitous) The system shall update the Slack app manifest to subscribe to the
  `worldcup_match` metadata event type.

## Implementation Plan

### Step 1: Add metadata subscription to Slack manifest

Update `docs/slack-app-manifest.yaml` to register the metadata event type under `features`:

```yaml
features:
  bot_user:
    display_name: World Cup 2026
    always_online: true
settings:
  socket_mode_enabled: true
  event_subscriptions:
    metadata_subscriptions:
      - event_type: worldcup_match
```

This allows the app to read metadata from messages it posted.

### Step 2: Create match context builder (`src/handlers/match-context.js`)

A function that takes a `matchId` and returns a formatted context string for the AI:

```javascript
function buildMatchContext(matchId) {
  // 1. Get fixture data (tier-1 live cache -> tier-2 detail -> tier-3 schedule)
  // 2. Get live score (status, elapsed, home/away goals)
  // 3. Get events (goals, cards, subs)
  // 4. Get team names
  // 5. Format as a structured context string
  // 6. Returns null ONLY if matchId is invalid (no fixture found)
  //    Returns partial context if data is incomplete (e.g., "Not Started")
}
```

The context string format:

```text
Match: Mexico vs South Africa
Status: Second Half (67')
Score: Mexico 2 - 1 South Africa
Events:
- 23' Goal: J. Quinones (assist: E. Lira) - Mexico
- 45' Yellow Card: R. De Paul - South Africa
- 55' Goal: H. Lozano - Mexico
Teams: Mexico (MEX), South Africa (RSA)
```

Partial data handling:

- If `getLiveScore()` returns null (not started): context shows "Status: Not Started" with
  team names and kickoff time, no score or events.
- If `getFixtureEvents()` returns empty: context shows score but "No events yet."
- If `getLiveScore()` returns stale: context appends "(data may be outdated)"

This gives the AI everything it needs to answer "what's the score?", "who scored?",
"any cards?", etc. - even when partial.

### Step 3: Enhance chat handler for thread detection

Modify `src/handlers/chat.js` `app_mention` handler. Important: Update the handler
signature from `async ({ event, say })` to `async ({ event, say, client })` to access
the Slack Web API client for metadata lookup.

Thread detection logic:

1. Check if event has `thread_ts` (it's a reply in a thread)
2. Guard: Only attempt metadata lookup if `event.channel === BROADCAST_CHANNEL_ID`.
   This avoids `missing_scope` errors in private channels and scopes the feature correctly.
3. Call `conversations.history` with `latest=thread_ts, limit=1, inclusive=true` to fetch
   the parent message and read its metadata
4. Extract `matchId` from `message.metadata.event_payload.matchId`
5. If match ID found, call `buildMatchContext(matchId)` and pass to `ask(text, context)`
6. If not in a thread, wrong channel, or no metadata found, fall through to existing
   `ask(text)` behavior (general grounding via `ground.js`)

No in-memory reverse lookup needed. The Slack API call (~100ms) is the single source of
truth for metadata and survives dyno restarts. With at most a few thread mentions per minute,
this latency is negligible compared to the MIA call (~500-1000ms).

### Step 4: Validate matchId input safety

Since `matchId` comes from Slack metadata (which the bot itself wrote), it should always be
a valid integer. However, defensively:

- `buildMatchContext` accepts only numbers: `if (typeof matchId !== 'number') return null`
- `getFixtureById(matchId)` returns null for unknown IDs (no file system traversal risk
  since `loadFixtureDetail` constructs paths from the fixture's own team data, not the
  raw ID input)

## Test Strategy

### Unit tests (`src/test/match-context.test.js`)

- buildMatchContext returns formatted string with score, events, teams
- buildMatchContext with live cache data includes elapsed and status
- buildMatchContext with no events returns "No events yet."
- buildMatchContext for unknown fixture returns null
- buildMatchContext with stale cache includes staleness note
- buildMatchContext for not-started match returns partial context (teams + kickoff)
- buildMatchContext rejects non-numeric matchId (returns null)

### Unit tests (`src/test/chat.test.js` - additions)

- @mention in broadcast thread -> detects match, uses match context
- @mention in broadcast thread -> metadata lookup via Slack API
- @mention in broadcast thread with no metadata -> falls back to general grounding
- @mention in non-broadcast channel thread -> uses general grounding (skips lookup)
- @mention in channel (not thread) -> uses general grounding
- DM -> uses general grounding (unchanged)
- Thread-aware response still applies all guardrails (PII, toxicity)
- Invalid matchId from metadata -> falls back to general grounding

### Manual verification

- @mention bot in a live match thread -> responds with match-specific answer
- Ask "what's the score?" in thread -> returns live score
- Ask "who scored?" in thread -> lists goal scorers
- @mention bot in a non-broadcast thread -> generic response
- Restart dyno, ask in old thread -> still detects match via metadata API
