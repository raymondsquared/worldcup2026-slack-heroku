# Task 14: Schedule Command Update

## Details

Date: 2026-06-16

Dependencies: task-13

Decisions: none

## Description

Update the `/worldcup2026 schedule` command to show live scores for any in-progress matches
alongside the next upcoming matches. Currently, the command only shows the next 3 upcoming
(not-started) fixtures. With live data now available (Task 11/12), users should see a "Live
Now" section when matches are actively being played.

Liveness is detected using the canonical `isMatchLive()` helper matching the football API's
long status names (the form already stored in the live cache). This task also aligns the
existing poller to that same helper so there is a single source of truth for "is this match
live."

Per project convention, the codebase uses the API's long status names everywhere and never
the short codes. This task adopts that convention and removes the remaining short-code usages
left over from tasks 9/11 (in test fixtures and assertions).

## Acceptance Criteria

- (Event-driven) When one or more matches are in progress, the `/worldcup2026 schedule`
  command shall display a "Live Now" section above the upcoming fixtures showing each
  live match with team names, current score, and match status (e.g., "Second Half (67')").
- (State-driven) While no matches are in progress, the command shall display only the
  upcoming fixtures section (no empty "Live Now" section).
- (Ubiquitous) The schedule command shall continue to show up to 3 upcoming (not-started)
  fixtures below the live section.
- (Unwanted) If live data is stale (cache marked stale), the live section shall indicate
  this with "(data may be outdated)" rather than showing potentially wrong scores.
- (Unwanted) If a live match is at halftime (status `Halftime`, elapsed `null`), the command
  shall display the status with no empty `()` and shall still include the match in "Live Now".
- (Ubiquitous) The system shall determine liveness via a single `isMatchLive()` helper shared
  by the poller and the schedule command (no duplicate status list).
- (Ubiquitous) The codebase (production and tests) shall use the API's long status names only;
  no short status codes (`1H`, `HT`, `2H`, `FT`, etc.) shall remain.

## Implementation Plan

### Step 1: Make `isMatchLive()` canonical on long status names

The football API reports each fixture's state as both `status.short` (e.g. `1H`) and
`status.long` (e.g. `First Half`). The poller currently matches short codes, but the live
cache stores only `status.long` (`mapLiveFixture`), so a cache consumer like the schedule
command cannot reuse the short-code check. We standardize liveness on the long name so the
poller and the schedule command share one helper.

A match is considered live (in play) for these long status values:

- `First Half`
- `Kick Off`
- `Halftime`
- `Second Half`
- `2nd Half Started`
- `Extra Time`
- `Break Time`
- `Penalty In Progress`
- `Match Suspended`
- `Match Interrupted`

All other statuses are not live: `Time To Be Defined`, `Not Started`, `Match Finished`,
`Match Postponed`, `Match Cancelled`, `Match Abandoned`, `Technical Loss`, `WalkOver`,
`In Progress`.

Note: `Match Suspended` and `Match Interrupted` are treated as live (the match is in progress
but temporarily halted, e.g. weather), so the schedule keeps showing them until the match
resumes or is officially finished/abandoned.

Update `src/utils/dates.js` so `isMatchLive()` matches the long status names that the live
cache stores:

```javascript
function isMatchLive(status) {
  const liveStatuses = [
    'First Half',
    'Kick Off',
    'Halftime',
    'Second Half',
    '2nd Half Started',
    'Extra Time',
    'Break Time',
    'Penalty In Progress',
    'Match Suspended',
    'Match Interrupted',
  ];
  return liveStatuses.includes(status);
}
```

Update the one production caller, `src/live-data/poller.js:61`, to pass the long name:

```javascript
isMatchLive(f.fixture.status.long)
```

After this change, `status.short` is read nowhere in production code (`poller.js:61` is the
only `.short` reader in `src/`). The raw API fixture still carries both fields; we simply stop
reading the short one.

### Step 2: Add `getLiveFixtures()` to data module

Add a function to `src/data/index.js` that returns all fixtures currently in-progress, reusing
the canonical helper rather than defining a second status list. The cache stores `status` as
the long name (`mapLiveFixture`), so the merged fixture's `status` feeds `isMatchLive()`
directly.

```javascript
const { isMatchLive } = require('../utils/dates');

function getLiveFixtures() {
  if (!liveCache) return [];

  return fixtures
    .map((f) => {
      const live = liveCache.getFixture(f.id);
      return live ? { ...f, ...live } : null;
    })
    .filter((merged) => merged && isMatchLive(merged.status));
}
```

Export `getLiveFixtures` from the module.

### Step 3: Update `scheduleBlocks()` in `src/commands/worldcup.js`

Modify `scheduleBlocks()` to:

1. Call `getLiveFixtures()` to get in-progress matches.
2. If any live matches exist, render a `Live Now 🟢` header followed by one section block per
   match:
   - Format: `*Mexico 2 - 1 South Africa* · Second Half (67')`
   - Halftime (elapsed `null`): omit the `()` -> `*Mexico 0 - 0 South Africa* · Halftime`
   - If stale: append `_(data may be outdated)_`
3. Add a divider between the live and upcoming sections (only when both exist).
4. Continue showing up to 3 upcoming fixtures as before. If there are no upcoming fixtures but
   there ARE live matches, still render the live section (do not short-circuit to
   "No upcoming fixtures.").

```javascript
function liveBlocks(liveFixtures) {
  const blocks = [{ type: 'header', text: { type: 'plain_text', text: 'Live Now 🟢' } }];

  for (const fixture of liveFixtures) {
    const home = getTeamName(fixture.teams.homeTeamId);
    const away = getTeamName(fixture.teams.awayTeamId);
    const score = `${fixture.finalScore.home} - ${fixture.finalScore.away}`;
    const elapsed = fixture.elapsed != null ? ` (${fixture.elapsed}')` : '';
    const staleNote = fixture.stale ? ' _(data may be outdated)_' : '';
    blocks.push({
      type: 'section',
      text: {
        type: 'mrkdwn',
        text: `*${home} ${score} ${away}* · ${fixture.status}${elapsed}${staleNote}`,
      },
    });
  }

  return blocks;
}
```

Note: this reuses the team/score/elapsed shape already established by `formatLiveCard()` in
`src/broadcast/format.js` (task-12). We render the score inline here rather than calling
`formatLiveCard()` because the schedule needs compact one-line section blocks, not the
two-block LIVE card; the data fields (`finalScore`, `elapsed`, `status`, `stale`) are identical.
