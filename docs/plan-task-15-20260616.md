# Task 15: Architecture Diagram Update

## Details

Date: 2026-06-16

Dependencies: task-14

Decisions: none

## Description

Update `DIAGRAMS.md` so the architecture diagrams reflect the iteration-2 (live data)
architecture delivered by tasks 9-14. The current Container Diagram (C4 Level 2) still depicts
the iteration-1 system: a "Data Layer" backed by static JSON only, with no live-data subsystem
and no football data provider. Since then the app gained an entire live pipeline (API client,
poller, in-memory cache, differ, mapper), a tiered data layer, thread-aware match context, and
a new external system (the football API).

This is a LOW-criticality, documentation-only task. It writes no application code and changes
no behavior; it brings the diagrams back in sync with the shipped system so they remain a
trustworthy reference. No new architectural decisions are made here - the architecture being
documented was already decided in ADR-2 (caching, polling, source of truth, manual
persistence), so no ADR is created.

## Acceptance Criteria

- (Ubiquitous) The System Context diagram in `DIAGRAMS.md` shall include the football data
  API as an external system that the app polls for live match data.
- (Ubiquitous) The Container diagram in `DIAGRAMS.md` shall depict the live-data subsystem:
  API client, poller, in-memory cache, differ, and mapper, with their interactions.
- (Ubiquitous) The Container diagram shall show the tiered data layer (in-memory live cache ->
  per-fixture detail files -> static JSON schedule) rather than a single "Static JSON" store.
- (Ubiquitous) The Container diagram shall show the broadcaster being driven by live-data
  change events (poller -> diffs -> broadcaster -> Slack), not only interval-based pre-match
  cards.
- (Ubiquitous) The Container diagram shall show the manual sync script as the offline writer
  of static JSON and per-fixture detail files (consistent with ADR-2).
- (Ubiquitous) `DIAGRAMS.md` shall include a Live Score section with a data-flow diagram
  (poll -> map -> cache -> diff -> broadcast/update, plus the tiered read path used by the
  schedule command and thread-grounded AI), annotating the staleness fallback (5 consecutive
  failures -> cache marked stale).
- (Ubiquitous) Every Mermaid diagram in `DIAGRAMS.md` shall render without syntax errors, and
  the file shall pass `npm run lint:md`.
- (State-driven) While documenting components, the diagrams shall reference module names that
  exist in `src/` (no diagram node shall name a module that is not present in the codebase).

## Implementation Plan

Step-by-step plan for implementing this task.

1. Re-read the shipped architecture to ground the diagram in fact:
   - Entry/wiring: `src/index.js` (sets live cache on data layer, starts poller, wires
     `poller.onChanges -> scheduler.handleDiffs`, starts scheduler).
   - Live pipeline: `src/live-data/api.js` (calls `https://v3.football.api-sports.io`
     `/fixtures` with `x-apisports-key`), `poller.js` (passive 15-min / active 15-s lifecycle),
     `cache.js` (in-memory Map, stale flag), `differ.js` (event/score diffs), `mapper.js`
     (API -> cache shape).
   - Data layer: `src/data/index.js` tiered resolution (live cache -> detail files ->
     `fixtures.json`).
   - Consumers: `src/commands/worldcup.js` (schedule incl. Live Now), the chat handler
     (`src/handlers/chat.js` with `src/handlers/match-context.js`, thread-grounded AI), and
     the broadcaster (`src/broadcast/scheduler.js`, `src/broadcast/format.js`).
   - Offline: `scripts/sync-data.js` (writes JSON + detail files).
2. Update the System Context diagram: add the football data API as an external system,
   with an edge `app -- "polls live fixtures [HTTPS]" --> footballapi`. Keep existing Slack,
   MIA, and user nodes.
3. Rewrite the Container diagram to add a `Live Data Subsystem` grouping (API client,
   poller, cache, differ, mapper) and the external football API, and to replace the single
   "Static JSON" data node with the tiered data layer. Add edges:
   - `poller -- "GET /fixtures (live=all, date) [HTTPS]" --> footballapi`
   - `poller --> mapper --> cache`, `poller --> differ`
   - `differ -- "score/event diffs" --> broadcast` (via `onChanges`)
   - `cache --> data` (tier-1), `data --> detailfiles` / `data --> staticjson` (tiers 2/3)
   - `commands --> data`, `chat --> matchcontext --> data`
   - `sync["Sync Script (npm run sync)"] -- "writes" --> staticjson` and `--> detailfiles`
   - keep `broadcast -- "chat.postMessage / chat.update" --> slack` and the guardrail/MIA
     edges.
4. Keep the existing Agentic Flow diagram; verify it is still accurate (guardrail pipeline
   is unchanged by iteration 2) and leave it as-is unless a discrepancy is found.
4a. Add a new Live Score section (separate from the Agentic Flow, which documents the
   guardrail pipeline) with a data-flow diagram showing: football API -> poller (15s active
   poll) -> mapper -> cache; poller -> differ -> `onChanges` -> broadcaster (`chat.update` for
   score, `chat.postMessage` for event alerts) -> Slack; and the tiered read path (cache ->
   detail files -> static JSON) consumed by the schedule command's Live Now section and the
   thread-grounded AI match context, with the staleness path annotated (5 consecutive failures
   -> cache marked stale).
5. Update the diagram legend/notes if new node categories are introduced (e.g. Component vs
   Container vs External System), keeping the existing legend style.
6. Validate: confirm all Mermaid blocks parse, cross-check every node name against `src/`
   module names, and run `npm run lint:md` until clean. Do not run `npm test` expectations on
   behavior (no code changes), but `npm run lint` must pass.
