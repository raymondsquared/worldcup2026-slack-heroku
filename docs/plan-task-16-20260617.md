# task-16: Incomplete Event Filtering

## Details

Date: 2026-06-17

Dependencies: none

Decisions: none

## Description

The football API sometimes sends duplicate events - first with playerName as
null/empty/"Unknown", then seconds later with the real player name. The differ
treats these as separate events (different composite key), causing double-posts
to Slack. Add a guard in `postEventAlert` that filters incomplete events at
broadcast time. Var events bypass the filter (VAR uses event.detail, not player).

## Acceptance Criteria

- (Event-driven) When an event has playerName that is null, undefined, empty,
  whitespace-only, or "unknown" (case-insensitive), the system shall log the
  event and skip broadcasting to Slack
- (Event-driven) When a Var event is received, the system shall bypass the
  filter entirely and broadcast it (Var events never have playerName)
- (Event-driven) When an event has a valid playerName, the system shall
  broadcast it normally (no regression)
- (Ubiquitous) The filter shall run before any Slack API calls in
  postEventAlert (fail fast per TR-13)

## Implementation Plan

1. Add guard in `src/broadcast/scheduler.js` `postEventAlert()` immediately
   after the staleness check return (line 97) and before the
   `matchMessages.get` lookup (line 99) - this ensures no Slack API calls are
   made for filtered events (satisfies TR-13)
2. Guard logic:
   - If `event.type === 'Var'`, skip the filter entirely
   - Otherwise check `event.playerName`: if falsy, or after `.trim()` is empty,
     or `.trim().toLowerCase() === 'unknown'`, log and return early
3. Log format: `Filtered incomplete event for fixture ${matchId}: ${event.type}
   at ${event.minute}' (no valid player)`
4. Defensive fix: add fallback on line 148 fallback text:
   `text: \`${event.type}: ${event.playerName || 'Unknown'} ${event.minute}'\``
5. Add test cases in `src/test/broadcast-live.test.js`

## Assumptions

- The API sends null/undefined for both `playerExternalId` and `playerName` on
  incomplete events. This means the differ produces different composite keys for
  incomplete vs complete events, so the complete event is always emitted as new
  and passes the filter. If the API ever sends a valid `playerExternalId` with
  null `playerName`, the differ would not re-emit the corrected version (accepted
  limitation - not observed in production data).

## Test Strategy

New describe block "incomplete event filtering" in
`src/test/broadcast-live.test.js` with cases:

1. Goal with playerName null -> not posted, logged
2. Goal with playerName '' -> not posted
3. Goal with playerName 'Unknown' -> not posted (mixed case)
4. Goal with playerName 'UNKNOWN' -> not posted (upper case)
5. Card with playerName undefined -> not posted
6. subst with playerName '  ' (whitespace only) -> not posted
7. Var event -> posted (bypasses filter entirely, no playerName check)
8. Goal with valid playerName -> posted (baseline)
9. Missed Penalty (type Goal, detail 'Missed Penalty') with null -> not posted
