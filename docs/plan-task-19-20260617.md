# Task 19: Match-End Detection and Final Card

## Details

Date: 2026-06-17

Dependencies: task-18

Decisions: none

## Description

When a live match finishes, the broadcast card must transition from the live
state (green indicator, ticking elapsed minute) to a final state (red indicator,
"Match Finished", final score, complete scorer list). The system detects
match-end by tracking which fixture IDs were live in the previous poll - when a
fixture disappears from the live response, it has finished. The app fetches the
fixture's final state and updates the card. A passive-mode fallback also catches
recently-finished matches that may have been missed (e.g., after a dyno restart).

## Acceptance Criteria

- (Event-driven) When a fixture ID that was present in the previous live poll
  response is absent from the current live poll response, the system shall fetch
  the fixture's final state via the football API and update the in-memory cache
  with the finished fixture data.
- (Event-driven) When a match-end is detected, the system shall update the
  broadcast card to show a red indicator, "Match Finished" status, final score,
  and complete scorer list within one poll cycle (~15s).
- (Event-driven) When a match-end is detected, the system shall emit a diff with
  a `matchEnded: true` flag so downstream consumers (scheduler) can distinguish
  match-end from normal score/elapsed updates.
- (State-driven) While in passive mode checking for live matches by date, the
  system shall also detect recently-finished fixtures that have a tracked
  broadcast card and update those cards to the final state.
- (Unwanted) If the fixture fetch fails on match-end detection, the system shall
  log the error, mark the cached fixture as stale, and retry on the next passive
  check cycle rather than crashing or leaving the card in a stale live state.
- (Ubiquitous) The system shall not re-emit match-end for a fixture that has
  already been processed as finished (idempotent).

## Implementation Plan

1. Add `previousLiveIds` Set and `finishedIds` Set to `poller.js` to track
   fixture IDs that were live in the last active poll and those already processed
   as finished (idempotent). Keeping both in the poller avoids coupling differ to
   poller-specific lifecycle state.

2. In `differ.js`, add a pure helper `buildMatchEndDiff(fixtureId, data)` that
   constructs a diff object with `matchEnded: true`, final score, and events.
   No internal state - just builds the object. Must set `scoreChanged: true`
   unconditionally to guarantee `handleDiffs` triggers the card update (the
   helper has no previous state to compare, and the status transition requires
   a card refresh regardless of whether the numeric score changed).

3. In `pollLiveFixtures()` (poller.js), BEFORE the `wcFixtures.length === 0`
   early-return to passive mode, detect disappeared fixtures:
   - Build current live IDs set from wcFixtures.
   - Compare `previousLiveIds` with current live IDs.
   - For each ID in `previousLiveIds` that is NOT in current AND not in
     `finishedIds`:
     a. Call `fetchFixtureById(externalId)` to get final state.
     b. Map the response via `mapLiveFixture`.
     c. Update cache with the finished fixture data (status, events, score).
     d. Build match-end diff via `buildMatchEndDiff`.
     e. Add to `finishedIds`.
   - If fetch fails: log error, do NOT add to `finishedIds` (retries on next
     passive check).
   - Update `previousLiveIds` to current set.
   - THEN check `wcFixtures.length === 0` for passive switch.

   Order of operations is critical: cache must be updated with final state
   (status: "Match Finished", all events) BEFORE emitting the diff, because
   `updateCard` in the scheduler reads fixture data from the cache via
   `getFixtureById`.

4. In `checkForLiveMatches()` (poller.js, passive mode), add fallback logic:
   - For any World Cup fixture in today's date response with a finished status
     that is NOT in `finishedIds`: fetch final state, update cache, build
     match-end diff, emit via changesCallback, add to `finishedIds`.
   - The scheduler's `updateCard` already no-ops when no card exists for a
     fixture (returns early if `!ts`), so the poller does not need access to
     `matchMessages` - it emits for all finished fixtures and the scheduler
     safely ignores those without tracked cards.

5. In `handleDiffs()` (scheduler.js), no special-casing needed for `matchEnded`
   diffs. The existing flow already works:
   - `scoreChanged` or `elapsedChanged` triggers `updateCard`.
   - `updateCard` calls `getFixtureById` which returns cache data with
     status "Match Finished" and complete events.
   - `formatLiveCard` already renders the red indicator and full scorer list for
     "Match Finished" status (implemented in task-18).
   - The `matchEnded: true` flag in the diff is informational (for future use by
     task-24 post-match summary).

6. Reset `previousLiveIds` and `finishedIds` when poller stops (in `stop()`).

Known limitations:

- After dyno restart, `previousLiveIds`, `finishedIds`, and `matchMessages` are
  all empty. The passive fallback only works for matches that finish in the
  current process lifecycle. Message persistence is out of scope.
- Detection relies on fixture disappearance from the live endpoint, not on a
  specific status string. This correctly handles all terminal statuses ("Match
  Finished", "Match Finished After Extra Time", etc.).

Files to modify:

- `src/live-data/poller.js` - previousLiveIds/finishedIds tracking, match-end
  detection in active and passive modes
- `src/live-data/differ.js` - pure `buildMatchEndDiff` helper function
- `src/test/live-data-poller.test.js` - match-end detection tests
- `src/test/live-data-differ.test.js` - buildMatchEndDiff tests
- `src/test/broadcast-live.test.js` - handleDiffs match-end integration tests

## Test Strategy

Unit tests (Jest):

- differ.js:
  - `buildMatchEndDiff` returns diff with `matchEnded: true`, final score, events
  - `buildMatchEndDiff` returns correct structure with empty events array

- poller.js:
  - When a fixture disappears from live response, `fetchFixtureById` is called
  - Match-end diff is emitted via changesCallback with `matchEnded: true`
  - When ALL fixtures disappear (empty live response), match-end is detected
    before switching to passive mode
  - When fetch fails, error is logged and fixture is not added to finishedIds
    (retries later)
  - Fixture already in finishedIds is not re-processed (idempotent)
  - Passive check detects finished fixture and emits match-end diff
  - `stop()` resets previousLiveIds and finishedIds

- scheduler.js (handleDiffs):
  - Diff with `matchEnded: true` and `scoreChanged: true` triggers `updateCard`
  - Card update uses formatLiveCard which renders "Match Finished" with red dot
    (via existing cache -> getFixtureById flow)

Run `npm test` and `npm run lint` to verify.
