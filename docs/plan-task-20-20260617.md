# Plan: Task-20 - Per-Thread Match Personas

Date: 2026-06-17

## Requirements

- FR-14: When posting a broadcast card, assign a random persona (sporty, funny,
  or serious) and store it in Slack message metadata alongside matchId
- FR-15: When the bot is mentioned in a broadcast thread, read the persona from
  the thread parent's metadata and use the corresponding system prompt tone
- TR-12: Persona system prompts defined as templates in a dedicated module, with
  the persona key stored as a string in message metadata
  (event_payload: { matchId, persona })

## Acceptance Criteria (EARS format)

- WHEN a broadcast card is posted (tick or initial live card),
  THEN the metadata event_payload SHALL contain both matchId and a randomly
  selected persona key (one of: "sporty", "funny", "serious")

- WHEN a user mentions the bot in a broadcast thread,
  THEN the bot SHALL read the persona from the parent message metadata and
  use the corresponding system prompt tone for its AI response

- WHEN the parent message has no persona in metadata (legacy cards),
  THEN the bot SHALL fall back to the default neutral system prompt

- WHEN the bot is mentioned outside a broadcast thread (DM or other channels),
  THEN no persona SHALL be applied (default neutral prompt)

## Current Architecture

1. `src/broadcast/scheduler.js` posts cards with metadata:
   `{ event_type: 'worldcup_match', event_payload: { matchId } }`
   Two places: `tick()` (pre-match card) and `postEventAlert()` (initial live card)

2. `src/handlers/chat.js` reads metadata via `detectMatchFromThread()`:
   extracts `matchId` from `parentMessage.metadata.event_payload.matchId`
   Passes match context to `ask()` in the mention handler.

3. `src/mia/index.js` - `ask(userText, context)` builds messages with a
   hardcoded system prompt. Context is appended if provided.

## Implementation Plan

### Step 1: Create persona module (`src/mia/personas.js`)

New module exporting:

- `PERSONAS` object with keys: sporty, funny, serious
- Each value: a system prompt string tailored to that persona's tone
- `getRandomPersona()` - returns a random persona key
- `getSystemPrompt(personaKey)` - returns the full system prompt for a key,
  falling back to a neutral default if key is null/undefined/invalid

Persona tones:

- sporty: Energetic sports commentator style, uses action verbs, excitement
- funny: Light-hearted, witty, uses wordplay and humor about football
- serious: Analytical pundit, focuses on tactics, statistics, measured tone
- (default/null): Current neutral prompt (backward compatible)

### Step 2: Modify scheduler to include persona in metadata

In `src/broadcast/scheduler.js`:

- Import `getRandomPersona` from `../mia/personas`
- Add `matchPersonas` Map (matchId -> persona key) at module level
- In `tick()`: select persona via `getRandomPersona()`, store in matchPersonas,
  add `persona` to event_payload
- In `postEventAlert()` (initial live card path): same - select, store, include
- In `updateCard()`: re-include metadata with matchId + persona from
  matchPersonas map on every `chat.update()` call. This prevents Slack from
  stripping metadata on card updates (Slack clears metadata unless re-supplied).
- In `reset()`: clear matchPersonas

Updated metadata shape:

```javascript
metadata: {
  event_type: 'worldcup_match',
  event_payload: { matchId: fixture.id, persona: 'sporty' },
}
```

Why matchPersonas Map: `chat.update()` needs the persona but only receives
matchId + score + elapsed. Rather than threading persona through all callers
(poller -> handleDiffs -> updateCard), store it at card-post time and look it
up on update. Simple, local, no cross-module coupling.

### Step 3: Modify chat handler to read persona from metadata

In `src/handlers/chat.js`:

- `detectMatchFromThread()` currently returns just matchId (number|null)
- Change return to `{ matchId, persona }` object (or null if not in thread)
- Read `parentMessage.metadata.event_payload.persona` alongside matchId
- Pass persona to the `ask()` call

### Step 4: Modify MIA ask to accept and use persona

In `src/mia/index.js`:

- Import `getSystemPrompt` from `./personas`
- Change signature: `ask(userText, { context, persona } = {})`
- Replace hardcoded system prompt with `getSystemPrompt(persona)`
- Context injection unchanged (appended to system prompt)
- Update all callers: `ask(text)` -> `ask(text)` (unchanged, defaults apply),
  `ask(text, context)` -> `ask(text, { context })`,
  `ask(text, context, persona)` -> `ask(text, { context, persona })`

## Design Decisions

- Return shape change for detectMatchFromThread: Returns object instead of
  bare number. This is a contained change (only called in one place in the
  mention handler). Null return unchanged for the "not in thread" case.
  The function is exported for tests but the public contract is internal.

- matchPersonas Map for metadata preservation: Slack's chat.update strips
  metadata unless you re-supply it. Storing persona in a local Map (keyed by
  matchId) lets updateCard re-include metadata without threading persona through
  the poller/differ/handleDiffs call chain. Cleared on reset().

- Options object for ask(): Using `ask(text, { context, persona })` instead
  of positional params avoids the awkward "pass empty string for context" pattern
  and is future-proof for additional options (e.g., web search fallback in
  task-22).

- Persona selection is random per card post: No persistence needed beyond
  Slack metadata + in-memory Map. Thread persona is fixed once the card is
  posted.

- Backward compatible: Legacy cards without persona in metadata will get
  null, which maps to the existing neutral prompt. No migration needed.

- No persona on DM responses: Only broadcast threads get persona treatment.
  DMs keep the neutral helpful assistant tone.

## Known Limitations

- Persona is immutable once the card is posted (no "change persona" command yet,
  that's not in scope for this task)
- Math.random() for persona selection means distribution isn't perfectly uniform
  across small sample sizes, but acceptable for this use case
