# Plan: Task-22 - Post-LLM Fallback Integration

Date: 2026-06-17

## Requirements

- FR-11: When MIA responds with "I don't know" or equivalent (low confidence
  indicators), the app shall call the web search API with the sanitized user
  query, inject sanitized results as context, and call MIA again to produce a
  grounded answer
- TR-11: The "I don't know" detection shall use pattern matching on MIA's
  response text (configurable patterns: "I don't know", "I don't have
  information", "I cannot answer", "no data available")
- TR-17: All existing guardrails (PII masking, prompt defense, toxicity
  detection, audit trail) shall remain active for web-search-augmented
  responses

## Acceptance Criteria (EARS format)

- WHEN MIA's initial response matches a low-confidence pattern,
  THEN the system SHALL call webSearch() with the sanitized user query
  and retry MIA with the search context injected

- WHEN the web search fallback produces a new response,
  THEN all existing guardrails (PII demasking, toxicity filter, audit log)
  SHALL apply to the retry response

- WHEN the web search returns null (rate limited, empty, no key),
  THEN the system SHALL return MIA's original response unchanged

- WHEN MIA's initial response does NOT match a low-confidence pattern,
  THEN no web search SHALL be triggered (no extra latency on normal answers)

- WHEN the fallback retry response also matches a low-confidence pattern,
  THEN the system SHALL return that response as-is (no infinite loop)

## Current Architecture

- `src/mia/index.js` - ask() pipeline: PII mask -> sanitize -> toxic check ->
  ground -> build messages -> chat -> toxic filter -> demask -> log
- `src/web-search/index.js` - webSearch(query) returns formatted context string
  or null. Already sanitized (task-21). Uses `===` delimiters.
- `src/mia/guardrails.js` - sanitizeInput, maskPii, demaskPii, filterToxic
- `src/mia/ground.js` - ground() retrieves local data based on keyword matching

## Implementation Plan

### Step 1: Canonical phrase via system prompt + detector

Instead of pattern-matching many variations of "I don't know", we instruct MIA
via the system prompt to respond with an exact canonical phrase when it lacks
information. This is more reliable and eliminates false-positive risk.

In `src/mia/personas.js`:

```javascript
const LOW_CONFIDENCE_PHRASE = "I don't have enough information to answer this.";

const BASE_INSTRUCTION =
  'Answer questions using only the provided context.' +
  ` If you do not have enough information to answer, respond with exactly: "${LOW_CONFIDENCE_PHRASE}"`;
```

In `src/mia/low-confidence.js`:

```javascript
const { LOW_CONFIDENCE_PHRASE } = require('./personas');

function isLowConfidence(text) {
  if (!text || typeof text !== 'string') return false;
  return text.trim() === LOW_CONFIDENCE_PHRASE;
}
```

Exports: `isLowConfidence(text)` - returns true only if the trimmed response
is exactly the canonical phrase. No regex, no length heuristic, no
false-positive risk.

### Step 2: Modify ask() pipeline in `src/mia/index.js`

Insert fallback logic between step 6 (call MIA) and step 7 (filter toxic output):

```javascript
// 6. Call MIA
const rawOutput = await chat(messages);

// 6b. Fallback: if low confidence, try web search + retry
let finalOutput = rawOutput;
if (isLowConfidence(rawOutput)) {
  try {
    const searchContext = await webSearch(sanitized);
    if (searchContext) {
      // Rebuild messages with web search context appended
      const augmentedMessages = [
        {
          role: 'system',
          content: systemPrompt + (grounded ? `\n\nContext:\n---\n${grounded}\n---` : '')
            + `\n\n${searchContext}`,
        },
        { role: 'user', content: sanitized },
      ];
      finalOutput = await chat(augmentedMessages);
    }
  } catch (err) {
    // Search or retry failed - use original response gracefully
    console.error('Web search fallback failed:', err.message);
  }
}

// 7. Filter toxic output (applies to whichever response we use)
const filtered = filterToxic(finalOutput);
```

Key points:
- Web search context is appended AFTER local grounding context (both can coexist)
- `searchContext` already has `===` delimiters from task-21 formatAsContext
- Do NOT re-sanitize searchContext (it is already clean from task-21)
- try/catch ensures search API outages never crash the ask() pipeline
- Only one retry (no loop) - if retry is still low confidence, return it as-is
- If webSearch returns null (rate limited, empty), use original response

## Design Decisions

- Single retry, no loop: If the retry also returns low confidence, we accept it.
  Prevents infinite loops and unbounded API calls. The user gets the best answer
  available.

- Web search context appended after local context: Both grounding sources
  coexist in the system message. Local data (fixtures, groups, players) provides
  structured facts; web search provides real-time context. MIA can synthesize
  both.

- No re-sanitization of search context: Task-21's webSearch() already sanitizes
  all results (strips injection markers, validates URLs, truncates). Running
  sanitizeInput again would strip the `===` delimiters.

- Canonical phrase over pattern matching: Instead of regex-matching many "I
  don't know" variations (error-prone, false positives), we instruct MIA via
  the system prompt to use one exact phrase. Detection is a simple string
  comparison - no regex, no length heuristic, no false-positive risk.

- Fallback is transparent to the caller: ask() returns a string regardless of
  whether fallback was triggered. The caller (chat handler) doesn't need to know.
