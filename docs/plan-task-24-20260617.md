# Plan: Task-24 - Post-Match AI Summary

Date: 2026-06-17
Requirements: FR-21, FR-22, TR-17
Depends on: task-23 (Source Citation in AI Responses)

## Goal

After a match ends and the final card is posted, generate an AI-written recap
using MIA grounded on full match event data (goals, cards, subs, statistics)
and post it as a reply in the match thread. The recap uses the thread's persona
tone if one is set.

## Current State

- `differ.buildMatchEndDiff()` returns diffs with `matchEnded: true`
- `scheduler.handleDiffs()` processes diffs, calls `updateCard()` for
  score/elapsed changes, but does NOT handle match-end specially
- `matchMessages` Map stores fixtureId -> messageTs (thread parent)
- `matchPersonas` Map stores fixtureId -> persona key
- `live-data/cache` stores fixture data including events and statistics
- `mia/index.js` has `ask(userText, opts)` with `{ context, persona }` options
- Guardrails (PII, toxic, audit) are applied via the existing ask() pipeline

## Design

### Approach: Recap module in broadcast/ triggered by matchEnded diff

Create `src/broadcast/recap.js` containing:
1. `buildRecapContext(fixtureId)` - formats match events/stats as grounding text
2. `generateRecap(client, fixtureId)` - calls MIA and posts to thread

The scheduler's `handleDiffs` function gets a check: after updating the card,
if `diff.matchEnded === true`, call `generateRecap()`.

### Data Flow

```
poller detects match end
  -> differ.buildMatchEndDiff (matchEnded: true)
  -> scheduler.handleDiffs
    -> updateCard (final card with red dot + "Match Finished")
    -> recap.generateRecap(client, fixtureId)
      -> cache.getFixture(fixtureId) -> events, stats, score
      -> buildRecapContext(liveData, fixture)
      -> ask("Write a match recap", { context, persona })
      -> client.chat.postMessage to thread
```

### Recap Context Format

```
Match Summary:
===
Teams: USA 2-1 Mexico
Status: Match Finished

Goals:
- Pulisic 23' (USA)
- McKennie 67' (USA)
- Lozano 45' (Mexico)
- Dest 78' (USA, Own Goal)
- Pulisic 90+2' (USA, Penalty)

Cards:
- Dest Yellow 55' (USA)
- Alvarez Red 78' (Mexico)

Substitutions:
- Reyna -> Weah 60' (USA)
- Jimenez -> Raul 72' (Mexico)

Statistics:
- USA: Possession 55%, Shots on Target 6, Corners 5
- Mexico: Possession 45%, Shots on Target 4, Corners 3
===
```

Own Goals get "(Own Goal)" suffix. Penalty goals get "(Penalty)" suffix.
Missed penalties are included under Goals with "(Missed Penalty)" suffix.
The `event.detail` field provides this information.

### MIA Prompt for Recap

The recap call uses a dedicated user prompt instead of a real user question:

```
Write a brief, engaging match recap in 2-3 paragraphs. Cover the key moments,
turning points, and final result. Do not use XML format for this response -
respond with plain text only.
```

Important: The recap prompt overrides the XML response format since we want
plain text output (no XML wrapping). The system prompt still has
BASE_INSTRUCTION with XML rules, so the user message must explicitly say
"Do not use XML format" to override it.

Alternative: Use a separate RECAP_INSTRUCTION system prompt that skips
BASE_INSTRUCTION entirely. This is cleaner - no conflict with XML rules.

Decision: Create a `RECAP_PROMPT` system prompt in personas.js that includes
persona tone but NOT BASE_INSTRUCTION. This avoids XML formatting in recaps.

### RECAP_PROMPT Design

```javascript
const RECAP_PERSONAS = {
  sporty: 'You are an energetic sports commentator writing a match recap.' +
    ' Use action verbs and convey excitement about the beautiful game.',
  funny: 'You are a witty football writer with a great sense of humor.' +
    ' Use wordplay and light-hearted observations in your recap.',
  serious: 'You are an analytical football pundit writing a match recap.' +
    ' Focus on tactics, key decisions, and measured analysis.',
};

const DEFAULT_RECAP_PROMPT =
  'You are a football writer creating a concise match recap.';

function getRecapPrompt(personaKey) {
  return RECAP_PERSONAS[personaKey] || DEFAULT_RECAP_PROMPT;
}
```

### TR-17 Compliance: Guardrails Must Remain Active

The plan originally called `chat()` directly, bypassing guardrails. This
violates TR-17. Instead, add a `recap` option to `ask()` in `mia/index.js`
that:
- Uses a custom system prompt (RECAP_PROMPT, no XML/BASE_INSTRUCTION)
- Skips XML parsing (parseResponse)
- KEEPS: filterToxic on output, logInteraction for audit trail
- SKIPS: PII masking (recap input is machine-generated, not user text),
  sanitizeInput (grounding context is trusted internal data)

### ask() Enhancement

Add `opts.recap = true` flag:

```javascript
// In ask(), early branch for recap mode:
if (options.recap) {
  const recapMessages = [
    { role: 'system', content: options.systemOverride + '\n\nContext:\n---\n' + context + '\n---' },
    { role: 'user', content: userText },
  ];
  const rawOutput = await chat(recapMessages, { temperature: 0 });
  const filtered = filterToxic(rawOutput);
  logInteraction({ input: '[recap]', output: filtered.text });
  // If toxic, return null - caller skips posting
  if (!filtered.safe) return null;
  return filtered.text;
}
```

The recap caller checks for null:
```javascript
const recap = await ask(RECAP_USER_PROMPT, { context, recap: true, systemOverride });
if (!recap) return; // Toxic output or empty - skip posting
```

### Recap Module (src/broadcast/recap.js)

```javascript
const { ask } = require('../mia');
const { getRecapPrompt } = require('../mia/personas');
const cache = require('../live-data/cache');
const { getFixtureById, getTeamName } = require('../data');

const RECAP_USER_PROMPT =
  'Write a brief, engaging match recap in 2-3 paragraphs.' +
  ' Cover the key moments, turning points, and final result.';

async function generateRecap(client, fixtureId, threadTs, persona) {
  if (!threadTs) return; // No thread to post to

  const liveData = cache.getFixture(fixtureId);
  if (!liveData || liveData.stale) return; // No data or stale data - skip

  const fixture = getFixtureById(fixtureId);
  if (!fixture) return;

  const context = buildRecapContext(liveData, fixture);
  const systemOverride = getRecapPrompt(persona);

  const recap = await ask(RECAP_USER_PROMPT, {
    context,
    recap: true,
    systemOverride,
  });

  if (!recap) return; // Toxic output or empty - skip posting

  await client.chat.postMessage({
    channel: process.env.BROADCAST_CHANNEL_ID,
    thread_ts: threadTs,
    text: recap,
  });
}
```

### Scheduler Integration

In `handleDiffs`, after the card update for match-end diffs:

```javascript
// After card update, read threadTs AFTER updateCard (it may delete stale ts)
if (diff.matchEnded) {
  const threadTs = matchMessages.get(diff.fixtureId);
  const persona = matchPersonas.get(diff.fixtureId) || null;
  generateRecap(client, diff.fixtureId, threadTs, persona).catch((err) => {
    console.error(`Recap failed for fixture ${diff.fixtureId}:`, err.message);
  });
}
```

Important: Read `threadTs` AFTER `updateCard()` - if updateCard encounters
`message_not_found`, it deletes the ts from matchMessages. Reading after
ensures we do not post to a non-existent thread.

Fire-and-forget pattern (same as other async operations in handleDiffs) -
recap failure does not block event alerts or card updates.

## Risks

- Low: MIA timeout on recap generation (client timeout is 15s). Fire-and-forget
  pattern means it does not block card updates. Error is logged. No retry -
  recaps are best-effort.
- Low: Recap may be bland without statistics (some matches have no stats in
  early API responses). Context builder outputs "Statistics: not available."
- Mitigated: Stale cache data - generateRecap checks `liveData.stale` and
  skips if true. No recap is better than a wrong recap.
- Mitigated: TR-17 guardrails - recap goes through ask() with filterToxic +
  logInteraction. PII masking skipped (input is machine-generated, not user).

## Architecture Decision

Final signature: `generateRecap(client, fixtureId, threadTs, persona)`

Avoids circular imports (scheduler -> recap, not recap -> scheduler).
Cleaner and more testable - all dependencies passed as arguments.
