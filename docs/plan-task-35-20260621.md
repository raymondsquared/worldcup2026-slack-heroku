## Part 1: Live card shows "No goals yet" even after goals scored

### Details

Severity: HIGH (user-visible, persists entire match duration)

Affects: Live match card scorers line in broadcast channel

### Bug Description

The live match card in Slack displayed "No goals yet" on the scorers context
line for the entire match - from kickoff through full time - even when goals
had been scored and goal alert events were posting correctly in the thread.

### Root Cause

`updateCard` in `src/broadcast/scheduler.js` called `getFixtureById(matchId)`
which returns the static fixture from `fixtures.json` - it has no `events`
property. `formatLiveCard` then called `formatScorers(fixture.events, fixture)`
where `fixture.events` was `undefined`, always hitting the
`if (!events || events.length === 0) return 'No goals yet'` early return.

The live events (goals, cards, subs) are stored in the live-data cache
(`cache.getFixture(matchId).events`), not in the static fixture data. The card
was reading from the wrong data source. The same issue affected `postEventAlert`
when posting an initial live card for a match that hadn't received a pre-match
card.

### Fix

No new imports needed. `getFixtureById` in `src/data/index.js` already merges
live-data cache (Tier 1: `{ ...base, ...live }`) onto the static fixture, so
the returned object includes `status` and `events` from the poller cache. The
fix was simply confirming that `updateCard` and `postEventAlert` pass the
result of `getFixtureById` (which already has events) directly to
`formatLiveCard` - no manual cache merge required.

### Files Changed

- `src/broadcast/scheduler.js`
  - Added a clarifying comment that `getFixtureById` returns the merged fixture
  - No new dependencies needed (Tier 1 data resolution handles it)

### Why it wasn't caught earlier

The `formatScorers` tests test the formatter in isolation with explicit `events`
arrays. The `broadcast.test.js` tests mock `formatLiveCard` at the module level
and don't assert the scorers content inside the blocks. The bug was invisible
during development because the score line (home/away numbers) updated correctly

- only the scorers context line was wrong.

## Part 2: Poller delayed 10-14 minutes after kickoff

### Details

Severity: MEDIUM (data arrives late, not missing)

Affects: First live card update timing for all matches

### Bug Description

The live card sometimes didn't update until 10-14 minutes after kickoff. Goal
alerts and score changes were delayed by the same amount. The poller was still
in passive mode (15-minute interval) when the match started, and didn't switch
to active polling (15s) until the next passive tick discovered live matches.

### Root Cause

The passive poller (`checkForLiveMatches`) runs every `PASSIVE_INTERVAL`
(default 15 min). It only switches to active mode when it detects matches that
are already live from the API response. If the last passive tick fires at
T-5min (5 min before kickoff), the next one fires at T+10min - meaning the
poller is blind for the first 10 minutes of the match.

### Fix

Schedule a precise kickoff timer via `scheduleKickoffActivation()`: on each
passive tick, if no matches are live, schedule a one-shot `setTimeout` for the
exact next kickoff time. When it fires, switch to active. Zero wasted API calls,
max 15s latency after kickoff.

Called from `checkForLiveMatches` when no live matches are found, and from
`switchToPassiveMode` when returning from active mode (so multi-match days
schedule the next kickoff timer immediately, not 15 min later). Timer is
cleared on `stop()` and on any transition to active mode.

### Expected timeline

| Time     | Mode    | Interval  | Action                                            |
| -------- | ------- | --------- | ------------------------------------------------- |
| T-60 min | Passive | 15 min    | Normal passive tick, no matches imminent          |
| T-30 min | Passive | 15 min    | Normal passive tick                               |
| T-15 min | Passive | 15 min    | Pre-match card posted; kickoff timer scheduled    |
| T-5 min  | Passive | (waiting) | Timer pending, zero API calls                     |
| T+0:00   | Active  | 15 sec    | Timer fires, switches to active, first poll       |
| T+0:15   | Active  | 15 sec    | Live data arrives, card updates with score/events |
| T+5 min  | Active  | 15 sec    | Goal events detected, alerts posted in thread     |

### Files Changed

- `src/live-data/poller.js`
  - Added `kickoffTimer` state variable
  - Added `scheduleKickoffActivation()` function
  - `checkForLiveMatches`: calls `scheduleKickoffActivation()` when passive
  - `switchToPassiveMode`: calls `scheduleKickoffActivation()` on return
  - `switchToActiveMode`: clears kickoff timer
  - `stop`: clears kickoff timer

## Part 3: Scheduled daily restart to prevent duplicate posts

### Details

Severity: LOW (cosmetic - duplicate card/recap, not data loss)

Affects: Pre-match cards, match-end recaps

### Problem

Heroku automatically restarts dynos every ~24 hours ("daily cycling") to
maintain system health and clear memory leaks. This restart time is not
configurable - it drifts based on when the dyno last booted, meaning it can
land at any hour. For a real-time app that depends on in-memory dedup state,
an unpredictable restart mid-match or during a pre-match window causes
duplicate messages in Slack.

By scheduling our own restart at a known-safe hour, we reset the 24h clock
so Heroku's automatic cycle also lands in the safe window. This gives us
deterministic restart behavior with zero Heroku add-on cost.

On restart, in-memory state is lost:

- `posted` (Set) - tracks which pre-match cards have been posted
- `finishedIds` (Set) - tracks which match-end recaps have fired

The differ (goal/card/sub alerts) is already protected - it seeds all existing
events on first poll after restart and only emits new ones.

### Fix

Schedule a predictable daily restart at a safe hour via in-process timer.
At the configured hour, `process.exit(0)` triggers a clean exit and Heroku
auto-restarts the dyno with fresh state.

Env var: `DAILY_RESTART_HOUR_IN_UTC` (0-23, default: 11)

### Safe window analysis

All kickoff times are 00:00-04:00 or 16:00-23:30 UTC:

| Time (UTC)  | Activity                                     |
| ----------- | -------------------------------------------- |
| 04:00       | Last possible match ends (~02:00 kick + 2h)  |
| 05:00-11:00 | No matches, no pre-match windows, no recaps  |
| 11:00       | Restart here (1h before highlights digest)   |
| 12:00       | Highlights digest fires                      |
| 15:45       | First pre-match window opens (16:00 kickoff) |

Restart at 11:00 UTC: 7 hours after last match could end (no recap risk),
4h45m before first pre-match window (no duplicate card risk), 1h before
highlights digest (fresh state, missed-run recovery handles it).

### Files Changed

- `src/utils/restart-scheduler.js` (new; later renamed to
  `src/lib/restart-scheduler.js` in Part 9)
  - Reads `DAILY_RESTART_HOUR_IN_UTC` env var (default 11)
  - Calculates delay to next occurrence, sets `setTimeout` with `process.exit(0)`
  - Timer uses `.unref()` so it doesn't keep the process alive
  - Exports `scheduleRestart()` and `cancel()`
- `src/index.js` - require and call `scheduleRestart()` after services start
- `.env.example` - add `DAILY_RESTART_HOUR_IN_UTC=11`
- `README.md` - add env var to the list

## Part 4: Duplicate event alerts caused by API minute drift

### Details

Severity: MEDIUM (user-visible; repeated alerts clutter the match thread)

Affects: Threaded event alerts (subs, cards) in the broadcast channel

### Bug Description

During the Ecuador vs Curacao match the same event was posted to the thread
more than once, each copy showing a slightly different minute:

| Event                            | First post | Re-post | Minute moved |
| -------------------------------- | ---------- | ------- | ------------ |
| Sub: R. van Eijma -> D. Fonville | 75'        | 76'     | +1           |
| Sub: N. Angulo -> P. Estupinan   | 71'        | 70'     | -1           |
| Sub: G. Kastaneer -> J. Locadia  | 84'        | 83'     | -1           |

The player names and the event itself were identical; only the minute changed
between polls.

### Root Cause

The differ's `eventKey` function used
`${event.type}-${event.minute}-${event.playerExternalId || event.playerName}`
as the composite key. The football API revises an event's `time.elapsed` between
polls (clock corrections, VAR adjustments, provider re-syncs). When the minute
changes, the key changes, so `diffAll` treats the already-broadcast event as
brand-new and emits it again.

### Fix

Key each event on its stable identity instead of its minute:

| Event type        | Old key (drift-prone)      | New key (stable)                |
| ----------------- | -------------------------- | ------------------------------- |
| `subst`           | `subst-{minute}-{player}`  | `subst-{playerOff}-{playerOn}`  |
| `Card`            | `Card-{minute}-{player}`   | `Card-{detail}-{player}`        |
| `Goal`            | `Goal-{minute}-{player}`   | `Goal-{minute}-{player}` (kept) |
| `Var` / `default` | `{type}-{minute}-{player}` | unchanged (kept)                |

A substitution is uniquely the `(player off, player on)` pair; a card is
`detail + player` (a second yellow carries its own `detail`, so it stays
distinct); a goal keeps the minute, because a genuine brace is two real goals
by the same player and the minute is what separates them.

Residual risk (accepted, documented in code): a goal whose minute drifts can
still re-post, traded off to preserve brace detection.

### Files Changed

- `src/live-data/differ.js`
  - `eventKey` rewritten to a per-type stable key (subs keyed on the off/on
    pair, cards on detail+player, goals/VAR keep the minute)
- `src/test/live-data-differ.test.js`
  - New `minute drift (duplicate suppression)` describe block: drifted sub and
    card (yellow and red) do not re-emit, distinct subs stay independent, a
    brace still emits both goals, a second yellow is distinct from the first, a
    sub with no assist dedupes by player-off, and a drifted goal does re-emit
    (documents the accepted trade-off)

### Why it wasn't caught earlier

The existing differ test `handles API reordering` only covered goals inserted
at an earlier minute (a new key that _should_ emit). It did not cover the same
event reappearing with a changed minute. Subs and cards - the highest-frequency
events near the end of a match - were the most exposed because clocks are most
volatile in the closing minutes.

## Part 5: Event player names not resolved to the canonical squad name

### Details

Severity: LOW-MEDIUM (user-visible; the same player is shown two different ways
in one match thread, which reads as a data glitch)

Affects: Threaded event alerts and the live card / recap scorer lines, plus the
stored per-fixture detail files

### Bug Description

The football API labels the same player inconsistently between events. In the
Brazil vs Haiti fixture, Haiti's number 2 (externalId 20850) appeared under two
names within the same match:

| Minute | Event        | API playerName | Squad name |
| ------ | ------------ | -------------- | ---------- |
| 4'     | Yellow Card  | Carlens Arcus  | C. Arcus   |
| 46'    | Substitution | C. Arcus       | C. Arcus   |

The drift runs both ways - sometimes a full name where the squad uses an
abbreviation, sometimes the reverse, sometimes an accented vs ASCII spelling.

### Root Cause

`mapEvent` in `src/live-data/mapper.js` stored the raw API name verbatim
(`playerName: event.player.name`) and never consulted the squad files, which
already hold one canonical name per player keyed by `externalId`. Because both
the offline `scripts/sync-data.js` and the live poller build events through
`mapEvent`, the inconsistency reached both the static detail files and live
updates.

### Fix

Seed an `externalId -> name` map at module load from `getAllPlayers()` (a new
accessor in `src/data/index.js` that flattens every squad file) and resolve both
`playerName` and `assistPlayerName` through it; fall back to the raw API name
when an id is absent from every squad. The resolver is exported as
`resolvePlayerName` so the backfill reuses it.

### Backfilling already-synced files

The runtime fix corrects live matches going forward, but the per-fixture detail
files synced before this change still carried raw API names.
`scripts/sync-player-names.js` re-resolves `playerName` / `assistPlayerName` in
place using the same resolver; it is idempotent, supports `--dry-run`, and
leaves any unsynced id on its API name. 35 of 36 files, 280 name fields
corrected at consolidation.

### Files Changed

- `src/live-data/mapper.js`
  - Seed `externalPlayerIdToName` from `getAllPlayers()` at module load
  - Add `resolvePlayerName(externalId, apiName)` with API-name fallback
  - `mapEvent` resolves both `playerName` and `assistPlayerName`
- `src/data/index.js`
  - Add and export `getAllPlayers()` (flattens every squad file; tolerates an
    unreadable or malformed file rather than failing startup)
- `src/test/live-data-mapper.test.js`
  - Three new tests: resolves `playerName` by externalId, resolves the assist
    name, and falls back to the API name when the id is not in any squad
- `scripts/sync-player-names.js` (new)
  - Idempotent backfill of the static fixture detail files; `--dry-run` flag
- `src/data/fixtures/*.json`
  - Names-only backfill across the synced detail files

### Why it wasn't caught earlier

The existing mapper test asserted the event shape with a player whose API name
already matched the squad, so the pass-through looked correct. The bug only
shows when the API sends a name that differs from the squad for the same
`externalId` - most visibly when the same player appears in two events one poll
apart, which single-event unit fixtures never exercised.

## Part 6: Live card stays on the pre-match layout after kickoff

### Details

Severity: MEDIUM (user-visible; the most-watched surface shows the wrong card -
the pre-match teaser - for the opening passage of every match)

Affects: The broadcast match card in the channel (the pre-match card posted by
`tick`), at kickoff

### Bug Description

This is a delayed card TRANSITION, not a frozen card. When a match goes live the
card stays on its pre-match "Fixture Starting Soon!" layout (home vs away plus
kickoff time, with NO score and NO minute) and only switches to the live score
card when the first score or elapsed change arrives after kickoff. Once it does
become the live card it updates normally - every minute tick is an elapsed
change, so the clock climbs 2', 3', 4' on its own. The defect is purely that the
switch to the live card is late.

How long the wrong card lingers depends on the live feed: if the match clock is
advancing, the first minute tick flips the card within about a minute; but if
`elapsed` is null or not advancing at kickoff (status `Kick Off` / `Halftime` /
`Break Time`) and the score is still 0-0, nothing changes until the first goal -
which is itself a score change, so the card appears to update only "when the
first event comes through", and a 0-0 can sit on the pre-match card for a long
time.

Expected: once a match is live, the card transitions to the live score card
(live status, elapsed minute, 0-0, "No goals yet") within one active poll cycle
(~15s) of the first poll that observes the fixture live, without waiting for an
elapsed tick or an event.

### Root Cause

The pre-match-to-live transition is coupled to a score/elapsed change, but the
first poll that observes a fixture live deliberately emits no change:

1. `tick` posts the "Fixture Starting Soon!" card and stores its `ts` in
   `matchMessages`.
2. At kickoff the poller's one-shot timer (Part 2) switches to active mode and
   runs `pollLiveFixtures` every 15s, which calls `differ.diffAll(cacheEntries)`.
3. On the first observation of a fixture the differ's restart guard seeds
   `previousKeys` / `previousScores` / `previousElapsed` and `continue`s
   WITHOUT pushing a diff. So the very poll that means "this match is now live"
   emits nothing.
4. `handleDiffs` rewrites the pre-match card into the live card via `updateCard`
   ONLY when `scoreChanged || elapsedChanged`.

The restart guard cannot simply be removed: it exists so a worker restart
mid-match does not re-broadcast every historical event.

### Fix

On first observation, keep seeding dedup state and keep `newEvents` empty (so no
historical event is re-broadcast), but emit a one-time initial diff marked
`initial: true` with both change flags false. The scheduler calls `updateCard`
when `diff.initial` is set, in addition to the existing score/elapsed triggers.
`updateCard` already no-ops when no `matchMessages` timestamp is tracked, so a
mid-match worker restart (lost in-memory `ts`) does not post a spurious card.

The as-built guard line is
`if (scoreChanged || elapsedChanged || diff.initial || diff.matchEnded)` (Part 7
adds the fourth term). `diffAll` is only called from `pollLiveFixtures` (active
mode), so the initial diff fires only for fixtures observed during active
polling - exactly once per fixture per process lifetime.

### Files Changed

- `src/live-data/differ.js`
  - Restart guard rewritten to seed dedup state and emit an `initial: true` diff
    with `newEvents: []` and both change flags false
- `src/broadcast/scheduler.js`
  - `handleDiffs` calls `updateCard` when `diff.initial` in addition to the
    existing `scoreChanged || elapsedChanged` triggers
- `src/test/live-data-differ.test.js`
  - The three first-observation assertions now assert the initial-diff shape,
    plus an "initial diff is emitted exactly once" case
- `src/test/broadcast-live.test.js`
  - New `handleDiffs` case: an `initial: true` diff calls `client.chat.update`
    once and posts nothing to the thread

### Why it wasn't caught earlier

The differ tests assert behavior from the second `diffAll` call onward (they
seed with a first call and ignore its empty return), and the broadcast-live
tests drive `handleDiffs` with hand-built diffs that already carry
`scoreChanged: true`. No test exercised the gap between "first poll observes
the fixture live" and "first score/elapsed change".

## Part 7: Match-end card did not flip to the finished render when the feed status lagged

### Details

Severity: MEDIUM (user-visible; the final card can keep a green "live" header and
a stale minute after the match is over, out of step with the recap that posts at
the same moment)

Affects: The broadcast match card at full time (the `updateCard` render driven by
the match-end diff)

### Bug Description

When a match ends the broadcaster posts a recap and is supposed to leave the card
showing the finished state - red circle, "Match Finished", final score, no
climbing clock. But the card's finished/live decision was derived from the cached
fixture `status` string, which can still read "Second Half" at the instant the
fixture drops out of the live feed: the API stops returning the fixture before it
ever reports "Match Finished", so the cached status is frozen on its last live
value.

### Root Cause

The match-end signal and the card render used two different sources of truth:

- The differ emits a match-end diff carrying `matchEnded: true` (from
  `buildMatchEndDiff`), and `handleDiffs` already used that flag to fire the
  recap.
- But `formatLiveCard` decided "finished vs live" purely from the cached status
  string: `const isFinished = (fixture.status || '').startsWith('Match
Finished')`. When the feed never delivered a terminal status, the card
  rendered as live.

### Fix

Thread the differ's `matchEnded` flag through to the renderer as the
authoritative finished signal, falling back to the cached status string when the
flag is not set (the normal in-play refresh path):

- `handleDiffs` includes `diff.matchEnded` in the card-update guard and forwards
  it to `updateCard`, which passes it to `formatLiveCard`
- `formatLiveCard` gains a `matchEnded = false` parameter; `isFinished` becomes
  `matchEnded || (fixture.status || '').startsWith('Match Finished')`
- The default `matchEnded = false` keeps every existing caller on the unchanged
  behavior; only the match-end diff passes `true`

### Files Changed

- `src/broadcast/scheduler.js`
  - `handleDiffs` card-update guard gains `|| diff.matchEnded` and forwards the
    flag to `updateCard`
  - `updateCard` gains a `matchEnded = false` parameter, passed to
    `formatLiveCard`
- `src/broadcast/format.js`
  - `formatLiveCard` gains a `matchEnded = false` parameter; `isFinished` is the
    flag OR the cached-status check
- `src/test/broadcast-format.test.js`
  - Regression: a `matchEnded: true` render forces the finished card even when
    `fixture.status` still reads "Second Half"
- `src/test/broadcast-live.test.js`
  - `handleDiffs` match-end cases: a `matchEnded` diff triggers `updateCard`
    once, and does not re-post events already broadcast during the live phase

### Why it wasn't caught earlier

The earlier `formatLiveCard` tests set `fixture.status` to "Match Finished"
explicitly to exercise the finished render, so they never covered the case where
the match is over but the cached status still reads a live value.

## Part 8: sync-data unit test verified nothing (script was not requirable)

### Details

Severity: HIGH (a test-quality defect - the most logic-dense data script had
zero real coverage and any regression would pass CI)

Affects: `src/test/sync-data.test.js` and the testability of
`scripts/sync-data.js`

### Bug Description

`src/test/sync-data.test.js` was tautological. It set up `jest.mock` for the
api and mapper modules but never `require`d `scripts/sync-data.js`, so the mocks
were dead and no line of the real script ever ran. Every test reimplemented the
script's logic inline and asserted against its own copy. The suite was green and
proved nothing.

### Root Cause

`scripts/sync-data.js` could not be required without side effects:

1. A module-load env guard called `process.exit(1)` when `FOOTBALL_API_KEY` was
   unset - requiring the file under Jest would tear down the worker.
2. The CLI dispatch ran on import with no `require.main === module` guard.
3. There was no `module.exports`.

### Fix

Split the script into a run-only CLI wrapper and an importable library, so
side-effect-freeness is structural rather than guarded:

- `src/lib/sync-data.js` (the library) holds every helper and subcommand plus
  the `COMMANDS` registry, and only exports - requiring it runs and writes
  nothing.
- `scripts/sync-data.js` (the CLI wrapper, ~37 lines) validates the env key
  BEFORE requiring the library, parses argv, and dispatches via the imported
  `COMMANDS`.

The test suite now requires the real library (with api/mapper mocked) and
asserts on the genuine exports: `normalizeName`, `getDateRange`,
`buildTeamNameMap`, `resolveTeamId`, `generateTeamId`, plus two mocked-api
subcommand cases.

### Files Changed

- `src/lib/sync-data.js` (moved from `scripts/lib/sync-data.js`)
  - All helpers, subcommands, `COMMANDS` registry; exports only
  - Relative requires and `DATA_DIR` updated for the path
- `scripts/sync-data.js` (now a thin CLI wrapper)
  - Validates `FOOTBALL_API_KEY`, parses argv, dispatches
- `src/test/sync-data.test.js`
  - Repointed to require the real lib; real assertions on exported helpers

### Why it wasn't caught earlier

A passing suite reads as coverage. The gap was only visible by reading the test
against the script and noticing the missing `require`.

## Part 10: `DAILY_RESTART_HOUR_IN_UTC=0` was silently rewritten to 11

### Details

Severity: MEDIUM (a valid config value was ignored - midnight restarts never
happened; no crash, so it would pass unnoticed until someone checked the logs)

Affects: `src/lib/restart-scheduler.js` (the Part 3 module)

### Bug Description

The restart hour was parsed as `Number(process.env.DAILY_RESTART_HOUR_IN_UTC) || 11`.
Hour `0` (midnight) is a valid, intended value, but `0` is falsy, so the `||`
treated it as "missing" and substituted the default `11`. An operator who set
`DAILY_RESTART_HOUR_IN_UTC=0` to restart at midnight UTC would silently get
11:00 UTC instead.

### Root Cause

`Number(x) || default` is the wrong idiom for any numeric setting whose valid
range includes `0`. It conflates "value is `0`" with "value is absent". It also
silently accepts garbage: `Number('abc')` is `NaN` (falsy, falls back), but
`25` or `11.5` are truthy and pass through unchecked.

### Fix

Parse explicitly with `Number.isInteger(n) && n >= 0 && n <= 23` via a
`parseRestartHour(raw)` helper. Falls back to `DEFAULT_RESTART_HOUR` (11) only
when the value is genuinely missing, non-numeric, fractional, or out of range.

### Files Changed

- `src/lib/restart-scheduler.js` - replaced the `Number(env) || 11` expression
  with `DEFAULT_RESTART_HOUR` + `parseRestartHour()` (integer-and-range check)
- `src/test/restart-scheduler.test.js` - new `DAILY_RESTART_HOUR_IN_UTC parsing`
  describe block: hour `0` schedules for midnight, and out-of-range (`24`),
  non-numeric (`noon`), and fractional (`11.5`) values each fall back to 11

## Part 11: No top-level process handlers - silent worker crashes

### Details

Severity: MEDIUM (an unhandled rejection or uncaught exception could kill the
worker dyno with no log line, making a production crash hard to diagnose)

Affects: `src/index.js` (the worker entry point)

### Bug Description

`src/index.js` had no `process.on('unhandledRejection')` or
`process.on('uncaughtException')` handler. The app is a long-lived worker dyno
full of fire-and-forget async work. A single rejected promise or uncaught throw
would either print Node's terse default warning or terminate the process without
context.

### Fix

Register two last-resort handlers in `src/index.js`, before `start()` is
invoked:

- `unhandledRejection`: logs the stack and keeps running - a single dropped
  promise (e.g. a fire-and-forget Slack post) should not take the whole app down.
- `uncaughtException`: logs the stack and exits non-zero - the process is in an
  undefined state, so let Heroku recycle the dyno (the same recovery the Part 3
  restart scheduler relies on).

The two handlers are deliberately asymmetric.

### Files Changed

- `src/index.js` - added module-level `unhandledRejection` (log, keep running)
  and `uncaughtException` (log, `process.exit(1)`) handlers ahead of the existing
  `start().catch(...)`

Note: `src/index.js` is the run-only entry point; nothing requires it, so there
is no unit test for these handlers. Behavior verified by inspection.
