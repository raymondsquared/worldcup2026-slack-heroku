## Part 1: Event alerts do not show which team the event belongs to

### Details

Severity: MEDIUM (user-visible, every card/sub/missed-penalty alert)

Affects: Threaded event alerts in the broadcast channel

### Bug Description

Threaded event alerts named the player and minute but never the team. A yellow
card read "Yellow Card . J. Alcivar . 38'" with no indication of which side
the player was on. For a neutral viewer following a match they were not
watching (e.g. Ecuador vs Curacao), there was no way to tell who was being
cautioned, subbed, or who missed a penalty.

Goal alerts were less affected because they already print the score line with
both team names, but they still did not flag the scorer's own team.

### Root Cause

The alert formatters in `src/broadcast/format.js` rendered only `playerName`
and `minute`:

```js
text: `${emoji} *${label}* . ${player} . ${minute}`,   // formatCardAlert
```

The mapped `event.teamId` (now correctly resolved after task-35's mapper fix)
was available on every event but never used by the card, sub, missed-penalty,
or goal formatters.

### Fix

Add a small `teamFlagPrefix(event)` helper that returns the team flag followed
by a space when `event.teamId` resolves, and an empty string otherwise (so an
event without a mapped team does not render a stray white-flag fallback):

```js
function teamFlagPrefix(event) {
  return event.teamId ? `${getFlag(event.teamId)} ` : '';
}
```

Applied to the four player-attributed alerts:

| Alert         | Before                            | After                                  |
| ------------- | --------------------------------- | -------------------------------------- |
| Goal          | Goal! Scorer . 23'                | Goal! {flag} Scorer . 23'              |
| Card          | Yellow Card . J. Alcivar . 38'    | Yellow Card . {flag} J. Alcivar . 38'  |
| Missed Penalty| Missed Penalty . Taker . 55'      | Missed Penalty . {flag} Taker . 55'    |
| Sub           | Sub . On <-> Off . 60'            | Sub . {flag} On <-> Off . 60'          |

`formatVarAlert` is intentionally left unchanged: VAR decisions are not
attributed to a single team in the source data.

### Files Changed

- `src/broadcast/format.js`
  - Added `teamFlagPrefix(event)` helper (guards on `event.teamId`)
  - Prefixed goal, card, missed-penalty, and sub alerts with the team flag
- `src/test/broadcast-format.test.js`
  - Added `teamId` to the event fixtures and asserted the flag appears
  - Added a regression test: omits the flag (no white-flag leak) when
    `event.teamId` is null

### Why it depends on task-35

The flag is only meaningful because task-35 fixed the mapper to seed the
team-id map at module load. Before that fix every `event.teamId` was `null`,
so this helper would have produced no flag for any event.

## Part 2: Internal team IDs did not match official FIFA codes

### Details

Severity: LOW (data-model consistency; not user-visible)

Affects: `teams.json`, `countries.json`, `fixtures.json`, per-team player
squads, per-fixture detail files

### Problem

Internal team `id` values were the app's primary key (used as
`homeTeamId` / `awayTeamId`, player squad filenames `players/{id}.json`, and
fixture detail filenames `{id}-{home}-{away}.json`). For 44 of 48 teams the
`id` already equalled the official FIFA code stored in `countries.json`
`fifaCode`. Four did not:

| Internal id | fifaCode | Country                 |
| ----------- | -------- | ----------------------- |
| BOS         | BIH      | Bosnia & Herzegovina    |
| CAP         | CPV      | Cape Verde Islands      |
| CON         | COD      | Congo DR                |
| CUR         | CUW      | Curacao                 |

This inconsistency meant the primary key and the canonical FIFA code diverged
for a subset of teams, which is a latent source of confusion and matching bugs.

### Fix

Rename the four internal ids so that `id === fifaCode` for all 48 teams:
`BOS -> BIH`, `CAP -> CPV`, `CON -> COD`, `CUR -> CUW`.

This is a primary-key rename, so it touches data content and filenames:

- Content: every quoted `"<id>"` and player-id prefix `"<id>-"` in the data and
  sample/test JSON
- Filenames: 4 player squad files (`players/{id}.json`) and 5 fixture detail
  files (`{id}-{home}-{away}.json`) renamed via `git mv` so history is preserved

Flag rendering is unchanged: `getFlag` keys off `flagISO` (e.g. `CW` -> Curacao
flag), not the team id, so the 🇨🇼 / 🇧🇦 / 🇨🇻 / 🇨🇩 flags render exactly as before.

### Verification of substring safety

The four old codes were grepped with word boundaries across the entire repo
(excluding `node_modules`). They appeared only as quoted JSON tokens and
player-id prefixes - no English-word substrings (no "config", "second",
"boss") and no occurrences in source `.js` logic. The rename was therefore a
safe, scoped token replacement plus file moves.

### Files Changed

- `src/data/teams.json` - 4 `id` values
- `src/data/countries.json` - 4 `id` values (now equal to existing `fifaCode`)
- `src/data/fixtures.json` - `homeTeamId` / `awayTeamId` occurrences
- `src/test/highlights-api/fixtures.sample.json` - sample `awayTeamId`
- `src/test/highlights-adapter.test.js` - assertion `'CUR'` -> `'CUW'`
- Renamed (git mv):
  - `players/BOS.json -> BIH.json`, `CAP.json -> CPV.json`,
    `CON.json -> COD.json`, `CUR.json -> CUW.json`
  - `fixtures/26-SUI-BOS.json -> 26-SUI-BIH.json`,
    `3-CAN-BOS.json -> 3-CAN-BIH.json`,
    `13-ESP-CAP.json -> 13-ESP-CPV.json`,
    `21-POR-CON.json -> 21-POR-COD.json`,
    `9-GER-CUR.json -> 9-GER-CUW.json`

### Why it wasn't a problem before

The flag always rendered from `flagISO`, and most lookups resolved by id
internally and consistently, so the mismatch never surfaced as a visible bug.
Aligning the ids is a preventative consistency change, not a defect fix.
