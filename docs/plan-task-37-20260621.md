## Motivation

Before this task, MIA's retrieval used a shallow keyword router. It had exactly
three regex branches, each wired by hand to one data accessor:

```js
if (/fixture|match|schedule|play|game|next|when/.test(text)) { getUpcomingFixtures(3) }
if (/group|standing|table/.test(text))                       { getGroups() }
if (team && /player|roster|squad|who/.test(text))            { getPlayers(team.id) }
```

Every new kind of question means another `a|b|c` keyword list plus another
accessor call, and the router still misses paraphrases and never narrows to the
entity the user named. Three concrete gaps surfaced while testing real queries:

1. Team-aware fixtures: "when is the next england match" injects the global
   next-3 fixtures (`getUpcomingFixtures(3)`), which on a given day are three
   other teams' games, so England's actual next match is absent from context.
2. Past results and events: "what was the score in the brazil game" or "who
   scored" gets either an empty context or upcoming fixtures, even though 36
   finished fixtures and their full event lists (goals, cards, scorers) already
   live in the data layer via `getFixtureEvents`.
3. Player lookup: a named player without their team ("how many goals has harry
   kane scored", "what number does mbappe wear") returns an empty context,
   because the router only recognizes team entities, never player names.

In every gap the data is local and available; the router just does not retrieve
it. When context is empty or wrong, MIA returns low confidence and the
web-search fallback fires for a question the app could have answered offline.

The goal of this task is to stop hand-coding `a|b|c` routers: expose the local
data accessors as tools and let the model choose which function to call. Adding
a new capability then becomes "declare one tool", not "add a regex branch plus
prompt wiring".

## Prior behavior (what tool calling replaced)

The keyword router returned a single context string assembled from the three
branches above, and `ask()` injected it into the system prompt before the one
and only `chat()` call. Observed retrieval for representative queries:

| Query                                       | router injects          | Problem                          |
| ------------------------------------------- | ----------------------- | -------------------------------- |
| when is the next england match              | global next-3 fixtures  | not filtered to England          |
| what was the score in the brazil game       | upcoming fixtures       | no finished-result retrieval     |
| who scored for england yesterday            | England roster          | roster has no events / scorers   |
| how many points does mexico have            | (empty)                 | "points" is not a keyword        |
| how many goals has harry kane scored        | (empty)                 | no player-name lookup            |
| what number does mbappe wear                | (empty)                 | no player-name lookup            |

The data layer already exposes accessors that answer all of these:
`getAllFixtures`, `getUpcomingFixtures`, `getFixtureById`, `getFixtureEvents`,
`getGroups`, `getPlayers`, and `getAllPlayers` (the flattened squad index added
in task-35 Part 5, which makes player-name resolution clean).

## Client capability seam (extended by this task)

`src/mia/client.js` was already most of the way there. `chat()` POSTs to the
OpenAI-compatible `/v1/chat/completions` endpoint and spreads `opts` into the
body, so `chat(messages, { tools })` could send a `tools` field. The one gap was
the return value: the original `chat()` returned only `message.content`, dropping
`message.tool_calls`, so the tool-call contract could not complete.

This task added `chatWithTools(messages, tools, opts)`, which returns the full
assistant message (`{ role, content, tool_calls }`) so the loop can read
`tool_calls`. `chat()` still returns `content`, so recap and other
string-returning callers are unaffected:

```js
async function chat(messages, opts = {}) {
  const message = await postChatCompletion({ model: INFERENCE_MODEL_ID, messages, ...opts });
  return message.content;
}

async function chatWithTools(messages, tools, opts = {}) {
  return postChatCompletion({ model: INFERENCE_MODEL_ID, messages, tools, ...opts });
}
```

## Design (as built)

`ask()` runs a client-side tool-calling loop as its primary retriever, replacing
the up-front regex retrieval (the keyword router) with model-selected tool calls.
When the loop grounds nothing, a direct single model call (no tool context)
answers the question as a deterministic fallback. (See the Decision below and ADR-6.)

### Decision: model-driven tool calls vs more regex

| Option                              | Decision | Reason                                                                          |
| ----------------------------------- | -------- | ------------------------------------------------------------------------------- |
| Model-driven tool calls (chosen)    | Adopted  | The model picks the function; a new capability is one schema, not a new branch. |
| Keep extending keyword regex        | Rejected | Each question type needs another keyword list and wiring; misses paraphrases.   |
| Dump all data into every prompt     | Rejected | Sending every fixture, event, and squad each call is costly and dilutes focus.  |

### Decision: client-side loop vs Heroku Agents endpoint

| Option                              | Decision | Reason                                                                          |
| ----------------------------------- | -------- | ------------------------------------------------------------------------------- |
| Client-side tool loop (chosen)      | Adopted  | Minimal change to the existing client; guardrails stay in-process; full control.|
| Heroku Agents endpoint (server-side)| Deferred | Heavier setup and less app-side control; revisit if the client loop is limiting.|

### Decision: fallback when the tool loop grounds nothing

| Option                              | Decision | Reason                                                                          |
| ----------------------------------- | -------- | ------------------------------------------------------------------------------- |
| Direct single model call (chosen)   | Adopted  | No-context call still answers; low-confidence + web-search then catches thin.   |
| Keep keyword router as fallback     | Rejected | Keeps the dead regex router and a second divergent retrieval path to maintain.  |

### Tool catalog

Each tool is a thin, read-only wrapper over an existing accessor. No tool
mutates state or touches the filesystem outside the data layer.

This task ships seven tools. Six are thin, read-only wrappers over an existing
accessor; the seventh (`get_player_goals`) is derived logic (see the
Player-goals tool section below):

| Tool name             | Arguments        | Backing data                          | Answers                                  |
| --------------------- | ---------------- | ------------------------------------- | ---------------------------------------- |
| get_upcoming_fixtures | teamId?, limit?  | getAllFixtures / getUpcomingFixtures  | "next England match", "what is on next"  |
| get_team_results      | teamId           | getAllFixtures (finished only)        | "what was the score", "did Brazil win"   |
| get_fixture_events    | fixtureId        | getFixtureEvents                      | "who scored", "any red cards"            |
| get_standings         | group?           | getGroups                             | "group table", "how many points"         |
| get_team_squad        | teamId           | getPlayers                            | "list the Brazil squad"                  |
| get_player            | name             | getAllPlayers (task-35 index)         | "what number", "what position", "is X in"|
| get_player_goals      | name             | getAllPlayers + getFixtureEvents      | "how many goals has Messi scored"        |

Notes:

- The first six tools are thin wrappers: each maps to one existing accessor with
  no new business logic. The three documented gaps (team fixtures, past
  results/events, player lookup) are all covered by this set.
- `get_player_goals` is the one tool that needs new cross-fixture logic rather
  than a thin accessor wrapper (it tallies `Goal` events by `externalId` across
  fixtures), so it carries its own counting-rule decisions and tests within this
  task - see the Player-goals tool section below. Player-stat attributes not in
  the data (height, age) remain out of scope and fall through to web search.
- Resolution helpers: `resolveTeam(name)` maps a team name or FIFA code to a
  `teamId` via `teams.json`; `resolvePlayer(name)` maps a name to a squad record
  via the `getAllPlayers()` index. Both live in the new tools module.

### Player-goals tool (derived logic)

`get_player_goals(name)` is the one tool that is not a thin accessor wrapper: it
resolves the player to an `externalId` (via the `getAllPlayers()` index from
task-35 Part 5), scans each fixture's events (`getFixtureEvents`), and tallies the
`Goal` events attributed to that id. It returns a small structured result -
`{ name, team, teamId, goals, penalties, fixtures }`, where `fixtures` lists the
contributing `{ fixtureId, opponent, minutes }`. It is registered in
`TOOL_SCHEMAS` / `TOOL_HANDLERS` and dispatched through the same `dispatchTool`
whitelist, read-only like the rest.

The tally has its own correctness rules, decided from the actual event data
(local `Goal` events carry a `detail` of `Normal Goal`, `Own Goal`, or
`Penalty`, alongside a separate `Var` event type; the broadcast path can also
emit `detail: 'Missed Penalty'` as a `Goal`):

- Match on the stable `playerExternalId`, never the display name, so two players
  who share a surname are never conflated.
- The tally is an ALLOWLIST on `detail`: only `Normal Goal` and `Penalty` count,
  and `penalties` is reported separately as a subset of `goals`. An allowlist
  (not a denylist of only `Own Goal`) is immune to any other `Goal` detail the
  feed emits, most importantly `Missed Penalty`.
- `Own Goal` is EXCLUDED - standard stats never credit an own goal to the
  scorer (verified: Aymen Hussein has a `Normal Goal` and an `Own Goal` in one
  fixture; the tool reports 1, not 2).
- `Missed Penalty` is EXCLUDED - it is not a goal; it shares the `Goal` type
  only because of how the API reports it. None exist in the finished data, so
  this is guarded by an injected-data test.
- VAR-reversed goals need NO special handling: the events feed is already
  post-reconciliation, so a cancelled goal leaves no surviving `Goal` event for
  that player in the fixture (verified for both VAR-cancelled goals in the
  data). Counting surviving `Goal` events is therefore correct without
  cross-referencing `Var` events.
- Both finished and live (in-progress) fixtures are counted, so a "top scorer"
  query stays current during a match. Status is read from `getFixtureById(id)`
  (the live-cache-merged view), not the static `getAllFixtures()` status, which
  is stale until the schedule re-syncs.

Behaviour contract: an unrecognized name returns `null` (consistent with
`get_player`); a recognized player who has not scored returns a structured zero
(`goals: 0, penalties: 0, fixtures: []`), not an error and not null. Assists are
out of scope (the data carries `assistPlayerExternalId`, but this tool tallies
goals only).

### Tool-calling loop (`retrieveWithTools` in `src/mia/index.js`)

`retrieveWithTools` is the primary retriever. It returns `{ rawOutput,
toolsCalled }` only when at least one tool actually ran and the model then
produced a final answer; in every other case (no tool ran, loop exhausted, over
budget, missing `tool_call` id, or any error) it returns `null` and `ask()`
falls through to the direct call. This is the actual control flow:

```js
const MAX_TOOL_ITERS = 4;
const LOOP_BUDGET_IN_MS = 10000;

async function retrieveWithTools(sanitized, systemPrompt) {
  const deadline = Date.now() + LOOP_BUDGET_IN_MS;
  const messages = [
    { role: 'system', content: `${systemPrompt}\n\n${TOOL_INSTRUCTION}` },
    { role: 'user', content: sanitized },
  ];
  const toolsCalled = [];

  try {
    for (let i = 0; i < MAX_TOOL_ITERS; i++) {
      if (Date.now() >= deadline) break; // wall-clock budget exceeded -> direct call

      const msg = await chatWithTools(messages, TOOL_SCHEMAS, { temperature: 0 });

      if (!msg.tool_calls || msg.tool_calls.length === 0) {
        // Model answered without requesting (further) tools. Only treat it as a
        // grounded tool answer if a tool actually ran this loop; otherwise the
        // caller makes a plain direct call.
        if (toolsCalled.length > 0 && msg.content) {
          return { rawOutput: msg.content, toolsCalled };
        }
        return null;
      }

      const assistantContent = msg.content && msg.content.trim() ? msg.content : '.';
      messages.push({ role: 'assistant', content: assistantContent, tool_calls: msg.tool_calls });
      for (const call of msg.tool_calls) {
        if (!call.id) return null; // contract: every tool_call id must be answered
        const name = call.function && call.function.name;
        const args = safeParseArgs(call.function && call.function.arguments);
        const result = dispatchTool(name, args); // whitelist only; never throws
        if (name) toolsCalled.push(name);
        messages.push({ role: 'tool', tool_call_id: call.id, content: JSON.stringify(result) });
      }
    }
  } catch (err) {
    console.error('MIA tool loop failed:', err.message);
    return null; // any error -> direct model call
  }

  return null; // loop exhausted without a final answer -> direct model call
}
```

The final answer still uses the existing XML format
(`<answer>/<confidenceScore>/<isFootballRelatedScore>`), so `parseResponse`,
the low-confidence check, and the web-search fallback are unchanged. Tool turns
are intermediate; only the last assistant message is parsed. A model that emits
no `tool_calls` at all is NOT treated as a tool answer - `toolsCalled` stays
empty, `retrieveWithTools` returns `null`, and `ask()` answers with a direct call
(`retrievalPath = direct`).

### Where it slots into ask()

The guardrail pipeline is untouched and still wraps everything. `ask()` runs in
this order (recap mode is a separate early branch that bypasses the pipeline and
logs `retrievalPath = recap`):

1. maskPii - keep the map for demasking the response
2. sanitizeInput
3. filterToxic (input) - reject before any model call (`retrievalPath = blocked`)
4. retrieval, exactly one of:
   - (a) caller supplied authoritative `context` (e.g. a thread match context):
     answer from it in a single `chat()` call -> `retrievalPath = context`
   - (b) otherwise PRIMARY: `retrieveWithTools` drives the read-only tool loop.
     If it returns a tool-grounded answer -> `retrievalPath = tool_call`. If it
     returns `null` (no tool ran, exhausted, over budget, or error), `ask()`
     makes a direct single `chat()` call with no tool context ->
     `retrievalPath = direct`
5. parseResponse (XML contract) + low-confidence check
6. escalate: if the answer is low-confidence AND on-topic, run the web-search
   retry -> `retrievalPath = web_search`
7. filterToxic (output) - swap a toxic answer for `TOXIC_REPLY`
8. demaskPii - restore the user's real PII, last, so PII exists only in the return
9. logInteraction (masked input/output) recording the resolved `retrievalPath`
   (and `toolsCalled` on the tool path)

### Telemetry (makes SC-8 measurable)

SC-8 ("answered from local data without a web search") is not observable today:
`logInteraction` in `src/mia/audit.js` records only `input` and `output`, with
no signal for which retrieval path produced the answer. This task adds a
`retrievalPath` field to `logInteraction` with one of:

- `tool_call` - the tool loop produced the final answer (one or more tools ran)
- `context` - the caller supplied authoritative context, answered in one call
- `direct` - the direct-call safety net produced the answer (unsupported model,
  no tool call, loop exhausted, or over budget)
- `web_search` - the low-confidence web-search fallback produced the answer

Optionally include `tools_called: [names]` on the `tool_call` path. This lets us
measure the web-search fallback rate for football questions on real traffic.
Without this field SC-8 can only be asserted from unit tests.

`index.js` has these `logInteraction` callsites: the normal answer path, the
toxic-input early-reject path, and the recap path. `retrievalPath` applies to
the answer-producing paths - set it to `tool_call`, `context`, `direct`, or
`web_search` on the normal path. The toxic-input reject is tagged `blocked` (no
retrieval ran),
and recap is tagged `recap` (it bypasses the grounding pipeline by design); both
keep their existing behavior otherwise. `retrievalPath` is an added optional
field, so the change is backward-compatible with the current
`logInteraction({ input, output, timestamp })` signature.

### Rollout

The tool loop is the unconditional primary retriever, gated only by a one-shot
capability probe (no runtime flag). Operational properties:

- The capability probe (removed) was the deploy-time gate; verify tool support
  manually if `INFERENCE_MODEL_ID` changes.
- The `web_search` retrieval-path rate for football questions is observable from
  the telemetry, so the SC-8 improvement is measurable on real traffic.
- There is no runtime toggle to roll back. If the tool path must be disabled for
  a broken model, the lever is `INFERENCE_MODEL_ID` plus the probe; in normal
  operation the direct-call safety net already absorbs a model that emits no
  `tool_calls`, exhausts the loop, or runs over budget.

## Risks and trade-offs

- Extra round-trips: each tool call adds a request (model -> tool_call ->
  execute -> model). Bounded by `MAX_TOOL_ITERS` (4); more latency and tokens
  than the single-call direct path. MIA's `ask()` is invoked only from the Events API
  handlers (`app_mention` and the DM `app.message` in `src/handlers/chat.js`),
  which reply asynchronously via `say()` - there is no Slack 3-second
  request-response deadline on this path (the `/worldcup` slash command in
  `src/commands/worldcup.js` does not call `ask()`). So the cost is the user
  waiting longer for the bot reply, not a request timeout.
- Wall-clock budget: `client.js` enforces a per-request timeout
  (`TIMEOUT_IN_MS = 15000`), so a degenerate 4-iteration loop could run up to
  ~60s. Add a total loop budget (target ~10s) that falls back to the direct call
  on timeout, so a slow model degrades to the single-call path instead of making
  the user wait a minute. The fallback must trigger on loop timeout, not only on
  loop exhaustion.
- Empty / wrong tool results: a tool can return an empty result (for example
  `get_team_results` for a team with no finished match early in the tournament,
  or `get_player` for an unrecognized name) or the model can pick a tool that
  does not fit the question. In both cases the model receives a valid-but-thin
  tool result, produces a low-confidence XML answer, and the existing
  web-search fallback handles it - the same graceful degradation as a thin
  direct-call answer. This path is covered by the empty-result tests in
  Verification.
- Model support: function calling may be unavailable for the configured model.
  Mitigated by the capability probe (deploy-time gate) and the direct-call
  fallback, which answers when the model emits no `tool_calls` at runtime.
- Runaway / cost: the iteration cap and the read-only whitelist bound the loop;
  there is no recursion into arbitrary tools.
- Determinism: temperature 0 is retained for both tool selection and the final
  answer.

## Security

- Tools are a fixed whitelist of read-only data accessors. `dispatchTool`
  rejects any name not in the catalog and never evaluates arbitrary input.
- Tool arguments are validated against the schema and parsed defensively
  (`safeParseArgs`); a malformed argument yields a structured error, not a throw.
- Tool results are local JSON fixtures with no PII; PII masking and demasking
  still wrap the user text and the final answer, and the toxic filter still runs
  on the output.
- The system prompt continues to forbid revealing system messages, policies, or
  available functions (existing `SCOPE_INSTRUCTION` rule), so tool names are not
  disclosed.

## Out of scope

- Heroku Agents server-side orchestration (separate task if the client loop
  proves limiting).
- New player-stat fields not present in the data (height, age, market value);
  these continue to fall through to web search.
- Streaming responses and parallel tool-call optimization.
- Changes to persona behavior, recap mode, or the web-search fallback contract.
