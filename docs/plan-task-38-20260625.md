# Task 38: Setup Environment and Core MCP Server

## Details

Date: 2026-06-25

Dependencies: none

Decisions: [adr-7, adr-8]

## Description

This task combines environment verification and core MCP server implementation.
It establishes the foundational infrastructure for the Football API MCP server by:

1. Verifying Heroku configuration is ready (FOOTBALL_API_KEY, INFERENCE_URL, INFERENCE_KEY)
2. Adding the MCP SDK dependency to the project
3. Creating the directory structure for MCP server code
4. Implementing the base MCP server with STDIO transport using `@modelcontextprotocol/sdk`
5. Building the tool registration framework
6. Creating shared error sanitization utility to prevent API key leaks

This task enables both MCP tool implementation (task-39) and chart feature development (task-40) to proceed in parallel.

## Acceptance Criteria

### Environment Verification

- (Ubiquitous) The system shall verify `FOOTBALL_API_KEY` exists in Heroku config vars
- (Ubiquitous) The system shall verify `INFERENCE_URL` exists in Heroku config vars
- (Ubiquitous) The system shall verify `INFERENCE_KEY` exists in Heroku config vars
- (Ubiquitous) The `package.json` shall include `@modelcontextprotocol/sdk` as a dependency
- (Ubiquitous) The `src/mcp/` directory shall exist for MCP server code

### Core MCP Server

- (Ubiquitous) The system shall implement MCP server using StdioServerTransport from `@modelcontextprotocol/sdk`
- (Ubiquitous) The server shall provide a tool registration framework for registering MCP tools
- (Ubiquitous) The server shall implement `sanitizeError(error)` utility function that replaces API key values with `[REDACTED]`
- (Event-driven) When a tool call fails with status 429, the system shall handle rate limit errors gracefully
- (Ubiquitous) The `src/mcp/football-server.js` file shall exist with server initialization code
- (Ubiquitous) The server shall run on Node.js 22+ (existing engine requirement)

### Quality

- (Ubiquitous) The code shall pass `npm run lint`
- (Ubiquitous) The code shall follow existing ESLint and Prettier configuration

## Implementation Plan

### Step 1: Verify Heroku Configuration (15 minutes)

1. Run `heroku config:get FOOTBALL_API_KEY -a <app-name>` to verify API key exists
2. Run `heroku config:get INFERENCE_URL -a <app-name>` to verify inference URL exists
3. Run `heroku config:get INFERENCE_KEY -a <app-name>` to verify inference key exists
4. If any are missing, document requirements and stop (this is a blocker)

### Step 2: Add MCP SDK Dependency (10 minutes)

1. Run `npm install @modelcontextprotocol/sdk --save`
2. Verify installation: check `package.json` includes the dependency
3. Run `npm test` to ensure no regressions

### Step 3: Create Directory Structure (5 minutes)

1. Create `src/mcp/` directory: `mkdir -p src/mcp`
2. Verify directory exists

### Step 4: Implement Core MCP Server (2-3 hours)

1. Create `src/mcp/football-server.js` with:
   - Use CommonJS require syntax (project uses CommonJS, not ES modules):

     ```javascript
     'use strict';
     const { Server } = require('@modelcontextprotocol/sdk/server/index.js');
     const { StdioServerTransport } = require('@modelcontextprotocol/sdk/server/stdio.js');
     ```

   - Create server instance with name "football-api-mcp" and version "1.0.0"

2. Implement `sanitizeError(error)` utility function:

   ```javascript
   function sanitizeError(error) {
     const apiKey = process.env.FOOTBALL_API_KEY;
     if (!apiKey || !error) return error;
     
     const message = error?.message || String(error);
     const sanitized = message.replace(new RegExp(apiKey, 'g'), '[REDACTED]');
     
     if (error instanceof Error) {
       const sanitizedError = new Error(sanitized);
       sanitizedError.stack = error.stack?.replace(new RegExp(apiKey, 'g'), '[REDACTED]');
       return sanitizedError;
     }
     
     return sanitized;
   }
   ```

3. Implement tool registration framework:
   - Use `server.setRequestHandler('tools/list', async () => {...})` to list registered tools
   - Use `server.setRequestHandler('tools/call', async (request) => {...})` to handle tool calls
   - Create array to store tool definitions
   - Create map to store tool handler functions

4. Add rate limit (429) error handling:
   - Catch 429 responses from Football API
   - Return sanitized error message indicating rate limit hit
   - Use sanitizeError() to ensure no API key leaks

5. Implement server startup:

   ```javascript
   const transport = new StdioServerTransport();
   await server.connect(transport);
   console.error('Football API MCP server running on stdio');
   ```

6. Update Procfile to add MCP server process:
   - Open `Procfile` in project root
   - Add line: `mcp-football: node src/mcp/football-server.js`
   - Save file
   - Note: This enables Heroku to discover and register the MCP server (FR-2, FR-3)

### Step 5: Lint and Format (10 minutes)

1. Run `npm run lint:js` to check for linting errors
2. Run `npm run format` to format code
3. Fix any linting issues

### Step 6: Manual Verification (20 minutes)

1. Test that the MCP server can be started: `node src/mcp/football-server.js`
2. Verify it outputs "Football API MCP server running on stdio" to stderr
3. Test stdin/stdout communication (send valid MCP protocol JSON)
4. Verify no crashes, no API key leaks in error messages
5. Stop the server (Ctrl+C)
6. Note: Full Heroku registration verification happens in task-43 (Heroku Deployment)

- **Module import:** Start the server with `node src/mcp/football-server.js` and verify no import errors
- **Error sanitization:** Trigger an error that would expose API key, verify it's replaced with `[REDACTED]`
- **STDIO communication:** Send a `tools/list` request via stdin, verify response on stdout
- **Rate limit handling:** Simulate 429 response, verify graceful error message

- Run `npm test` to ensure no regressions in existing test suite
- Run `npm run lint` to verify code style compliance
