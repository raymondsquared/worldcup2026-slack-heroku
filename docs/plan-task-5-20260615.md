# task-5: Heroku MIA Client + Guardrails

## Details

Date: 2026-06-15

Dependencies: task-2

Decisions: none

## Description

Build the Heroku Managed Inference Agents (MIA) client and guardrail modules.
The client calls MIA's OpenAI-compatible `/v1/chat/completions` endpoint using
native `fetch`. Guardrails sit around the client: input sanitization (prompt
defense), PII masking with demasking, toxicity filtering on inputs and
outputs, secure data grounding, and an audit logger. Kept intentionally
simple for demo purposes.

Addresses TR-3, TR-4, TR-5, TR-6, TR-7, TR-8, and C-6 (including deployment
runbook updates previously scoped as task-9).

## Acceptance Criteria

- The system shall export a `chat(messages)` function from `src/mia/client.js`
  that calls the MIA `/v1/chat/completions` endpoint
- The system shall use native `fetch` with no external HTTP or OpenAI SDK
  dependencies
- The system shall read `INFERENCE_URL`, `INFERENCE_MODEL_ID`, and
  `INFERENCE_KEY` from environment variables (not hardcoded)
- The system shall send `Authorization: Bearer ${INFERENCE_KEY}` in all
  requests to MIA
- The system shall add `INFERENCE_URL`, `INFERENCE_MODEL_ID`, and
  `INFERENCE_KEY` to `REQUIRED_ENV_VARS` in `src/index.js`
- When user input contains PII (emails, phones), the system shall mask them
  with indexed tokens (e.g., `[EMAIL_1]`) and retain a mapping for demasking
- When MIA responds with tokens (e.g., `[EMAIL_1]`), the system shall demask
  them back to original values before returning to the user
- When user input contains prompt injection patterns, the system shall
  sanitize the input before inclusion in prompts
- When user input or MIA output contains toxic content, the system shall
  reject it with a safe fallback message (input rejects before calling MIA)
- The system shall ground the LLM by retrieving only relevant data slices
  based on user intent (secure data retrieval)
- The system shall log all AI interactions with timestamps and masked content
- When `npm test` is run, the system shall pass with tests covering client,
  guardrails, and audit
- The deployment runbook (`infrastructure/README.md`) shall include
  `heroku config:set` commands for all MIA and broadcast env vars
- When `npm run lint` is run, the system shall pass with no errors

## Implementation Plan

1. Create `src/mia/client.js` - MIA client:
   - `chat(messages, opts)` calls `POST ${INFERENCE_URL}/v1/chat/completions`
   - Sends `Authorization: Bearer ${INFERENCE_KEY}` and
     `Content-Type: application/json` headers
   - Request body: `{ model: INFERENCE_MODEL_ID, messages, ...opts }`
   - Response parsing: `response.json().choices[0].message.content`
   - Guards for empty/malformed responses (empty choices array)
   - Uses `AbortSignal.timeout(15000)` to prevent hanging requests
   - Throws on non-2xx responses with status and error body
   - Returns assistant message content string

2. Create `src/mia/guardrails.js` - input/output guardrails:
   - `maskPii(text)` - regex-based detection and masking of emails and phones.
     Returns `{ masked: string, map: { '[EMAIL_1]': 'original', ... } }`
   - `demaskPii(text, map)` - restores original PII values from the map
   - `sanitizeInput(text)` - strips "ignore previous instructions", system
     overrides (`system:`), and delimiter escapes (`---`, backtick fences)
   - `filterToxic(text)` - checks for harmful content (`harm yourself`,
     `self-harm`), returns `{ safe: bool, text: string }`. Applied on both
     input (reject before MIA) and output (block before user)

3. Create `src/mia/ground.js` - secure data retrieval:
   - `ground(userText)` - keyword-based intent detection
   - Matches -> `getUpcomingMatches(3)`
   - Groups -> `getGroups()`
   - Players -> `getPlayers(teamId)` (team detected by name/id)
   - Returns empty string if no intent matched (LLM says "I don't know")

4. Create `src/mia/audit.js` - audit logger:
   - `logInteraction({ input, output, timestamp })`
   - Logs to stdout as structured JSON (Heroku captures via logplex)

5. Create `src/mia/index.js` - orchestrator that wires it all together:
   - `ask(userText, context)` - the single entry point for AI calls
   - Pipeline: mask -> sanitize -> filterToxic (input) -> ground -> MIA ->
     filterToxic (output) -> demask -> log -> return

6. Create `src/test/mia.test.js` - unit tests:
   - Mock `fetch` for client tests
   - Test PII masking patterns (email, phone)
   - Test PII demasking restores originals
   - Test sanitization strips injection attempts
   - Test toxic filtering blocks on input and output
   - Test ground returns relevant data slices
   - Test ground returns empty for unmatched queries
   - Test audit logger produces structured JSON
   - Test full pipeline: mask -> filterToxic -> ground -> call -> demask -> log

7. Update `infrastructure/README.md` - add `heroku config:set` commands for
   MIA vars (`INFERENCE_URL`, `INFERENCE_MODEL_ID`, `INFERENCE_KEY`) and
   broadcast vars (`BROADCAST_CHANNEL_ID`, `BROADCAST_INTERVAL_IN_MS`)

8. Run `npm test` and `npm run lint`

## Test Strategy

Run `npm test` and `npm run lint`.

- Mock `global.fetch` to test client without network calls
- `maskPii` catches and indexes: `john@test.com` -> `[EMAIL_1]`,
  `555-123-4567` -> `[PHONE_1]`; returns map for demasking
- `demaskPii` restores `[EMAIL_1]` -> original value using map
- `sanitizeInput` strips: `ignore previous instructions`, `system:`,
  delimiter escapes (`---`, triple backticks)
- `filterToxic` flags `harm yourself` / `self-harm` and returns fallback
- `ground` returns data for match/group/player queries, empty for unmatched
- `ask()` integration: toxic input rejected before MIA, PII masked before
  call, response demasked for user, audit log keeps masked version
