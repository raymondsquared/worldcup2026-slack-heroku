# PRD: Heroku MIA Slack World Cup 2026

Date: 2026-06-13

A Slack app that showcases Heroku Managed Inference Agents (MIA) by bringing AI-powered World
Cup 2026 features - match broadcasting and conversational AI - into Slack workspaces.

Decisions: adr-1

## Background and Strategic Fit

Problem: Heroku Managed Inference Agents is a capable AI platform, but lacks a tangible,
engaging reference app that demonstrates its capabilities in a real-world, relatable context.
Developers evaluating Heroku MIA need a working example that goes beyond "hello world" -
something that shows grounded AI responses, guardrails, and platform integration in action.

Opportunity: FIFA World Cup 2026 (US/Mexico/Canada, June-July 2026) is a globally followed
event. Teams on Slack want to follow matches without context-switching away from their
workspace. This creates a natural domain for a showcase app.

Strategic fit:

- Demonstrates Heroku MIA capabilities (conversational AI, grounded responses, prompt engineering)
- Shows Heroku platform strengths (easy deployment, config vars, Socket Mode, scalability)
- Timely and engaging - leverages an active real-world event
- Low-risk - no customer data, no production dependencies, self-contained demo
- Reusable pattern - the architecture (Slack + AI + domain data) transfers to customer use cases

Who experiences this problem:

- Heroku customers evaluating MIA for their own Slack integrations
- Heroku internal teams wanting a demo/reference app
- Developer advocates needing showcase material

## Success Criteria

- SC-1: App connects to Slack via Socket Mode and responds to slash commands and AI chat
- SC-2: AI conversational responses are grounded in World Cup data (not hallucinated)
- SC-3: Match cards are broadcast to a channel on schedule without manual intervention
- SC-4: All AI interactions pass through guardrails (input sanitization, PII masking, toxicity filtering)
- SC-5: App deploys to Heroku with a single `git push` and runs successfully
- SC-6: Codebase serves as a clear, readable reference for developers building similar apps

## Scope

In scope:

- `/worldcup2026 schedule` slash command - quick view of the next 3 upcoming matches
- `/worldcup2026 groups` slash command - show tournament bracket/group table
- AI-powered conversational responses via @mention and DM (using Heroku MIA) - handles all
  other queries (schedules, teams, groups, etc.) naturally through conversation
- Proactive match broadcasting to a configured Slack channel
- Static/mock World Cup data (representative sample: ~24 teams, 6 groups, ~52 matches)
- Security guardrails per AGENTS.md (PII masking, prompt defense, toxicity detection, audit trail)
- Heroku deployment configuration (Procfile, config vars)
- Block Kit UI for all user-facing messages

## Out of Scope

- Live API integration with real-time match data feeds
- Persistent database (in-memory store only; data resets on dyno restart)
- Multi-workspace distribution (single workspace install)
- Admin dashboard or web UI outside of Slack
- Full 48-team tournament data (using representative sample only)
- Internationalization / multi-language support
- OAuth install flow for public Slack App Directory distribution
- Push notifications via email or SMS
- Interactive features: predictions, polls, trivia, leaderboards (deferred to future iteration)
- Reaction-based subscriptions and DM notifications (deferred to future iteration)
- Extensive slash command routing - AI chat handles most queries conversationally

## Requirements

### Functional Requirements

- FR-1: The app shall register a `/worldcup2026` slash command with two subcommands: `schedule`
  and `groups`
- FR-2: `/worldcup2026 schedule` shall return a Block Kit message showing the next 3 upcoming
  matches with dates, times, and teams
- FR-3: `/worldcup2026 groups` shall return a formatted group/bracket table showing all groups
  and their teams
- FR-4: When a user @mentions the bot in a channel, the app shall respond in the same thread
  with an AI-generated answer grounded in World Cup data
- FR-5: When a user sends a DM to the bot, the app shall respond conversationally using Heroku
  MIA, grounded in World Cup data
- FR-6: The AI shall be able to answer questions about schedules, teams, groups, and
  players using the static data as context
- FR-7: The app shall post match cards to a configured broadcast channel approximately 15
  minutes before each match's scheduled kickoff time
- FR-8: Match cards shall include team names and kickoff time

### Technical Requirements

- TR-1: The app shall be built with Node.js (>=22) using the @slack/bolt framework
- TR-2: The app shall connect to Slack using Socket Mode (WebSocket, no public HTTP endpoint
  required)
- TR-3: AI inference shall use Heroku Managed Inference Agents via its OpenAI-compatible REST API (`/v1/chat/completions`)
- TR-4: The AI client shall use native `fetch` (no external HTTP or OpenAI SDK dependencies)
- TR-5: All user input shall be sanitized before inclusion in AI prompts (prompt injection defense)
- TR-6: All AI outputs shall be filtered for toxicity and PII leaks before display to users
- TR-7: PII patterns (emails, phone numbers, SSNs, card numbers) shall be detected and masked
  with tokens before sending to MIA
- TR-8: All AI interactions shall be logged with timestamps, masked content, and trust signals (audit trail)
- TR-9: World Cup data shall be stored as static JSON files with query helper functions
- TR-10: Application state shall be stored in-memory using Maps where needed
- TR-11: A scheduler shall check for upcoming matches at regular intervals and trigger broadcasts
- TR-12: The app shall be deployable to Heroku via `git push` with a Procfile and config vars
- TR-13: Code shall pass ESLint and Prettier checks
- TR-14: Tests shall be written using Jest and pass via `npm test`

### Constraints

- C-1: No persistent database - all state is ephemeral (acceptable for a showcase/demo app)
- C-2: Static data only - no live sports API integration
- C-3: Single Slack workspace deployment (not distributed via App Directory)
- C-4: Must follow all guardrails defined in AGENTS.md
- C-5: Secrets must be stored in Heroku config vars, never in code
- C-6: Heroku MIA endpoint and model determined by config vars (not hardcoded)

## Assumptions

- A-1: The Heroku MIA endpoint is available and the model accepts OpenAI-compatible chat completion requests
- A-2: The Slack workspace has a bot installed with appropriate scopes (chat:write, commands, im:write, app_mentions:read)
- A-3: Socket Mode is sufficient for the demo (no need for Events API HTTP endpoint)
- A-4: In-memory state loss on dyno restart is acceptable for a showcase app
- A-5: A representative data sample (24 teams, 6 groups, 52 matches) is sufficient to demonstrate all features
- A-6: Users will interact with the bot in English
