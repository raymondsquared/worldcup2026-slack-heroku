# PRD: Live Data, Thread-Aware Mentions, and Real Squads

Date: 2026-06-16

Iteration 2 of the World Cup 2026 Slack app - transitioning from static demo data to
a live, data-driven experience powered by a football data API.

Decisions: adr-2

## Background and Strategic Fit

Problem: The World Cup 2026 tournament is live (June 11 - July 19, 2026), but the Slack app
still uses hardcoded mock data with fabricated player names and no live scores. When users
@mention the bot in a match broadcast thread asking "what's the score?" or "who scored?",
it cannot answer - it has no live feed and no awareness of which match the thread belongs to.

This undermines the app's purpose as a compelling showcase for Heroku Managed Inference
Agents. AI grounded in static data feels like a toy; AI grounded in real, live match data
demonstrates genuine platform capability.

Opportunity: By integrating the football API for live match data, the bot becomes genuinely
useful during the tournament - answering real questions with real answers in real-time. This
transforms the showcase from "look what it could do" to "look what it does."

Strategic fit:

- Demonstrates Heroku MIA grounded in dynamic, external data (not just static JSON)
- Shows real-time data integration on Heroku (polling, caching, event detection)
- Timely - tournament is actively running; every day without this is a missed demo opportunity
- Natural extension - builds on existing architecture without rewriting
- Showcases Slack platform features (message metadata, thread context, card updates)

Who experiences this problem:

- Slack workspace members actively following matches - they want live answers
- Developers evaluating MIA - static mocks don't demonstrate real-world grounding
- Heroku advocates - need a working live demo during the tournament window

## Success Criteria

- SC-1: Live score data appears in broadcast cards within 30 seconds of a goal
- SC-2: Thread-aware mentions return match-specific answers (not generic World Cup responses)
  when asked in a broadcast thread
- SC-3: Real player names appear in all AI responses - no fabricated or mock player names
- SC-4: App degrades gracefully when the football API is unreachable (no crashes, stale cached
  data served with indication)
- SC-5: Polling does not exceed the football API rate limits during multi-match periods
  (4 simultaneous group-stage matches)
- SC-6: Thread context survives dyno restarts - bot correctly identifies match from Slack
  message metadata after restart

## Scope

In scope:

- Live score polling from the football API every 15 seconds during active matches
- Minute-by-minute event broadcasting to match threads (goal alerts, cards, substitutions)
- Broadcast card updates - edit posted match cards with current score as match progresses
- Thread-context detection via Slack message metadata - bot knows which match a thread is about
- Thread-grounded AI - MIA responses include live match data when asked in a broadcast thread
- Real player and team data - full 48-team squads with real names, positions, jersey numbers
- Data sync script (`npm run sync`) - manual script to fetch/refresh data from API
- Resilient degradation - exponential backoff on API failures, cached data fallback
- Updated `/worldcup2026 schedule` command to show live scores for in-progress matches

## Out of Scope

- Real-time WebSocket streaming (polling at 15s intervals is sufficient for Slack UX)
- Persistent database (in-memory cache with JSON file fallback remains the pattern)
- Multi-provider support (the football API only; no abstraction layer for provider switching)
- Historical match analytics or statistics dashboards
- User preferences or subscriptions (e.g., "notify me about USA matches")
- Predictions, polls, trivia, or interactive features (deferred per original PRD)
- Custom Slack App Directory distribution (single workspace install via Socket Mode)
- Player photos or media assets
- Automated score-to-disk persistence (manual sync script only)

## Requirements

### Functional Requirements

- FR-1: The app shall poll the football API's `/fixtures?live=all` endpoint every 15 seconds
  when any World Cup match is in progress, caching the response in memory
- FR-2: When a new goal event is detected (by comparing cached events between polls), the app
  shall post a formatted goal alert message to the match's broadcast thread including scorer
  name, minute, and updated score
- FR-3: When a new card or substitution event is detected, the app shall post a formatted
  event alert to the match's broadcast thread
- FR-4: The app shall edit the original broadcast card message with the current score whenever
  the score changes during a live match
- FR-5: When broadcasting a match card, the app shall attach the match ID as Slack message
  metadata (`event_type: worldcup_match`, `event_payload: { matchId }`)
- FR-6: When the bot is @mentioned in a broadcast thread, it shall read the parent message's
  metadata to identify the match, then ground its AI response on that match's live data
  (score, events, statistics)
- FR-7: Outside of broadcast threads, the bot shall continue to handle @mentions and DMs
  using the existing general grounding logic with real player/team data
- FR-8: The `/worldcup2026 schedule` command shall show live scores for any in-progress
  matches alongside the next upcoming matches
- FR-9: Sync scripts (`npm run sync:teams`, `sync:fixtures`, `sync:players`,
  `sync:fixture-events`, `sync`) shall fetch data from the football API and write to
  local JSON files. Fixture events use two-step: discover finished via date, fetch detail by ID.
- FR-10: The app shall function with cached JSON data when the football API is unreachable,
  serving stale data rather than erroring

### Technical Requirements

- TR-1: Use native `fetch` (Node 22 built-in) for the football API HTTP calls - no new HTTP
  dependencies
- TR-2: Each fixture entry in `fixtures.json` shall include an `externalId` field
  fields for mapping to the football API fixture IDs; sequential internal IDs are canonical
- TR-3: Per-fixture detail files (`fixtures/{id}-{homeTeamId}-{awayTeamId}.json`) shall be
  created by the sync script for finished fixtures; the app shall never auto-write fixture
  files during operation
- TR-4: The live data poller shall implement exponential backoff on API failures
  (15s -> 30s -> 60s -> 120s max) and mark data as stale after 5 consecutive failures
- TR-5: The API client shall never log authentication headers or API keys in error output
- TR-6: The poller lifecycle shall check `/fixtures?date=today` every 15 minutes; when live
  matches are detected, switch to 15s aggressive polling; stop when no live matches remain
- TR-7: Slack rate limiting shall use Bolt's built-in `@slack/web-api` retry-after handling;
  application logic shall prioritize user responses over broadcast updates over event posts,
  and drop event posts older than 30 seconds
- TR-8: All existing guardrails (PII masking, prompt defense, toxicity detection, audit trail)
  shall remain active for thread-aware AI responses

### Constraints

- the football API paid tier required (~$50-100/mo for World Cup 2026 live data)
- the football API account must be provisioned before task execution begins
- Heroku dyno must support continuous operation (Basic+ plan; Eco dynos sleep)
- Node.js 22+ required (native fetch)
- Slack app manifest must register `metadata.event_type` for message metadata
- Single API call (`/fixtures?live=all`) returns all live matches - no per-match polling
- ~2,400 API calls/day estimated (15s × ~10 hours active matches/day)
- Slack message metadata payload limited to 4KB (sufficient - storing one integer matchId)

## Assumptions

- the football API covers World Cup 2026 with live fixture data including events, squads, and
  standings (confirmed via research; exact league/season ID to be verified with test call)
- the football API's paid tier provides near-real-time score updates (15-30 second latency)
- The Slack workspace has fewer than 50 active match threads at any time (in-memory
  cache is sufficient at this scale)
- Users accept 15-30 second delay between real-world events and Slack notifications
- The `chat:write` and `channels:history` OAuth scopes (already granted) are sufficient
  for reading message metadata and editing posted messages
- 48 teams × ~26 players each ≈ 1,200 player records stored as per-team files
  (`players/{teamId}.json`), loaded on demand
- Heroku ephemeral filesystem means `fixtures/{id}-{home}-{away}.json` files persist between
  requests but reset on deploy - acceptable since they are committed to git via sync script
