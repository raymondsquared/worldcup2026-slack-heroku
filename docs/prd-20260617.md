# PRD: Web Search, Personas, Card Redesign, and Post-Match Summary

Date: 2026-06-17

Iteration 3 of the World Cup 2026 Slack app - adding intelligent web search,
per-thread commentary personas, redesigned broadcast cards, event filtering
fixes, and auto-generated AI match recaps.

Decisions: adr-3

## Background and Strategic Fit

Problem: The World Cup 2026 group stage is underway (June 11 - July 4 group
matches). The Slack app now has live data polling, real-time event broadcasts,
and thread-aware AI grounding (iteration 2). However, several gaps remain:

1. The AI can only answer questions from local data. Questions about news,
   injuries, tactics, or historical context get "I don't know" responses -
   making the bot feel limited as a showcase for Heroku Managed Inference.
2. All AI responses use the same neutral tone regardless of context - missing
   an opportunity to make match threads more engaging and show personality.
3. The live broadcast card freezes between goals (minute stuck at last score
   change) and never transitions to a final state when the match ends.
4. Duplicate events appear in broadcast threads (API sends incomplete event
   first, then the complete version seconds later).

Opportunity: Fixing these issues transforms the app from "working demo" to
"compelling experience" - the kind of thing people share in Slack channels
because it's genuinely useful and entertaining during matches. Each feature
builds on existing architecture without rewriting.

Strategic fit:

- Demonstrates MIA grounded in dynamic external data (web search fallback)
- Shows agentic patterns (post-LLM fallback, multi-step reasoning)
- Showcases Slack platform features (message metadata for personas, card updates)
- Fixes production bugs that undermine credibility (frozen cards, duplicate events)
- Timely - group stage is peak engagement; every match day without these is missed

Who experiences these problems:

- Slack users following matches - frozen cards look broken, duplicates are noise
- Developers evaluating MIA - "I don't know" responses don't showcase capability
- Match thread participants - same tone for every match is monotonous

## Success Criteria

- SC-1: When the AI cannot answer from local data, it falls back to web search
  and provides a sourced answer within 8 seconds (end-to-end: ~1s first LLM
  call + ~1s web search + ~1.5s second LLM call + overhead)
- SC-2: Web search responses include source citations ("According to ESPN...")
- SC-3: Each broadcast thread has a persona (sporty/funny/serious) and all AI
  responses in that thread use the persona's tone consistently
- SC-4: No duplicate events are broadcast to match threads (events with
  incomplete player data are filtered before posting)
- SC-5: The live card minute updates on every poll cycle (~15s) even when no
  goals are scored
- SC-6: When a match finishes, the broadcast card updates to show "Match
  Finished" with final score and complete scorer list within 30 seconds
- SC-7: After match end, an AI-generated recap is posted to the match thread
  using the thread's persona tone

## Scope

### In Scope

- Web search API client with input sanitization and error handling
- Post-LLM fallback detection ("I don't know" / low confidence response check)
- Web result sanitization layer (treat fetched content as untrusted data)
- Source citation in AI responses ("According to...")
- Rate limiting / cost guard for web search (max N calls per minute)
- Three personas: sporty, funny, serious (system prompt variants)
- Persona stored in Slack message metadata on the broadcast card
- Persona detection from thread parent metadata (same pattern as matchId)
- Persona selection: random on card post, changeable via thread command
- Event filter: skip broadcast when playerName is null/undefined/empty/"unknown"
  (case-insensitive, log only). Exception: Var events always pass through
- Card redesign: 3-block layout (header + score section + scorers context)
- Full status text in header (no abbreviations: "First Half", "Halftime", etc.)
- Flag emojis on score line and before each scorer name in context
- Scorers grouped by team: home goals · away goals (with minutes)
- Card refresh on every active poll cycle (not just on score change)
- Match-end detection via status transition (live to finished)
- Final card: red indicator + "Match Finished" + score + complete scorer list
- Auto-generated match recap posted to thread after final card
- Recap grounded on full match events (goals, cards, subs, statistics)
- Recap uses the thread's persona for tone consistency
- Tests for all new code paths

### Out of Scope

- Web search as always-on augmentation (only fallback for unanswerable queries)
- Persistent search result caching across dyno restarts
- User preference for enabling/disabling web search
- Streaming web search results (single request-response)
- Per-user persona preferences (persona is per-thread only)
- Custom/user-defined personas beyond the three built-in ones
- Persona affecting broadcast event alerts (only affects AI chat responses)
- Match recap editing or regeneration by users
- Recap for matches that were not actively tracked (only live-tracked matches)
- WebSocket-based live updates (staying with 15s polling)
- Pre-LLM intent classifier for web search (decided against in ADR-3)

## Requirements

### Functional Requirements

- FR-11: When MIA responds with "I don't know" or equivalent (low confidence
  indicators), the app shall call the web search API with the sanitized user
  query, inject
  sanitized results as context, and call MIA again to produce a grounded answer
- FR-12: Web search responses shall include source citations indicating where
  information came from (e.g., "According to ESPN" with URL)
- FR-13: Web search calls shall be rate limited to a configurable maximum per
  minute to control costs (default: 10 per minute)
- FR-14: When posting a broadcast card, the app shall assign a random persona
  (sporty, funny, or serious) and store it in the Slack message metadata
  alongside the matchId
- FR-15: When the bot is mentioned in a broadcast thread, it shall read the
  persona from the thread parent's metadata and use the corresponding system
  prompt tone for its response
- FR-16: Before broadcasting an event to a match thread, the app shall check
  `playerName` - if null, undefined, empty, or "unknown" (case-insensitive),
  log the event to console and skip broadcasting. Exception: Var events always
  broadcast regardless of playerName
- FR-17: The broadcast card shall use the redesigned 3-block layout: header
  (status indicator + full status text + elapsed minute), section (flag +
  team + score + team + flag), context (scorers with flag per scorer, grouped
  by team)
- FR-18: The broadcast card header shall use full status text with no
  abbreviations ("First Half", "Second Half", "Halftime", "Extra Time",
  "Match Finished") and include elapsed minute where applicable
- FR-19: The broadcast card shall be updated on every active poll cycle
  (approximately every 15 seconds) to keep the elapsed minute current,
  regardless of whether the score has changed
- FR-20: When a match transitions from a live status to "Match Finished" (or
  equivalent finished status), the app shall update the broadcast card to
  show a red indicator, "Match Finished" status, final score, and complete
  scorer list
- FR-21: After posting the match-end final card, the app shall generate an
  AI-written match recap using MIA grounded on the full match event data and
  post it as a reply in the match thread
- FR-22: The post-match recap shall use the thread's persona tone if one is
  set in the message metadata

### Technical Requirements

- TR-9: The web search API client shall use native fetch (Node 22), include a
  configurable timeout (default 10s), and never log the API key in error output
- TR-10: Web search results shall be treated as untrusted data: sanitized with
  the existing guardrails pipeline (strip injection markers, delimit clearly
  from instructions) before injecting into MIA context
- TR-11: The "I don't know" detection shall use pattern matching on MIA's
  response text (configurable patterns: "I don't know", "I don't have
  information", "I cannot answer", "no data available")
- TR-12: Persona system prompts shall be defined as templates in a dedicated
  module, with the persona key stored as a string in message metadata
  (event_payload: { matchId, persona })
- TR-13: The event filter shall run in the scheduler's postEventAlert function
  before any Slack API calls are made (fail fast, no wasted API calls)
- TR-14: The card redesign shall use Slack Block Kit: header block (plain_text),
  section block (mrkdwn), and context block (mrkdwn elements)
- TR-15: The card refresh shall be triggered by the poller's onChanges callback
  on every poll cycle, not just when scoreChanged is true. A new field
  (elapsedChanged or unconditional refresh) shall trigger the card update
- TR-16: Match-end detection shall track previously-seen live fixture IDs and
  detect when a fixture disappears from the live response (indicating it
  finished). The app shall then fetch the fixture's final state and update
  the card accordingly. As a fallback, the passive-mode date-based endpoint
  check shall also detect recently-finished matches and update any cards that
  were missed (e.g., after dyno restart)
- TR-17: All existing guardrails (PII masking, prompt defense, toxicity
  detection, audit trail) shall remain active for web-search-augmented
  responses and post-match recap generation

### Constraints

- Web search API key required (already provisioned)
- Web search pricing: ~$0.01 per search (budget-conscious rate limiting)
- Heroku MIA must support OpenAI-compatible chat completions (already confirmed)
- Slack message metadata payload limited to 4KB (storing matchId + persona
  string is well within limits)
- Post-LLM fallback adds ~2.5s latency on search path (accepted trade-off per
  ADR-3)
- Flag emojis require Unicode support in Slack client (universal on modern
  clients)
- No new npm dependencies for web search (native fetch is sufficient)
- Node.js 22+ required (native fetch, already in place)

## Assumptions

- Web search provider returns relevant World Cup 2026 results (football news is
  well-indexed by major search providers)
- MIA reliably indicates uncertainty with detectable phrases ("I don't know",
  "I don't have information") rather than hallucinating answers
- The football API's event data includes null/undefined player fields for
  incomplete events (confirmed via production observation)
- Slack renders flag emojis consistently across desktop, mobile, and web clients
- Three personas (sporty, funny, serious) provide sufficient variety without
  requiring user customization
- The football API's status field transitions cleanly from live statuses to
  "Match Finished" (confirmed via API documentation)
- Post-match recap generation completes within MIA's 15s timeout (match event
  data for a single match is well within context limits)
