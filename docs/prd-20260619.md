# PRD: Daily Highlight Playlist Review

Date: 2026-06-19

Iteration 4 of the World Cup 2026 Slack app - adding a daily automated review
of the official FIFA highlight playlist from the previous day's matches,
enriching fixture data with highlight URLs, and posting an AI-generated digest
to the broadcast channel.

Decisions: ADR-4 (Daily Scheduler - In-Process Clock via setInterval),
ADR-5 (Topic Grounding in the Trust Layer - Scope Enforcement for Off-Topic
Questions)

## Background and Strategic Fit

Problem: The World Cup 2026 group stage is in full swing. The Slack app
broadcasts live scores, posts event alerts, and generates AI recaps - but once
a match is over, users must manually hunt through video platforms to find
official highlight reels. The official FIFA playlist publishes professional
highlights for every match, typically within hours of the final whistle.
There is no bridge between this content and the app's match data.

Opportunity: A daily highlight digest transforms the app from "live-only" to a
daily touchpoint that keeps engagement high even between match days. Users open
Slack in the morning and see a curated summary of yesterday's action with
direct links to official highlights. This also enriches the fixture data model
(adding `highlightsURL` to `fixtures.json`), enabling future features like
including highlight links in match context for AI answers.

Strategic fit:

- Demonstrates MIA content generation (AI writes the daily digest, grounded on
  real match results and video data)
- Extends the data enrichment pattern (same as sync scripts enriching fixtures)
- Adds a new engagement loop (daily digest vs. real-time alerts)
- Leverages official FIFA content (credible, high-quality, always available)
- Low API cost (single daily fetch, minimal quota usage)
- No new Slack infrastructure (posts to existing broadcast channel)

Who benefits:

- Slack users who missed matches live - morning catch-up with highlight links
- Users who watched matches - quick recap + official highlight to rewatch/share
- Developers evaluating the app - shows MIA generating editorial content, not
  just Q&A

## Success Criteria

- SC-1: Daily digest posts to broadcast channel at 12:00 UTC (within 2 min)
  on days when previous-day highlights exist
- SC-2: At least 90% of highlight videos are correctly matched to fixtures.
  Automated validation: log `matched_count / total_published_count` each run.
  Alert (`console.warn`) if ratio drops below 0.9
- SC-3: Matched fixtures have `highlightsURL` persisted in `fixtures.json`
  within the same job run
- SC-4: No digest posted on rest days (zero highlights found = zero posts)
- SC-5: Playlist API usage stays minimal (single daily fetch with pagination)
- SC-6: Digest is engaging, concise (2-4 paragraphs), includes all matched
  highlights with links, grounded on actual scores/events

## Scope

### In Scope

- Playlist API client (fetch playlist items) with API key auth
- Pagination support (handle playlists exceeding 50 items)
- Daily scheduler (configurable time via env var, default 12:00 UTC)
- Startup missed-run detection (run immediately if missed today's window)
- Video-to-fixture matching with team name normalization and alias support
- Fixture data enrichment: add optional `highlightsURL` field to
  `fixtures.json`
- AI-generated daily digest: MIA produces a summary with highlight links
- Digest posted to broadcast channel as a standalone message (not threaded)
- Sync script extension: `npm run sync:highlights` for operator backfill
  (fetches full playlist, matches all videos regardless of date)
- Configurable playlist ID via env var (`HIGHLIGHTS_PLAYLIST_ID`)
- Configurable API key via env var (`HIGHLIGHTS_API_KEY`)
- Error handling: graceful degradation if playlist API fails (log, skip,
  retry next day)
- Observability: log matched/unmatched video counts, match ratio, API usage
- Tests: unit tests for matcher (team name variations, edge cases),
  integration tests with mock API responses, digest generation tests
- Guardrails: video titles sanitized before MIA context, audit trail for
  digest generation

### Out of Scope

- OAuth or user-specific playlists (API key auth only, public playlist)
- Video content analysis or transcription (title-based matching only)
- Embedding video players in Slack (link unfurling handles this natively)
- User preferences for digest timing or opt-out
- Real-time highlight notifications per match (daily batch only)
- HTML scraping or RSS feeds (proper API only)
- Custom highlight playlists beyond the configured one
- Highlight links surfaced in AI chat responses (future iteration)
- Multi-language digest support

## Requirements

### Functional Requirements

- FR-23: The app shall fetch the configured highlight playlist daily at a
  configurable time (default 12:00 UTC) and identify highlight videos
  published on the previous UTC calendar day
- FR-24: The app shall match highlight videos to local fixtures by parsing
  team names from video titles. Matching shall support full country names,
  FIFA codes, and country aliases (from `countries.json`), all
  case-insensitive and diacritic-stripped. Both home and away teams must
  appear in the title for a match. Videos without "Highlight" in the title
  shall be skipped
- FR-25: When a highlight video is matched to a fixture, the app shall
  update the fixture entry in `fixtures.json` by adding or updating a
  `highlightsURL` field with the video URL
- FR-26: After processing the playlist, the app shall generate an
  AI-written daily digest using MIA, grounded on the previous day's match
  results (scores, key events from fixture data) and the matched highlight
  video links
- FR-27: The daily digest shall be posted to the broadcast channel as a
  standalone message (not threaded to any match card). If no highlight videos
  were found for the previous day, no message shall be posted
- FR-28: A manual sync command (`npm run sync:highlights`) shall allow
  operators to fetch the full playlist and backfill `highlightsURL` fields
  for all matchable fixtures regardless of publish date
- FR-29: On startup, if the current UTC time is past the configured run
  time and today's run has not completed, the app shall execute the
  highlights job immediately (missed-run recovery)
- FR-30: The app shall provide a `/worldcup2026 highlights` subcommand that
  renders recent matched highlights on demand as Block Kit cards (flags,
  score, stage/group, kickoff time, and a "Watch" link button), reusing the
  daily-digest card layout. The command is registered alongside `schedule`
  and `groups`, and all command discovery surfaces (Slack app manifest
  `description`/`usage_hint`, the architecture diagram, and the in-command
  usage text) shall list `highlights`
- FR-31: The `highlights` command shall surface the most recent N fixtures
  that have a `highlightsURL`, ordered newest-first across all match days
  (not limited to the previous calendar day). N shall default to 5 and be
  overridable via `HIGHLIGHTS_COMMAND_FIXTURES_LIMIT`; an invalid value
  falls back to the default. When no highlights exist, a friendly empty-state
  message shall be shown
- FR-32: The `/worldcup2026 schedule` and `groups` cards shall render with
  team flags and clear visual structure (headers, dividers, context lines).
  Groups shall be ordered alphabetically (A->L) and teams ranked within each
  group, showing points and goal difference. UK constituent countries that
  share the `GB` flag ISO (England, Scotland) shall render distinct Slack
  named flag emoji
- FR-33: Match cards (the `highlights` command and the daily digest) and the
  `schedule` view shall display kickoff time as a viewer-localized Slack date
  token paired with a fixed shared UTC reference
- FR-34: The assistant's trust-layer grounding boundary shall keep off-topic
  (non-football) questions in scope. The system prompt (primary boundary) shall
  instruct the model to redirect off-topic / general-knowledge questions instead
  of answering them. When the model returns a low-confidence answer, the post-LLM
  web-search fallback shall run only if the question is football-related by
  either signal: the model's `<isFootballRelatedScore>` (0-100, threshold 70) OR
  a deterministic keyword check; otherwise the low-confidence answer is returned
  unchanged so general-knowledge questions are not answered via web search

### Technical Requirements

- TR-18: The playlist client shall use the playlist items API endpoint with
  API key authentication. The client shall use native fetch (Node 22),
  include a configurable timeout (default 10s), and never log the API key
  in error output
- TR-19: Playlist fetching shall handle pagination (pageToken/nextPageToken)
  to retrieve all items when the playlist exceeds 50 videos per page
- TR-20: The video-to-fixture matcher shall normalize team names (NFD
  decomposition + strip combining marks, lowercase) and support matching
  against `teams.json` names, team IDs (FIFA codes), and `countries.json`
  aliases. Matching requires both home and away teams from the fixture to
  appear in the video title. Tie-breaking: exact name > alias; closest date
  if multiple fixtures match; prefer title containing "Highlights" over
  "Extended" or "Full Match" if multiple videos match one fixture
- TR-21: Fixture data writes shall be atomic: read `fixtures.json`, merge
  `highlightsURL` into matched entries, write to a temp file in the same
  directory, then rename over the original. A module-level mutex (promise
  queue) shall serialize concurrent write attempts within the process
- TR-22: The daily scheduler shall use setInterval with a time-of-day check
  (compare current UTC hour/minute to configured run time). It shall persist
  `lastRunDate` to `src/data/.last-highlights-run` to survive restarts and
  prevent duplicate runs. The file is updated only after successful digest
  post (or when zero highlights found)
- TR-23: Video titles shall be treated as untrusted data: sanitized with
  the existing `sanitizeInput()` guardrail before inclusion in MIA context.
  The digest prompt shall use clear delimiters (same `---` pattern as
  existing grounding)
- TR-24: All existing guardrails (PII masking, toxicity detection, audit
  trail) shall remain active for the digest generation path. The digest
  call shall use the `ask()` function's recap mode for direct system prompt
  control
- TR-25: The playlist API key shall be stored as an environment variable
  (`HIGHLIGHTS_API_KEY`) and the playlist ID as (`HIGHLIGHTS_PLAYLIST_ID`).
  The feature shall gracefully degrade (log warning, skip digest) if either
  is not set
- TR-26: Each run shall log: total videos fetched, videos from previous
  day, matched count, unmatched count, and match ratio. If
  `matched / published < 0.9`, emit a `console.warn` alert
- TR-27: Live-card rendering shall never emit an empty Slack block element
  (e.g. an empty `context` text), which Slack rejects as `invalid_blocks`:
  the scorers line falls back to "No goals yet" when no goal maps to either
  team, and `updateCard` shall log the Slack error code on failure rather
  than swallow it. Tests that mutate `fixtures.json` shall write to a
  throwaway copy (via `HIGHLIGHTS_FIXTURES_PATH`) so parallel suites that
  `require` the file are not raced

### Constraints

- Playlist API quota limits apply (varies by provider). Single daily fetch
  with pagination uses minimal quota (1-10 requests/day)
- API key requires provider account setup (documented in README)
- FIFA playlist ID is assumed stable for the tournament duration;
  configurable via env var as fallback
- Fixture data write is serialized via mutex within the worker process.
  CLI sync scripts run as separate processes; atomic rename prevents
  corruption
- Heroku worker dyno must remain running for the daily scheduler to fire
  (same constraint as existing broadcast scheduler)
- Node.js 22+ required (native fetch, already in place)
- No new npm dependencies (native fetch for HTTP, fs for file I/O)

## Assumptions

- The FIFA highlight playlist contains official highlight videos for all
  World Cup 2026 matches, published within hours of match completion
- Video titles consistently include both team names in a parseable format
  (confirmed by examining existing FIFA World Cup playlists from prior
  tournaments)
- The playlist API returns publish timestamps in UTC, allowing reliable
  date filtering
- A single daily run at 12:00 UTC captures all highlights from the previous
  day (matches end by ~05:00 UTC at the latest for US/Mexico/Canada time
  zones)
- The playlist is publicly accessible (no authentication beyond API key)
- Fixture data from `fixtures.json` has accurate team IDs and dates for
  matching
- Slack auto-unfurls video links in messages, providing thumbnail previews
  without additional work
