# PRD: Live-Data Reliability and Data-Model Consistency

Date: 2026-06-21

Iteration 5 of the World Cup 2026 Slack app - primarily a reliability iteration
that hardens the live-broadcast path discovered during match-day operation: the
live card now shows real scorers, the poller activates exactly at kickoff, the
worker restarts on a predictable schedule to survive Heroku's dyno cycling,
event alerts identify the team involved, duplicate event alerts caused by API
minute drift are suppressed, event player names resolve to one canonical squad
name, and the internal team-id model is aligned to official FIFA codes. The
iteration also takes on one MIA grounding improvement surfaced by the same
match-day use: replacing the hand-coded keyword router with
model-driven tool calling so MIA retrieves the right local data (team fixtures,
past results and events, player records) instead of falling back to web search
for questions the app can already answer offline.

Decisions: proposes ADR-6 (MIA client-side tool calling) for the grounding
change (ADR-5 is already used for trust-layer topic grounding); the reliability
work needs no new ADR (extends ADR-2 live-data caching/polling and the ADR-4
in-process scheduler pattern)

## Background and Strategic Fit

Problem: With the highlight digest (iteration 4) shipped, the app went into
real match-day use and several live-path defects surfaced that earlier unit
tests did not catch because they only exercised formatters in isolation:

1. The live match card showed "No goals yet" for the entire match even though
   goal alerts posted correctly in the thread. Every live event resolved to a
   null team, so the scorers line could never group goals to a side.
2. The poller stayed in 15-minute passive mode through kickoff and only flipped
   to 15-second active polling on the next passive tick, leaving a 10-14 minute
   blind window at the start of matches.
3. Heroku cycles worker dynos roughly every 24 hours at an unpredictable time.
   A restart inside a pre-match window or just after full time re-posts a card
   or recap because the dedup state is in memory.
4. Threaded event alerts (cards, subs, missed penalties) named only the player
   and minute, so a viewer could not tell which team the event belonged to.
5. Threaded event alerts were posted more than once during a match, each copy
   showing a slightly different minute. The football API revises an event's
   minute between polls, and the differ keyed its dedup on the minute, so a
   re-reported substitution or card looked like a brand-new event.
6. Four teams used an internal id that differed from their official FIFA code
   (BOS/CAP/CON/CUR), leaving the primary key and the canonical code divergent
   for a subset of the tournament.
7. The same player appeared under different names within one match. The football
   API labels a player inconsistently between events (a card read "Carlens
   Arcus" while his substitution read "C. Arcus"), and the mapper stored each
   raw API name verbatim instead of the one canonical squad name.
8. MIA searched the web for questions the app could answer from local data.
   MIA's retrieval used a hand-coded keyword router with three regex branches, so
   "when is the next England match" injected the global next-3 fixtures (not
   England's), past-result and scorer questions matched no branch at all, and a
   player named without their team was invisible. Empty or wrong context drove a
   low-confidence answer and the web-search fallback, even though finished
   fixtures, match events, standings, and squads are all present locally.
9. The live match card stayed on its pre-match "Fixture Starting Soon!" layout
   after kickoff and only switched to the live score card when the first change
   arrived (a goal, card, sub, or any score/elapsed delta). The differ's
   first-observation restart guard seeds dedup state and emits no diff, and the
   card is only rewritten by `updateCard` on a score/elapsed change, so a 0-0
   match showed the pre-match card for minutes - and because the first goal
   changes the score, the card appeared to update only "when the first event
   comes through".

Opportunity: These are credibility bugs. A broadcast card that says "No goals
yet" at 5-1, a duplicate recap, or the same substitution posted twice
undermines the showcase more than any missing feature. Fixing the live path
makes the app trustworthy during the exact moments it gets the most attention -
live matches.

Strategic fit:

- Protects the most-watched surface (live match threads) from visible defects
- Reduces wasted football-API calls (precise kickoff activation vs. polling)
- Makes worker behavior deterministic on Heroku without a paid add-on
- Improves data-model hygiene (one canonical team code) to prevent future
  matching bugs

Who experiences these problems:

- Slack users following matches - frozen scorer lines and duplicate posts read
  as broken
- Operators - unpredictable dyno cycling makes incidents hard to reason about
- Future contributors - divergent team ids are a latent source of confusion

## Success Criteria

- SC-1: When a goal is scored, the live card scorers line shows the correct
  scorer grouped under the scoring team within one active poll cycle (~15s)
- SC-2: The poller is in active mode within 15 seconds of kickoff, with no
  10-14 minute blind window at the start of a match
- SC-3: No duplicate pre-match cards or post-match recaps are produced as a
  result of Heroku's 24h dyno cycling
- SC-4: Every threaded event alert (goal, card, sub, missed penalty) identifies
  the team involved via its flag, while VAR alerts (not team-attributed) are
  unaffected
- SC-5: All 48 teams have an internal id equal to their official FIFA code, and
  flag rendering is unchanged by the alignment
- SC-6: An event whose minute is revised by the API between polls is posted to
  the thread exactly once, while a genuine brace still produces two goal alerts
- SC-7: A player is shown under one consistent name across every event in a
  match (the canonical squad name), regardless of how the API labels them in a
  given poll
- SC-8: Questions answerable from local data (a named team's next match, a past
  result or its scorers, a player's squad record) are answered from local data
  via tool calls without a web search, while questions whose data is genuinely
  absent locally still fall back to web search as before. Measurable via the
  retrieval-path field in the interaction log: with the tool path live, the
  `web_search` rate for football questions drops versus the keyword-router-only
  baseline.
- SC-9: When a match goes live, its broadcast card transitions from the
  pre-match "Fixture Starting Soon!" layout to the live score card within one
  active poll cycle (~15s) of the first poll that observes the fixture live,
  without waiting for a goal or other event, and without re-posting historical
  events to the thread.

## Scope

### In Scope

- Runtime team-id resolution so live events carry a resolved team
- Live card and recap sourcing live events from the Tier-1 cache merge
- A precise one-shot kickoff activation timer in the poller
- A scheduled in-process daily restart at a safe UTC hour, configurable by env
- Team flag prefix on goal/card/sub/missed-penalty alerts
- A minute-independent event dedup key so API minute drift does not re-post
  the same substitution or card
- Resolving event player and assist names to the canonical squad name by
  `externalId`, plus a one-time backfill of the already-synced fixture files
- Renaming four internal team ids to their official FIFA codes, including the
  associated data content and per-team / per-fixture filenames
- Replacing the keyword router with MIA client-side tool calling: a
  whitelist of read-only tools over the existing data accessors (team fixtures,
  past results, match events, standings, squads, player lookup), with a direct
  single model call as the deterministic fallback
- An initial card-refresh signal so the live card transitions from the
  pre-match layout at kickoff without waiting for the first event
- Threading the differ's `matchEnded` flag into the card renderer so the card
  flips to the finished layout in lockstep with the recap, instead of trusting a
  cached feed status that can lag on a live value
- Tests covering the above and updated assertions for the renamed ids

### Out of Scope

- Any change to README or architecture diagrams beyond what already documents
  the live path
- Re-rendering already-finished match cards retroactively
- Heroku paid scheduler add-ons or platform-level restart configuration
- Changes to highlight matching or persona behavior
- The Heroku Agents server-side orchestration endpoint (the tool-calling work
  uses the client-side `/v1/chat/completions` loop; server-side agents are a
  later task if the client loop proves limiting)
- New per-player stat fields not present in local data (height, age, market
  value); these continue to fall through to web search

## Requirements

### Functional Requirements

These tasks are reliability fixes; user-visible behavior is captured in the
Success Criteria above and realized through the Technical Requirements below.

### Technical Requirements

- TR-27 (existing, iteration 4): Live-card rendering shall never emit an empty
  Slack block element; the scorers line falls back to "No goals yet" when no
  goal maps to either team, and `updateCard` shall log the Slack error code on
  failure. (Referenced here because task-35 corrects the underlying data so the
  fallback is no longer hit during normal play.)
- TR-28: The live broadcast path shall be correct and deterministic during a
  match. Specifically:
  - (a) Live-data correctness: the mapper shall resolve every live event's team
    at runtime. The external-to-internal team map shall be seeded from
    `teams.json` at module load (not only by the offline sync script), so the
    live card scorers line and the recap attribute goals/cards/subs to the
    correct team rather than rendering "No goals yet". The live card and recap
    shall read events from the Tier-1 cache merge produced by `getFixtureById`.
  - (b) Precise kickoff activation: while passive, the poller shall schedule a
    one-shot timer for the next kickoff and switch to active polling when it
    fires, eliminating the passive-interval blind window. The timer shall be
    rescheduled on return to passive mode and cleared on stop or on transition
    to active mode.
  - (c) Scheduled daily restart: the worker shall schedule an in-process
    restart (`process.exit(0)`, relying on Heroku auto-restart) at a safe UTC
    hour configurable via `DAILY_RESTART_HOUR_IN_UTC` (default 11), so the 24h
    dyno cycle lands in a window with no matches or pre-match windows and the
    in-memory dedup state resets predictably. The timer shall `unref()` so it
    does not keep the process alive.
- TR-29: Threaded event alerts shall display the team involved. A team-flag
  prefix shall be added to goal, card, missed-penalty, and substitution alerts,
  derived from the resolved `event.teamId`. When the team cannot be resolved
  the prefix shall be empty (no white-flag fallback leak). VAR alerts, which
  are not attributed to a single team, shall remain unchanged.
- TR-30: Internal team ids shall equal official FIFA codes for all 48 teams.
  The four divergent ids shall be renamed (`BOS->BIH`, `CAP->CPV`, `CON->COD`,
  `CUR->CUW`) across `teams.json`, `countries.json`, `fixtures.json`, sample
  and test JSON, and the associated player-squad and fixture-detail filenames
  (the data layer resolves `players/{id}.json` and
  `{id}-{home}-{away}.json`). Flag rendering, which keys off `flagISO`, shall be
  unchanged.
- TR-31: The event differ shall not re-post an event when the football API
  revises its minute between polls. The dedup key shall be minute-independent
  for events that occur at most once (substitutions keyed on the off/on player
  pair; cards keyed on detail plus player), while goals and VAR retain the
  minute so a genuine brace stays distinct. The accepted residual risk that a
  goal whose minute drifts may re-post shall be documented in the code.
- TR-32: Event player names shall be resolved to the canonical squad name. The
  mapper shall seed an `externalId`-to-name map from the squad files at module
  load and rewrite each event's `playerName` and `assistPlayerName` to the
  squad name for that id, falling back to the raw API name when the id is absent
  from every squad. The same resolution shall apply to both the live poller and
  the offline sync (both route through `mapEvent`), and a repeatable, idempotent
  script shall backfill the player names in fixture detail files synced before
  this change.
- TR-33: MIA shall retrieve grounding data through model-driven tool calling
  rather than the keyword router. The inference client shall expose a
  tool-calling entry point that surfaces the assistant message `tool_calls`
  (the existing `chat()` string return is preserved for recap and other
  callers). `ask()` shall offer the model a fixed whitelist of seven read-only
  tools backed by the existing data accessors - upcoming fixtures (optionally
  filtered by team), past team results, fixture events, standings, team squad,
  player lookup by name, and a derived player-goals tally - execute requested
  tools in a loop bounded by a
  maximum iteration count and a total wall-clock budget, and feed `role: tool`
  results back until the model returns a final answer in the existing XML
  format. `dispatchTool` shall reject any name not on the whitelist and never
  evaluate arbitrary input; tool arguments shall be parsed defensively. The
  behavior shall be gated on a one-shot probe confirming the configured
  `INFERENCE_MODEL_ID` supports function calling, with a direct single model call
  (no tool context) as the deterministic fallback when the model is unsupported,
  the loop yields no tool call, or the loop exceeds its wall-clock budget.
  `logInteraction` shall record which retrieval path produced each answer
  (`tool_call`, `context`, `direct`, or `web_search`) so the web-search fallback
  rate is observable and SC-8 is measurable. All
  input/output guardrails (PII masking, sanitize, toxic filtering, audit) and
  the web-search low-confidence fallback shall be unchanged. The derived
  player-goals tally (`get_player_goals`, counting `Goal` events by `externalId`
  across fixtures) is the one tool requiring new cross-fixture logic rather than
  a thin accessor wrapper, carrying its own counting rules (own goals excluded,
  penalties reported as a subset) and dedicated tests.
- TR-34: The live broadcast card shall transition from the pre-match layout to
  the live score card at kickoff without waiting for a score, elapsed, or event
  change. On the first observation of a fixture, the differ's restart guard
  shall continue to seed dedup state without re-broadcasting historical events
  (`newEvents` stays empty), but shall emit a one-time initial diff (carrying an
  `initial` marker, with `scoreChanged` and `elapsedChanged` both false) so the
  broadcaster rewrites the existing pre-match card via `updateCard`. The
  scheduler shall call `updateCard` when the diff is `initial` in addition to
  the existing score/elapsed-change triggers; `updateCard`'s existing no-op when
  no message timestamp is tracked keeps a mid-match restart (lost in-memory
  timestamp) from posting a spurious card. The match-end diff
  (`buildMatchEndDiff`) is still built outside `diffAll`, so the initial-diff
  logic does not touch it; the scheduler shall additionally call `updateCard` on
  a match-end diff and forward its `matchEnded` flag through to `formatLiveCard`,
  which shall treat that flag as the authoritative match-over signal (rather than
  the cached fixture status, which can lag on a live value when the feed stops
  returning a finished fixture) so the card flips to the finished render in
  lockstep with the recap.

### Constraints

- Heroku worker dyno cycling (~24h) is not configurable; the in-process restart
  exists to make the cycle land in a safe window, not to prevent it
- The kickoff timer and restart timer are in-memory and reset on restart, which
  is acceptable because both are re-derived on startup
- Goal and VAR dedup keys retain the minute to keep a brace distinct, so a goal
  whose minute drifts between polls may re-post; this residual risk is accepted
  (a brace is common, goal-minute drift is rare) and documented in the code
- No new npm dependencies (native timers, fs, existing data files)
- Node.js 22+ (already in place)
- Tool calling adds inference round-trips (model -> tool_call -> execute ->
  model); this is bounded by a maximum iteration count and is acceptable for a
  Slack command, but it is more latency and tokens than the single-call
  direct path

## Assumptions

- The football API reports a resolvable external team id on every live event;
  all 48 teams have an `externalId` populated in `teams.json`
- Kickoff times in `fixtures.json` are accurate enough to schedule activation
  within the desired 15s latency
- Heroku restarts a worker that exits cleanly, so `process.exit(0)` is
  sufficient to recycle the dyno
- All match kickoff times fall outside the 05:00-11:00 UTC window, making
  11:00 UTC a safe default restart hour
- Internal team ids are used only as opaque keys (never parsed for meaning), so
  renaming them to FIFA codes is behavior-preserving
- A substitution occurs at most once per (off, on) player pair and a player
  receives each card type at most once per match, so a minute-independent key
  built from those fields does not suppress genuinely distinct events
- The API's `event.player.id` matches the squad `externalId`, and each
  `externalId` is unique across all 48 squads, so resolving a name by id yields
  exactly one canonical squad name per player
- The configured `INFERENCE_MODEL_ID` supports OpenAI-compatible function
  calling (a `tools` request returns `tool_calls`); this is verified by a
  one-shot probe before deploy, and the direct-call fallback keeps an unsupported
  model working at runtime (it absorbs a model that emits no `tool_calls`)
- A pre-match card is posted before kickoff in the common case, so the initial
  diff has a tracked message timestamp to rewrite; when none exists (a worker
  that starts after kickoff), `updateCard` no-ops on the initial diff and the
  existing `postEventAlert` first-event fallback still posts a live card
